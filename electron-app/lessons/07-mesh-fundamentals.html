Lesson 07

Mesh Fundamentals

Vertices, triangles, MeshFilter, MeshRenderer, and how data flows

What Is a Mesh?

A mesh is the geometric data that defines a 3D shape. It's composed of vertices, edges, and faces (triangles). In Unity, a mesh is a data container — the actual rendering is handled by other components.

MESH ANATOMY: Vertex 0 /\ / \ / \ / \ /________\ Vertex 1 Vertex 2 This triangle: - 3 vertices (positions in 3D space) - 3 edges (connections between vertices) - 1 face (the triangle itself) - 3 indices: [0, 1, 2] (vertex order)

Vertex Attributes

Each vertex isn't just a position — it carries multiple attributes that the shader uses:

Attribute Type Purpose
Position Vector3 Location in object space
Normal Vector3 Direction the surface faces (for lighting)
UV (UV0) Vector2 Texture coordinates (0-1 mapping to texture)
UV2, UV3... Vector2 Lightmap UVs, additional texture channels
Tangent Vector4 Used with normal maps for correct lighting
Color Color32 Per-vertex color (blending, LOD, effects)
BoneWeights BoneWeight Skeletal animation influence
🧠 Vertex Data Size

Each attribute adds memory. A basic vertex (position + normal + UV) is about 32 bytes. With tangent, color, and multiple UVs, it can reach 60+ bytes. For a 100K vertex mesh, that's 3-6 MB of vertex data.

Triangles and Indices

GPUs only render triangles. All mesh faces are triangulated. The index buffer defines which vertices form each triangle.

VERTEX SHARING via Index Buffer: Vertices: Triangles (indices): 0: (-1, 0, 0) Triangle A: [0, 1, 2] 1: ( 0, 1, 0) Triangle B: [0, 2, 3] 2: ( 1, 0, 0) 3: ( 0,-1, 0) 1 /|\ / | \ 0 | 2 Vertices 0 and 2 are SHARED \ | / between both triangles \|/ (no duplication needed) 3
💡 Why Indices Matter

Without indexing, you'd need to duplicate vertices for every triangle corner. A cube with 6 faces (12 triangles) would need 36 vertices instead of 8. Index buffers save massive amounts of memory.

MeshFilter vs MeshRenderer

This is a common interview question. These two components work together but have distinct roles:

COMPONENT RELATIONSHIP: ┌─────────────────────────────────────────────────────────────────┐ │ GameObject │ ├─────────────────────────────────────────────────────────────────┤ │ │ │ ┌─────────────────┐ ┌──────────────────────┐ │ │ │ MeshFilter │────────▶│ MeshRenderer │ │ │ │ │ │ │ │ │ │ Holds the │ │ Renders the mesh │ │ │ │ MESH DATA │ │ using MATERIAL(s) │ │ │ │ (geometry) │ │ │ │ │ │ │ │ Bounds, shadows, │ │ │ │ .mesh │ │ lighting probes, │ │ │ │ .sharedMesh │ │ layer, static flags │ │ │ └─────────────────┘ └──────────────────────┘ │ │ │ └─────────────────────────────────────────────────────────────────┘
MeshFilter MeshRenderer
Contains reference to Mesh asset How to draw (materials, settings)
No visual effect if alone Requires MeshFilter to have geometry
.mesh (instance copy) .materials[] (one per submesh)
.sharedMesh (shared asset) .bounds (for culling)
⚠️ .mesh vs .sharedMesh

Accessing meshFilter.mesh creates an instance copy — useful for runtime modification but adds memory. Use .sharedMesh when you don't need to modify it (read-only).

SubMeshes

A single mesh can have multiple submeshes, each rendered with a different material. This is how multi-material objects work.

SUBMESH EXAMPLE: Car Body Mesh: ├── SubMesh 0: Body panels → Material[0]: Car Paint ├── SubMesh 1: Windows → Material[1]: Glass ├── SubMesh 2: Chrome accents → Material[2]: Chrome └── SubMesh 3: Tires → Material[3]: Rubber One GameObject, one MeshRenderer, 4 materials renderer.materials.Length == 4
💚 VRify Relevance

CAD data often comes with many materials. Each submesh = separate material = separate draw call. For performance, consider combining materials or using texture atlases to reduce submesh count.

How Mesh Data Gets to the GPU

Understanding the data flow helps explain performance characteristics:

MESH DATA FLOW: ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ CPU Memory │ → │ GPU Upload │ → │ GPU VRAM │ │ │ │ (one time) │ │ │ │ Mesh asset │ │ │ │ Vertex │ │ vertices[] │ │ │ │ Buffer │ │ indices[] │ │ │ │ │ │ normals[] │ │ │ │ Index │ │ uvs[] │ │ │ │ Buffer │ └──────────────┘ └──────────────┘ └──────────────┘ Static mesh: Uploaded once, stays in VRAM Dynamic mesh: Re-uploaded when modified

Static vs Dynamic Meshes

Static Meshes Dynamic Meshes
Uploaded once at load time Re-uploaded when data changes
Marked as read-only on GPU Kept in CPU-writeable memory
Eligible for static batching Can use dynamic batching (if small)
Most scene geometry Procedural meshes, deforming objects

Bounds and Culling

Every renderer has bounds — an axis-aligned bounding box (AABB) used for culling and spatial queries.

MESH BOUNDS: ┌─────────────────┐ │ │ Axis-Aligned Bounding Box (AABB) ┌────┼───────┐ │ - Simple rectangle/box │ │ ▲ │ │ - Fast intersection tests │ └────┼──┼─────────┘ - May be larger than actual mesh │ │ │ └─────────┴──┘ Actual mesh (rotated) AABB expands when mesh rotates!

Key points:

  • Renderer.bounds — world-space bounds (affected by transform)
  • Mesh.bounds — local-space bounds (mesh-relative)
  • Bounds are used for frustum culling — object skipped if bounds outside camera
  • Wrong bounds = visible objects being culled (pop-in issues)

Mesh Topology

Unity supports different primitive types, though triangles are most common:

Topology Use Case
Triangles Standard 3D geometry (default)
Lines Debug visualization, wireframes
Points Point clouds, particle systems
Quads Legacy, avoid (triangulated internally)
💚 VRify Relevance

Point cloud data naturally uses Points topology. However, WebGL rendering of points can be inconsistent. Often point clouds are rendered as small quads (billboards) or converted to meshes for better cross-platform behavior.

Creating Meshes at Runtime

For procedural geometry, you can create meshes via code:

Mesh mesh = new Mesh();

// Define vertices
mesh.vertices = new Vector3[] {
    new Vector3(0, 0, 0),
    new Vector3(1, 0, 0),
    new Vector3(0, 1, 0)
};

// Define triangle (indices into vertex array)
mesh.triangles = new int[] { 0, 1, 2 };

// Calculate normals automatically
mesh.RecalculateNormals();
mesh.RecalculateBounds();

// Assign to MeshFilter
GetComponent().mesh = mesh;

Key Takeaways

  • A mesh is geometry data: vertices (with attributes) + triangle indices
  • Vertex attributes: position, normal, UV, tangent, color, bone weights
  • Index buffer enables vertex sharing, saving memory
  • MeshFilter holds the mesh data; MeshRenderer draws it with materials
  • Use .sharedMesh for read-only access, .mesh creates a copy
  • SubMeshes allow multiple materials per mesh (one draw call each)
  • Bounds determine culling — incorrect bounds cause visual issues
  • Static meshes upload once; dynamic meshes re-upload on change
