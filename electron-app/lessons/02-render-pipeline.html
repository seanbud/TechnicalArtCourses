Lesson 02

Render Pipeline Overview

From geometry collection to pixels on screen

The Journey of a Triangle

Every frame, Unity must figure out what to draw and how to draw it. This is a multi-stage process that transforms 3D scene data into a 2D image on your screen.

Stage 1: Culling — What Can We Skip?

Before any actual rendering, Unity determines what doesn't need to be drawn. This is critical for performance.

Frustum Culling

The camera's view frustum is a pyramid-shaped volume representing what the camera can see. Any object completely outside this volume is skipped.

Near Plane ┌─┐ / \ / \ Objects outside / \ the frustum are / 👁 \ NOT rendered / Camera \ / \ / \ └───────────────────┘ Far Plane
🧠 Bounding Volumes

Unity uses bounding boxes (AABB) for fast frustum tests. It doesn't check every triangle — just whether the object's box intersects the frustum. This is why Renderer.bounds matters for culling accuracy.

Occlusion Culling

Even if an object is in the frustum, it might be hidden behind another object. Occlusion culling skips objects that are completely blocked from view.

  • Requires baked occlusion data (Window → Rendering → Occlusion Culling)
  • Uses a cell-based system to determine visibility
  • Most beneficial for interior scenes with walls/rooms
  • Not automatic — you must set up occluders and bake the data

Stage 2: Collecting Renderable Objects

After culling, Unity loops through remaining objects with Renderer components (MeshRenderer, SkinnedMeshRenderer, SpriteRenderer, etc.) and collects them for rendering.

Render Queue Sorting

Objects are sorted by their material's render queue value:

Queue Value Purpose
Background 1000 Skyboxes, backgrounds
Geometry 2000 Opaque objects (default)
AlphaTest 2450 Alpha-cutout (grass, leaves)
Transparent 3000 Transparent objects
Overlay 4000 UI, lens flares
💡 Sorting Strategy

Opaque objects: Rendered front-to-back (closest first). Why? Early-Z rejection — fragments behind already-drawn pixels are skipped by the GPU.

Transparent objects: Rendered back-to-front (farthest first). Why? Alpha blending requires knowing what's behind to blend correctly.

Stage 3: Batching — The "Changing Brushes" Metaphor

Here's where your interview knowledge really matters. Every time Unity tells the GPU to draw something, it issues a draw call. Think of it like this:

🎨 The Painter Metaphor

Imagine a painter who must change brushes for different paint types. Each brush swap takes time. If you paint 100 red strokes, then 100 blue strokes — that's only 2 brush swaps. But alternating red-blue-red-blue for 200 strokes means 200 swaps. Same amount of paint, vastly different efficiency.

Materials trigger "brush changes." Each unique material (or material instance with different properties) typically requires a new draw call.

Types of Batching

Static Batching

  • Combines meshes marked as Static at build time
  • Creates one big mesh in memory (uses more RAM)
  • Objects cannot move at runtime
  • Same material required

Dynamic Batching

  • Combines small meshes at runtime (< 300 vertices)
  • CPU overhead to combine each frame
  • Only works with same material
  • Often disabled because overhead > benefit

GPU Instancing

  • Draws multiple copies of the same mesh in one call
  • Each instance can have different transform and properties
  • Material must support instancing (checkbox in inspector)
  • Great for forests, crowds, debris

SRP Batcher (URP/HDRP)

  • Doesn't reduce draw calls — reduces CPU work per draw call
  • Caches material data on GPU, skips re-uploading
  • Works with different materials (of compatible shaders)
  • This is the modern approach — enable in URP settings

Worst-Case Scenario: The Draw Call Explosion

BAD: 100 GameObjects, 100 different materials = 100+ draw calls Scene: ├── Rock_01 → Material_A ├── Rock_02 → Material_B (different texture!) ├── Rock_03 → Material_C (different color!) ├── Crate_01 → Material_D ├── Crate_02 → Material_E ... GOOD: 100 GameObjects, 1 texture atlas, 1 material = ~1-5 draw calls Scene: ├── Rock_01 → SharedMaterial (atlas UV: 0,0) ├── Rock_02 → SharedMaterial (atlas UV: 0.5,0) ├── Rock_03 → SharedMaterial (atlas UV: 0,0.5) ├── Crate_01 → SharedMaterial (atlas UV: 0.5,0.5) ...
💚 VRify Relevance

With 250 clients sending uncoordinated data, expect material chaos. Part of your pipeline should consolidate materials — combine textures into atlases, standardize shaders. Garbage in → optimized out.

Stage 4: GPU Execution

Once Unity has prepared the draw calls, it sends commands to the GPU. This is the graphics pipeline we'll cover in detail in Lesson 3 (Shaders), but here's the overview:

SIMPLIFIED GPU PIPELINE: ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ VERTEX │ → │ RASTERIZE │ → │ FRAGMENT │ → │ OUTPUT │ │ SHADER │ │ (to pixels)│ │ SHADER │ │ BLENDING │ └─────────────┘ └─────────────┘ └─────────────┘ └─────────────┘ Transform Determine Calculate Combine with vertices to which pixels pixel color framebuffer screen space are covered per fragment

Quick GPU Terminology

  • Vertex: A point in 3D space with attributes (position, normal, UV)
  • Fragment: A potential pixel (after rasterization, before final output)
  • Depth Buffer (Z-Buffer): Stores depth per pixel; used to determine what's in front
  • Depth Test: Compare fragment depth to buffer; skip if behind existing pixel
  • Framebuffer: The final 2D image being constructed

Stage 5: Camera and Final Output

Cameras define what renders and where it goes:

  • Target Texture: Can render to a RenderTexture instead of screen (reflections, portals, minimaps)
  • Depth: Camera render order; higher depth renders later (on top)
  • Clear Flags: What to clear before rendering (skybox, solid color, don't clear)
  • Culling Mask: Which layers this camera renders

Stacked Cameras (URP)

URP allows camera stacking — a Base Camera renders the scene, then Overlay Cameras add on top without clearing:

Base Camera (Depth 0): Renders 3D world ↓ Overlay Camera (Depth 1): Renders UI layer ↓ Final composite image

The Statistics Window

Use Window → Analysis → Frame Debugger and the Game view Stats panel to understand rendering:

  • Batches: Number of draw calls (lower is better)
  • SetPass calls: Material/shader switches (lower is better)
  • Triangles/Vertices: Geometry complexity
  • Saved by batching: How many draws were combined

Key Takeaways

  • Culling (frustum + occlusion) determines what to skip before any rendering
  • Objects are sorted by render queue — opaque front-to-back, transparent back-to-front
  • Draw calls are expensive; minimize them via batching and material consolidation
  • The "changing brushes" metaphor: group objects by material to avoid constant material switches
  • SRP Batcher is the modern URP approach — reduces CPU overhead rather than draw call count
  • Use Frame Debugger to understand exactly what's being drawn and why
