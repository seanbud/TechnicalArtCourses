Lesson 08

UI Rendering

Canvas modes, batching, overdraw, and optimization strategies

UI is Different

Unity UI (uGUI) uses a completely different rendering system than 3D geometry. Understanding this is crucial for performance, especially for browser targets where UI can easily become the bottleneck.

🧠 Key Difference from 3D

3D uses MeshRenderer. UI uses CanvasRenderer. The Canvas component collects all its children, generates meshes, and submits them to the GPU. This happens on the main thread.

Canvas Render Modes

Every UI element must be under a Canvas. The Canvas's render mode determines how it's drawn:

CANVAS RENDER MODES: ┌─────────────────────────────────────────────────────────────────┐ │ SCREEN SPACE - OVERLAY │ ├─────────────────────────────────────────────────────────────────┤ │ • Renders AFTER all 3D (always on top) │ │ • Ignores cameras entirely │ │ • Resolution: matches screen pixels exactly │ │ • Use for: HUD, menus, always-visible UI │ └─────────────────────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────────────────────┐ │ SCREEN SPACE - CAMERA │ ├─────────────────────────────────────────────────────────────────┤ │ • Rendered by a specific camera │ │ • Can be placed at specific distance (allows 3D between UI) │ │ • Resolution: affected by camera settings │ │ • Use for: UI that needs post-processing, depth interaction │ └─────────────────────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────────────────────┐ │ WORLD SPACE │ ├─────────────────────────────────────────────────────────────────┤ │ • UI exists as a 3D object in the scene │ │ • Has a RectTransform defining size in world units │ │ • Affected by perspective, occlusion, lighting │ │ • Use for: Labels on objects, VR/AR UI, diegetic UI │ └─────────────────────────────────────────────────────────────────┘
💚 VRify Relevance

For geospatial visualization, you'll likely use World Space for labels/markers on mine sites, and Screen Space Overlay for tools/controls. Users who "hardly know how to use an iPad" need visible, clear UI — consider larger touch targets.

How Canvas Rendering Works

The Canvas doesn't render elements individually. It batches UI elements into combined meshes:

CANVAS BATCH GENERATION: Canvas ├── Button 1 (Image + Text) ├── Button 2 (Image + Text) ├── Panel (Image) └── Label (Text) Step 1: Collect all CanvasRenderer elements Step 2: Sort by depth, material, texture Step 3: Generate combined meshes (batches) Step 4: Submit to GPU If all use same atlas → 1-2 draw calls If all use different textures → 1 draw call each

What Breaks Batching

Breaks Batch Why Solution
Different textures Can't combine in one draw Use texture atlases/sprite sheets
Different materials Different shader/settings Share materials when possible
Different masking RectMask2D boundaries Group masked UI together
Depth interleaving Z-order interrupts grouping Organize hierarchy by material

The Hierarchy Matters

Canvas processes children in hierarchy order (depth). If elements using different materials are interleaved, batching breaks:

BAD (5 draw calls): GOOD (2 draw calls): Canvas Canvas ├── Image_A (Material 1) ├── Image_A (Material 1) ├── Image_B (Material 2) ← break ├── Image_C (Material 1) ├── Image_C (Material 1) ← break ├── Image_E (Material 1) ├── Image_D (Material 2) ← break ├── Image_B (Material 2) └── Image_E (Material 1) ← break └── Image_D (Material 2) Same elements, different order = different performance

Canvas Rebuild: The Hidden Cost

When anything in a Canvas changes, the entire Canvas may need to rebuild its batched meshes. This is expensive.

⚠️ Rebuild Triggers

These actions trigger a Canvas rebuild:
• Enabling/disabling UI elements
• Changing text content
• Changing Image sprite
• Modifying RectTransform (position, size, anchors)
• Adding/removing elements from hierarchy

Multiple Canvases Strategy

The solution: separate dynamic and static elements into different Canvases.

SINGLE CANVAS (bad): MULTIPLE CANVASES (good): Canvas StaticCanvas (rarely changes) ├── Background (static) ├── Background ├── Health Bar (updates often) └── Ability Icons ├── Ability Icons (static) ├── Damage Numbers (spawning) DynamicCanvas (updates often) └── Minimap (updates often) ├── Health Bar └── Minimap Any change rebuilds the entire canvas DamageNumberCanvas (pooled) └── Damage Numbers Each canvas has isolated rebuild

Overdraw: The Silent Killer

Overdraw occurs when pixels are drawn multiple times. UI is particularly prone to this because elements often overlap.

OVERDRAW EXAMPLE: ┌────────────────────────────────┐ │ Background Panel │ ← Pixel drawn once │ ┌──────────────────────────┐ │ │ │ Inner Panel │ │ ← Same pixel drawn AGAIN │ │ ┌────────────────────┐ │ │ │ │ │ Button │ │ │ ← Same pixel drawn 3rd time │ │ │ ┌──────────────┐ │ │ │ │ │ │ │ Icon │ │ │ │ ← Same pixel drawn 4th time │ │ │ └──────────────┘ │ │ │ │ │ └────────────────────┘ │ │ │ └──────────────────────────┘ │ └────────────────────────────────┘ This center pixel is drawn 4 times! With transparency: all 4 layers must calculate

Reducing Overdraw

  • Disable Raycast Target on decorative elements (reduces event processing too)
  • Use 9-slice sprites instead of full images when possible
  • Avoid full-screen overlays — use vignettes or partial coverage
  • Eliminate unnecessary backgrounds — if a panel is always covered, remove it
  • Use the Frame Debugger to visualize what's actually being drawn
🧠 Overdraw Visualization

In Scene view, use the Overdraw visualization mode (dropdown at top left). Brighter = more overdraw. Aim for mostly dark (1-2 draws per pixel).

World Space UI Optimization

World Space canvases require special consideration:

The Anchoring Strategy

Instead of having many small World Space canvases (one per label), consider:

MANY CANVASES (expensive): ANCHORED APPROACH (better): Each label = separate Canvas One World Space Canvas ├── Label1 Canvas ├── Container at anchor 1 ├── Label2 Canvas │ └── Label1 element ├── Label3 Canvas ├── Container at anchor 2 ├── ... │ └── Label2 element └── Container at anchor 3 Each canvas = rebuild overhead └── Label3 element Containers are empty GameObjects at world positions. Labels parent to containers but batch together.

This approach:

  • Reduces canvas count (fewer updates)
  • Enables batching (same canvas = can batch)
  • Maintains world positioning (via parent container positions)

Disable Canvas When Obscured

World Space UI doesn't auto-cull based on visibility. If labels are behind the camera or too far away, disable the Canvas component (not the GameObject) to stop rendering while preserving layout.

CanvasRenderer vs MeshRenderer

CanvasRenderer (UI) MeshRenderer (3D)
Managed by parent Canvas Independent rendering
Canvas generates combined mesh Each has own mesh
Main thread mesh building Static at load time
2D rendering (mostly) Full 3D with all features
No shadows, limited lighting Full lighting support

Text Rendering

Text is particularly expensive. Each character = geometry. Use TextMeshPro (TMPro) for better quality and performance:

  • Legacy Text: Bitmap fonts, blurry at large sizes, limited features
  • TextMeshPro: SDF (Signed Distance Field) fonts, crisp at any size, outlines/effects
💚 VRify Relevance

For users who "hardly know how to use an iPad," clear, readable text is essential. TextMeshPro with proper font sizes and contrasting colors will make UI more accessible. Consider larger buttons and clear iconography.

Key Takeaways

  • UI uses CanvasRenderer, not MeshRenderer — entirely different system
  • Three render modes: Overlay (always on top), Camera (rendered by camera), World Space (in 3D scene)
  • Canvas batches elements — same texture/material can combine into one draw
  • Different textures or materials in interleaved order break batching
  • Canvas rebuild is expensive — happens when any element changes
  • Use multiple canvases: separate static from dynamic elements
  • Overdraw is a major UI cost — avoid stacked transparent panels
  • For World Space UI, use the anchoring strategy to enable batching
  • Use TextMeshPro for text — better quality and performance
