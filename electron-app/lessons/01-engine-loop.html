Lesson 01

Unity Engine Loop

Game loop architecture, Update vs FixedUpdate, and frame timing

The Big Picture

Unity's engine is built around a deterministic execution loop that runs every frame. Understanding this loop is essential because it determines when your code executes and why certain patterns exist (like using FixedUpdate for physics).

The PlayerLoop System

Under the hood, Unity uses a PlayerLoop system that defines the exact order of operations. Think of it as a recipe that Unity follows every frame:

🧠 Key Insight

The PlayerLoop isn't just "call Update, then render." It's a deeply nested structure with dozens of subsystems (physics, animation, rendering) that execute in a precise order. You can even modify this loop programmatically if needed.

Simplified Execution Order

  1. Initialization — Awake(), OnEnable(), Start() (first frame only)
  2. Physics — FixedUpdate() (may run 0, 1, or multiple times)
  3. Input Events — Mouse, keyboard, touch polling
  4. Game Logic — Update()
  5. Animation — Animator updates, IK
  6. Late Game Logic — LateUpdate()
  7. Rendering — Culling, draw calls, post-processing
  8. End of Frame — Coroutine yields, cleanup

Update vs FixedUpdate: The Core Distinction

This is one of the most misunderstood concepts. Let's derive why they exist from first principles:

The Problem with Variable Frame Rates

Update() runs once per frame. If your game runs at 60 FPS, Update runs 60 times per second. At 30 FPS, it runs 30 times. The time between calls (deltaTime) varies.

⚠️ Why This Matters for Physics

Physics simulations need determinism. If you apply forces in Update(), a player on a slower computer will experience different physics behavior than someone on a fast computer. The same inputs could produce different results!

FixedUpdate: The Solution

FixedUpdate() runs on a fixed timestep (default: 0.02 seconds = 50Hz). Unity's physics engine (PhysX) needs this regularity to produce consistent simulations.

FRAME TIMING EXAMPLE (target 60 FPS, Fixed timestep 0.02s): Frame 1: Δt = 0.016s ├── FixedUpdate (catches up from 0.000 to 0.016) │ └── Runs 0 times (accumulated time < 0.02) └── Update Frame 2: Δt = 0.018s (total: 0.034s) ├── FixedUpdate │ └── Runs 1 time (0.034 > 0.02, now at 0.014 remaining) └── Update Frame 3: Δt = 0.025s (remaining 0.014 + 0.025 = 0.039) ├── FixedUpdate │ └── Runs 1 time (0.039 > 0.02, now at 0.019 remaining) └── Update Frame 4: Δt = 0.040s (big hitch! remaining 0.019 + 0.040 = 0.059) ├── FixedUpdate │ └── Runs 2 times (0.059 → 0.039 → 0.019) └── Update
💡 The Catch-Up Mechanism

If a frame takes too long, FixedUpdate will run multiple times to catch up. This maintains physics consistency but can cause a "spiral of death" if updates can't keep up. Unity has a Maximum Allowed Timestep setting to prevent this.

When to Use Each

Use FixedUpdate() Use Update() Use LateUpdate()
Rigidbody forces/velocity Input handling Camera follow
Physics raycasts Non-physics movement After animation adjustments
Anything needing consistency UI updates Anything depending on Update results

Frame-Rate Independence

For movement in Update(), multiply by Time.deltaTime to achieve frame-rate independence:

// Without deltaTime: moves 1 unit per FRAME (faster on fast machines)
transform.position += Vector3.forward;

// With deltaTime: moves 1 unit per SECOND (consistent everywhere)
transform.position += Vector3.forward * Time.deltaTime;

For FixedUpdate(), use Time.fixedDeltaTime (though it's usually constant).

Main Thread vs Render Thread

Unity runs game logic on the main thread, but rendering can partially happen on a separate render thread. The key points:

  • Main thread: All your MonoBehaviour code, physics, most engine systems
  • Render thread: GPU command submission, some graphics work
  • Job System: Worker threads for parallelizable tasks (Burst, DOTS)
💚 VRify Relevance

For browser targets, understanding the main thread is critical. JavaScript in browsers is single-threaded, and Unity WebGL also has threading limitations. Heavy Update() logic can freeze the entire page.

Key Takeaways

  • The PlayerLoop defines a precise execution order for all Unity systems
  • FixedUpdate exists for physics determinism — it runs on a fixed timestep, possibly multiple times per frame
  • Update runs once per frame with variable deltaTime
  • LateUpdate runs after Update and animations — perfect for cameras
  • Always multiply movement by deltaTime in Update for frame-rate independence
  • Heavy main-thread work blocks everything, especially critical for WebGL
