Lesson 09

3D File Formats & Data Ingestion

CAD, DXF, point clouds, and coordinate systems

The VRify Challenge

With 250+ clients sending data in different formats without coordination, you're dealing with the classic "garbage in, garbage out" problem. Understanding file formats is essential for building robust ingestion pipelines.

💡 Core Principle

The goal isn't just to import data — it's to normalize it. Different formats, coordinate systems, units, and structures must all convert to a consistent Unity-friendly format.

CAD Formats

CAD (Computer-Aided Design) formats are different from game/visualization formats. They prioritize precision over rendering efficiency.

Format Origin Characteristics Import Approach
STEP (.stp/.step) ISO Standard Solid models, NURBS surfaces Convert via middleware (PiXYZ, UAT)
IGES (.igs/.iges) Legacy Standard Surfaces and curves, older CAD Convert externally
DWG AutoCAD Proprietary, 2D/3D drawings Export to FBX or use SDK
DXF AutoCAD Exchange Open format version of DWG Parse directly or convert
🧠 CAD vs Game Geometry

CAD uses NURBS (smooth mathematical surfaces) and exact curves. Games need triangulated meshes. The conversion process (tessellation) can explode polygon counts if not controlled. A simple mechanical part might become millions of triangles!

DXF Deep Dive

You mentioned DXF specifically (and topography lines as closed loops). DXF is particularly relevant for mining/geospatial data.

What's in a DXF File

  • Points — individual coordinates
  • Lines — straight segments between two points
  • Polylines — connected sequences of lines (open or closed)
  • LWPolylines — lightweight polylines (2D)
  • Circles, Arcs, Ellipses — curve primitives
  • 3DFaces — triangular/quad surfaces
  • Text, Dimensions — annotations
  • Blocks — reusable component definitions

Topography Contour Lines

Topography data typically comes as closed polylines at specific elevations:

CONTOUR LINES (overhead view): Elevation 100m: ─────────────╲ Elevation 110m: ──────────╲ Elevation 120m: ────────╲ Elevation 130m: ─────╲ ╲ Peak Each contour = closed loop polyline All points on that line are at same elevation

Processing DXF for Unity

  1. Parse the DXF structure (entities, layers, blocks)
  2. Extract polylines and their elevation data
  3. Convert coordinates (see Coordinate Systems below)
  4. Either: Generate mesh from contours, or render as lines
💚 VRify Relevance

Contour lines can be rendered as LineRenderer components, or you can triangulate them into terrain meshes. Libraries like netDxf (C#) can parse DXF files directly in Unity.

Point Cloud Formats

Point clouds are fundamental to mining/surveying — LIDAR scans, drone photogrammetry, etc.

Format Characteristics Handling
LAS/LAZ LIDAR standard, LAZ is compressed LASzip libraries, stream processing
PLY Polygon File Format, simple Easy to parse, often from photogrammetry
E57 ASTM standard, structured scans Complex, handles scanner metadata
XYZ/PTS Plain text coordinates Simple parsing, large files
Potree Hierarchical octree format Streaming-ready, level-of-detail

Point Cloud Challenges

  • Scale: Millions to billions of points — can't render all at once
  • Rendering: Unity's point primitives are limited; often use billboards or voxels
  • LOD: Need spatial data structures (octree) for progressive loading
  • Memory: Browser targets have strict limits

Coordinate Systems

This is where mining/geospatial data gets tricky. Real-world coordinates don't match Unity's coordinate system.

COORDINATE SYSTEM DIFFERENCES: Unity (Left-handed, Y-up): Geographic (varies): Y - WGS84 (lat/long/altitude) │ - UTM zones (meters) │ - Local mining grids │_____ X - State Plane coordinates / Z Issues: - Unity uses meters but limited float precision (~6 decimal digits) - Real-world coordinates can be HUGE (millions of meters from origin) - Y/Z axis flipping between systems - Different units (feet, meters, survey feet)

Origin Shifting

The most important technique for geospatial data:

💡 Float Precision Problem

Unity uses 32-bit floats. At 1 million meters from origin, you lose ~0.1m precision. At 10 million meters, wobbling and jitter become visible. Solution: shift the origin to keep everything within ~10km of (0,0,0).

// Example: Shift real-world coordinates to Unity
Vector3 worldOrigin = new Vector3(500000, 0, 4500000); // UTM coords

Vector3 ToUnityPosition(Vector3 realWorld) {
    return new Vector3(
        realWorld.x - worldOrigin.x,
        realWorld.y,  // elevation
        realWorld.z - worldOrigin.z
    );
}

Data Validation Strategies

With 250 clients, you need automated validation:

  • Format Detection: Check file headers, don't trust extensions
  • Bounds Checking: Are coordinates reasonable for Earth?
  • Unit Detection: Heuristics for feet vs meters (building heights, etc.)
  • Topology Validation: Are meshes watertight? Any degenerate triangles?
  • Orientation Check: Is Y up or Z up?
💚 VRify Relevance

Build a validation pipeline that runs before import. Generate reports listing issues found. Let the ingestion system either auto-fix common problems or queue for manual review.

Key Takeaways

  • FBX and OBJ are Unity's native formats — CAD formats need conversion
  • DXF contains polylines, points, curves — contour lines are closed polylines at elevations
  • CAD geometry uses NURBS; must tessellate to triangles (control polygon count!)
  • Point clouds need LOD systems and spatial structures for large datasets
  • Coordinate systems differ between geographic and Unity — shift origin to avoid precision loss
  • Float precision breaks down far from origin — keep geometry within ~10km of (0,0,0)
  • Automate validation — with 250 clients, manual checking isn't feasible
