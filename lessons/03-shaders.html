Shader Fundamentals

What shaders are, how they're structured, and when each part runs

What Is a Shader, Really?

A shader is a program that runs on the GPU to determine how something looks. But here's the key insight that often confuses people:

🧠 Not Just "Per Pixel"

A "shader" isn't just one program — it's a collection of programs (called stages or passes) that run at different points in the graphics pipeline. The vertex shader runs per-vertex. The fragment shader runs per-fragment. They're separate programs bundled together.

The Graphics Pipeline Stages

Vertex Shader

Runs once per vertex. Its main job is to transform vertex positions from object space to clip space (the coordinate system used for rendering).

  • Input: Vertex attributes (position, normal, UV, tangent, color)
  • Output: Transformed position + data to pass to fragment shader
  • Common uses: Skinning, vertex animation, world-space calculations

Rasterization (Fixed Function)

This is not programmable — the GPU hardware does this. It takes your transformed triangles and figures out which pixels they cover, generating fragments.

🧠 Fragment ≠ Pixel

A fragment is a potential pixel. Multiple fragments can compete for the same pixel position (overlapping geometry). After depth testing and blending, the winning fragment becomes the final pixel.

Fragment Shader (Pixel Shader)

Runs once per fragment. This is where you calculate the final color, including lighting, texturing, and effects.

  • Input: Interpolated values from vertex shader (UVs, normals, world position)
  • Output: Color (RGBA) and optionally modified depth
  • This is usually the performance bottleneck — it runs millions of times per frame

Geometry Shader (Optional)

Runs after vertex shader, can add, remove, or modify primitives (points, lines, triangles). Rarely used because:

  • Performance is often poor (not well parallelized)
  • Compute shaders are usually a better choice for geometry generation
  • Some mobile GPUs don't support them

Tessellation (Optional)

Subdivides geometry into more triangles. Consists of Hull Shader → Tessellator → Domain Shader. Used for:

  • Displacement mapping (adding detail based on heightmaps)
  • LOD-independent smooth surfaces
  • Not commonly used in real-time games due to performance cost

Shader Structure in Unity

Unity shaders are written in ShaderLab format, which wraps HLSL code. Here's the conceptual structure:

Shader "Custom/MyShader"
{
    Properties
    {
        // Material inspector properties
        _MainTex ("Texture", 2D) = "white" {}
        _Color ("Color", Color) = (1,1,1,1)
    }
    
    SubShader
    {
        Tags { "RenderType"="Opaque" "Queue"="Geometry" }
        
        Pass
        {
            // Render state (blending, depth, etc.)
            ZWrite On
            
            HLSLPROGRAM
            #pragma vertex vert
            #pragma fragment frag
            
            // Your HLSL code here
            // vert() = vertex shader
            // frag() = fragment shader
            ENDHLSL
        }
        
        // Additional passes for shadows, outlines, etc.
        Pass { /* shadow pass */ }
    }
    
    Fallback "Diffuse"
}

Key Structural Concepts

Element Purpose
Properties Variables exposed in the Material Inspector
SubShader A rendering recipe; Unity picks the first one that works on the target hardware
Tags Metadata for render queue, pipeline compatibility
Pass One complete render of the geometry (multiple passes = multiple renders)
Fallback Shader to use if none of the SubShaders work

Multi-Pass Rendering

A shader can have multiple passes. Each pass renders the geometry again with different settings. Common uses:

  • Outline Pass: Render the object slightly larger in a solid color, then render normally on top
  • Shadow Pass: Special pass for shadow map generation
  • Depth Pre-Pass: Write only depth first, then render color (optimization)
⚠️ Performance Consideration

Each pass means the geometry is processed again. A 2-pass shader on 1000 objects = 2000 draw calls worth of vertex processing. Keep pass count minimal for performance-critical scenarios (like browser targets).

Shader Compilation

Shaders go through multiple compilation stages:

SHADER COMPILATION PIPELINE: ┌──────────────────┐ ┌──────────────────┐ ┌──────────────────┐ │ ShaderLab + │ → │ Intermediate │ → │ Platform GPU │ │ HLSL Source │ │ Bytecode │ │ Instructions │ └──────────────────┘ └──────────────────┘ └──────────────────┘ Your code At build time At build time OR first use (runtime)

Build-Time Compilation

When you build your game, Unity compiles shaders for the target platform(s). This produces shader variants — different versions for different keyword combinations.

💡 Variant Explosion

If your shader has 5 #pragma multi_compile keywords with 2 options each, you get 2^5 = 32 variants. This multiplies build time and size. Use shader_feature (strips unused) over multi_compile (includes all) when possible.

Editor Compilation

In the Unity Editor, shaders are compiled on-demand for your current graphics API. This is why the first time you view a new shader, there may be a brief pause.

Runtime Compilation (Warmup)

Some variants compile when first used in-game, causing shader stuttering. Solutions:

  • ShaderVariantCollection — preload expected variants at startup
  • Loading screens that force-use all shaders
  • Minimize variant counts in build settings

Shader Graph vs HLSL

Shader Graph Hand-Coded HLSL
Pros Visual, fast iteration, no syntax errors Full control, can optimize, custom features
Cons Can generate bloated code, limited to what nodes exist Steeper learning curve, easy to make mistakes
Output Generates HLSL behind the scenes Your HLSL is used directly

For your role, understanding both is valuable — Shader Graph for rapid prototyping, HLSL knowledge for understanding and debugging the generated code.

Compute Shaders: A Different Beast

Compute shaders are not part of the graphics pipeline. They're general-purpose GPU programs for parallel computation.

  • Don't render anything directly — output to buffers/textures
  • Used for: particle systems, procedural geometry, physics, image effects
  • Work in thread groups (e.g., 64 threads at a time)
  • Can generate geometry that's then used by regular shaders
💚 VRify Relevance

Compute shaders could be valuable for point cloud processing — millions of points can be transformed, filtered, or rendered using compute-to-vertex pipelines. However, WebGL 2.0 has limited compute support; check platform constraints.

Key Takeaways

  • A "shader" is actually multiple programs (vertex, fragment, optionally geometry/tessellation) bundled together
  • Vertex shader: runs per-vertex, transforms positions
  • Fragment shader: runs per-pixel, calculates color — usually the bottleneck
  • Unity shaders use ShaderLab format wrapping HLSL code
  • Passes: each pass renders geometry again; multiple passes have performance cost
  • Variants: different compiled versions for keyword combinations — watch for explosion
  • Compute shaders: general GPU compute, separate from rendering pipeline
