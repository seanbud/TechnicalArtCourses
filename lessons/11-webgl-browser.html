Lesson 11

WebGL & Browser Optimization

Memory limits, texture compression, and load time strategies

WebGL: A Different World

Building for browser is fundamentally different from desktop. Many assumptions don't hold, and constraints are much tighter.

⚠️ Key Constraints

• Single-threaded: Main thread handles everything (no Job System benefits)
• Memory limits: ~2GB max, often less in practice
• No compute shaders: WebGL 2.0 doesn't support them
• Initial download: Users wait for entire build to load
• Shader compilation: Happens in the browser, can cause stalls

Memory Budget

Browser memory management is tricky. Unlike desktop, there's no dedicated GPU memory — it's all shared with the browser process.

MEMORY BREAKDOWN (example 500MB budget): ┌─────────────────────────────────────────────────────────────────┐ │ Unity WASM heap │ ~150-200MB │ Code, managed objects │ ├───────────────────────────────────────────────────────────────────┤ │ Browser overhead │ ~50-100MB │ DOM, JS, browser │ ├───────────────────────────────────────────────────────────────────┤ │ GPU resources (shared) │ ~150-250MB │ Textures, buffers │ ├───────────────────────────────────────────────────────────────────┤ │ Audio, networking │ ~20-50MB │ Decoded audio, buffers │ └─────────────────────────────────────────────────────────────────┘ Exceeding memory triggers browser tab crash or reload!

Texture Memory Dominates

Textures are the biggest memory consumer. Calculate carefully:

Resolution RGBA Uncompressed With Mipmaps (×1.33)
256×256 256 KB 341 KB
512×512 1 MB 1.33 MB
1024×1024 4 MB 5.33 MB
2048×2048 16 MB 21.3 MB
4096×4096 64 MB 85.3 MB
💡 The 4K Texture Trap

A single 4K texture uses 64MB. Ten of them = 640MB — more than your entire budget! Use 1K or 2K max for most things. Reserve higher resolutions only for critical hero assets.

Texture Compression

GPU texture compression is essential. It reduces both download size and GPU memory.

Format WebGL Support Quality Compression Ratio
ETC2 ✅ WebGL 2.0 Good 6:1
ASTC ⚠️ Extension needed Excellent Variable (up to 36:1)
DXT/BC ⚠️ Extension (S3TC) Good 4:1 to 6:1
Basis Universal ✅ Transcodes to native Good Varies by target
💚 VRify Recommendation

Use ETC2 as your baseline — it's guaranteed in WebGL 2.0. For assets that must look perfect, provide ASTC with ETC2 fallback. Unity handles this with platform-specific overrides.

Initial Load Time

Users must download your entire build before anything runs. This is critical UX.

Build Size Breakdown

TYPICAL WEBGL BUILD: ├── Build/ │ ├── WebGL.data ← Assets (biggest, 50-90% of total) │ ├── WebGL.wasm ← Code (~10-30MB typical) │ ├── WebGL.framework.js ← Unity runtime │ └── WebGL.loader.js ← Bootstrap │ Total: 30-300MB+ depending on content

Size Reduction Strategies

  • Compression: Enable Brotli or gzip (server must support)
  • Strip Engine Code: Managed code stripping in Player Settings
  • Texture Compression: Crunch compression for RGB textures
  • Audio Compression: Use MP3/Vorbis, not WAV
  • Mesh Compression: Enable in import settings
  • Remove Unused: Resources folder includes everything!

Asset Bundles / Addressables

For large applications, don't include everything in the main build:

STREAMING ASSETS APPROACH: Initial Load (small): ├── Core UI and loading screen ├── Essential shaders └── Minimal startup assets Stream Later: ├── Site-specific data (on demand) ├── High-res textures (when needed) └── Optional features
  • Use Addressables for async loading
  • Show meaningful loading progress
  • Load assets near where they're needed
  • Unload when no longer needed

Shader Compilation Stalls

WebGL compiles shaders at runtime. First time a shader variant is used, there's a pause.

Mitigation Strategies

  • Shader Warmup: Force-use all shaders during loading screen
  • Reduce Variants: Minimize shader keywords/multi_compile
  • Prebuilt Shaders: Use ShaderVariantCollection
  • Simple Shaders: Complex shaders = longer compile time

Performance Best Practices

Rendering

  • Forward rendering: Safest for WebGL
  • Low shadow resolution: 512-1024 max
  • Minimal post-processing: Each effect has cost
  • Limit real-time lights: Use baked lighting where possible
  • MSAA 2x or FXAA: Not 4x/8x MSAA

Code Performance

  • Avoid GC allocations: More impactful than desktop
  • Pool objects: Instantiate/Destroy causes GC spikes
  • Incremental GC: Enable in Player Settings
  • Minimize Update() calls: Consolidate where possible

Physics

  • Simplify colliders: Use primitives (box, sphere) over mesh colliders
  • Reduce physics iterations: Lower solver iteration count
  • Disable unused features: Turn off continuous collision if not needed

Testing and Debugging

Development Workflow

  • Development Build: Enable for better debugging (larger size)
  • Browser DevTools: Use F12, Memory tab to track usage
  • Unity Profiler: Works with WebGL builds (connect via port)
  • Test on real browsers: Chrome, Firefox, Safari have different behaviors

Common Issues

Issue Cause Solution
Page goes blank/crashes Out of memory Reduce textures, unload assets
Long freeze on start Shader compilation Warmup shaders, reduce variants
Slow/stuttery GC spikes Object pooling, reduce allocations
Not loading CORS, server config Check server headers

Server Configuration

WebGL builds need proper server configuration:

# Required MIME types:
.wasm    →  application/wasm
.data    →  application/octet-stream
.br      →  Content-Encoding: br (Brotli)
.gz      →  Content-Encoding: gzip

# CORS headers (if hosting separately):
Access-Control-Allow-Origin: *

Key Takeaways

  • WebGL is single-threaded with ~2GB memory limit
  • Textures dominate memory — use compression, keep resolutions low
  • ETC2 is guaranteed in WebGL 2.0; use it as baseline
  • Build size affects load time — use compression, strip unused code
  • Addressables enable streaming content after initial load
  • Shader compilation causes stalls — warmup during loading
  • Use Forward rendering, low shadows, minimal post-processing
  • GC allocations are more painful — pool objects, avoid per-frame allocs
  • Configure server with correct MIME types and compression
