Lesson 17: Compute Shaders & WebGPU

The Browser GPU Revolution

For years, WebGL 2.0 held us back. We couldn't use Compute Shaders in the browser. Physics and heavy logic had to stay on the CPU (JavaScript/WASM), which is single-threaded and slow.

WebGPU changes everything. It brings modern GPU features—including Compute Shaders—to the web. VRify uses this to run massive simulations and animations directly on the GPU, bypassing the browser's main thread entirely.

What is a Compute Shader?

Standard shaders (Vertex/Fragment) are bound to the render pipeline. They draw triangles. Compute Shaders are "General Purpose GPU" (GPGPU) programs. They don't draw anything. They just crunch numbers.

Vertex/Fragment Shader

  • Input: Mesh vertices
  • Output: Pixels on screen
  • Structure: Fixed pipeline

Compute Shader

  • Input: Arbitrary data buffer
  • Output: Arbitrary data buffer
  • Structure: Free threads (Kernels)

The Animation Pipeline: Zero CPU Cost

Remember our 125,000 hex grid? Moving them with a Vertex Shader is fine for visual effects. But what if the hexes need to interact? What if they need to follow complex flocking rules or physics?

THE OLD WAY (CPU):
C# Loop (125k) -> Update Transform -> Upload to GPU -> Draw
Result: 5 FPS

THE COMPUTE WAY (GPU):
Compute Kernel (125k threads) -> Reads/Writes Buffer -> Vertex Shader Reads Buffer -> Draw
Result: 60+ FPS

The C# script does nothing but Dispatch() the compute shader. The data stays in VRAM.

Code: The Hexagon Wave Kernel

Here's a simplified Kernel that moves hexes based on a sine wave field.

// Hook up the buffer from C#
RWStructuredBuffer positions;
float time;

// Define thread count (e.g., 64 threads per group)
[numthreads(64, 1, 1)]
void CSMain (uint3 id : SV_DispatchThreadID)
{
    // 1. Get my current position
    float3 pos = positions[id.x];
    
    // 2. Calculate new height
    float height = sin(pos.x * 0.1 + time) * sin(pos.z * 0.1 + time);
    pos.y = height * 5.0;
    
    // 3. Write back to buffer
    positions[id.x] = pos;
}

In C#, you just dispatch it:

computeShader.SetBuffer(kernel, "positions", positionBuffer);
computeShader.SetFloat("time", Time.time);
// Dispatch 125,000 threads (divided by 64 threads per group)
computeShader.Dispatch(kernel, Mathf.CeilToInt(125000 / 64.0f), 1, 1);

You asked: "How do I click them?" or "How do I do Voronoi connections?"

Since we have no Colliders, we use the Compute Shader for Parallel Search.

Raycasting via Compute

To find which hex is under the mouse:

  1. Send Mouse Ray (Origin, Direction) to Compute Shader.
  2. Each thread checks intersection with its own hex (math).
  3. If hit, write id.x to a "Results" buffer.
  4. CPU reads the single int from the Results buffer.

Voronoi / Proximity

To draw lines between nearby points:

Checking every point against every other point is O(N²). For 125k points, that's 15 billion checks. Too slow even for GPU.

Solution: Spatial Hashing.

  1. Divide the world into a grid of buckets.
  2. Compute Shader Pass 1: Each point writes its ID into its local bucket.
  3. Compute Shader Pass 2: Each point only checks users in its own (and neighbor) buckets.

WebGPU Specifics

When targeting WebGPU in Unity:

  • HLSL to WGSL: Unity cross-compiles your compute shaders code to WGSL (WebGPU Shading Language).
  • Async Readback: Reading data back to CPU is asynchronous. You cannot get the result in the same frame without stalling. You request it, and get a callback a frame later.
  • Browser Support: Currently requires Chrome/Edge updates or specific flags enabled by end users (though rolling out rapidly).

Key Takeaways

  1. WebGPU enables Compute Shaders in the browser, unlocking massive simulation potential.
  2. Compute Shaders are generic programs. Input Buffer -> Math -> Output Buffer.
  3. Zero-Copy Pipeline: Compute updates positions, Rendering reads positions. Data never touches the CPU.
  4. Parallel Interaction: Use Compute to check 125,000 raycasts instantly instead of using Physics.
  5. StructuredBuffer is the main data structure used to pass arrays between C# and Compute.
