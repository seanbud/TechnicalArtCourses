Lesson 18: Scriptable Render Features Deep Dive

Hacking the Pipeline

URP isn't a black box. The "Scriptable" part means you can inject your own C# logic into the rendering process. You do this using **Scriptable Render Features**.

This is how you build tech art effects like Selection Outlines, X-Ray vision, Custom Blits, or rendering objects that "ignore" post-processing.

The Anatomy

There are two main classes you need to write:

The Feature (Config)

ScriptableRendererFeature

  • Exists as an Asset in the Project.
  • Stores settings (colors, materials).
  • Create() builds the Pass.
  • AddRenderPasses() injects the Pass.

The Pass (Logic)

ScriptableRenderPass

  • Does the actual work.
  • Configure(): Sets up targets.
  • Execute(): Sends commands to GPU.

The Render Loop & Injection Points

You can't just "render whenever." You must choose an insertion point via RenderPassEvent.

  • BeforeRenderingOpaques: Draw things behind the world (skybox replacements).
  • AfterRenderingOpaques: Draw transparents or overlays that need depth.
  • AfterRenderingSkybox: Good for standard transparency.
  • BeforeRenderingPostProcessing: Draw things that *should* get bloomed/color graded.
  • AfterRenderingPostProcessing: Draw UI or overlays that should look "raw" (like gizmos).

Case Study: The "Selection Outline"

Let's build the VRify implementation of "Select the yellow cubes."

Step 1: The Render Object Feature

Unity comes with a built-in feature called Render Objects. It's powerful. You can:

  • Filter objects by **Layer** (e.g., "Selected").
  • Override their material (e.g., "OutlineShader").
  • Override depth test (e.g., "Always" to see through walls).

Step 2: Custom Implementation (CommandBuffer)

If the built-in one isn't enough, we write code. We use a **CommandBuffer** to list instructions.

public override void Execute(ScriptableRenderContext context, ref RenderingData renderingData)
{
    CommandBuffer cmd = CommandBufferPool.Get("MyOutlinePass");
    
    // 1. Set a temporary render texture
    cmd.GetTemporaryRT(tempTexture.id, descriptor, FilterMode.Bilinear);
    cmd.SetRenderTarget(tempTexture.Identifier());
    cmd.ClearRenderTarget(false, true, Color.clear);
    
    // 2. Draw only the "Selected" objects into this texture as solid color
    DrawingSettings drawSettings = CreateDrawingSettings(shaderTags, ref renderingData, sortingCriteria);
    context.DrawRenderers(renderingData.cullResults, ref drawSettings, ref filterSettings);
    
    // 3. Blit that texture back to screen with a blur/edge-detect shader
    cmd.Blit(tempTexture.Identifier(), source, outlineMaterial);
    
    context.ExecuteCommandBuffer(cmd);
    CommandBufferPool.Release(cmd);
}

Command Buffers: The GPU To-Do List

When you call function in Execute, you aren't drawing immediately. You are populating a list. cmd.DrawMesh, cmd.Blit, cmd.SetGlobalFloat.

Only when you call context.ExecuteCommandBuffer(cmd) does that list get flushed to the GPU.

Common Use Cases for You

  1. Desaturation Mask: Render a mask of the "Mining Site Boundary" into the Stencil buffer. Then run a fullscreen pass that desaturates everything *outside* that stencil value.
  2. X-Ray: Render critical pipes/machinery with Test Always depth so they show on top of terrain, but with a ghost shader.
  3. Custom Shadows: Render a top-down view of the terrain into a texture, then project that texture onto the ground as a "fake" shadow map for performance.

Key Takeaways

  1. Scriptable Render Features let you inject custom passes into the URP loop.
  2. Feature = Config (Asset). Pass = Logic (Code).
  3. CommandBuffer is how you queue commands for the GPU.
  4. RenderPassEvent determines *when* your pass runs (e.g., before/after PostProcessing).
  5. Use them for localized effects (outlines, x-ray) that global post-processing can't handle.
