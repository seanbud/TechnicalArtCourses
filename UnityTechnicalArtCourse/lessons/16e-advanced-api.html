Lesson 16e: Advanced API & Memory Mechanics

Going Deeper: The Metal

In Lesson 16, we learned why valid GameObjects fail at scale. Now we will learn how to speak directly to the GPU driver. This lesson covers the specific C# API calls, memory initialization sequence, and the exact buffer structures required to render millions of objects without melting your CPU.

The API Trinity

You have three primary tools in `UnityEngine.Graphics`. Knowing which one to use is 90% of the optimization battle.

1. Graphics.DrawMesh

The "One-Off"

  • Use for: Single special effects, gizmos.
  • Data: CPU sends 1 matrix per call.
  • Perf: Slowest. 1 Draw Call per object (unless dynamic batched).

2. DrawMeshInstanced

The "Batcher"

  • Use for: 1k - 50k objects.
  • Data: CPU sends array of 1023 matrices (`Matrix4x4[]`).
  • Perf: Fast, but CPU must loop through arrays every frame.

3. DrawMeshInstancedIndirect

The "Fire & Forget"

  • Use for: 50k - Millions.
  • Data: CPU sends **pointers** to GPU Buffers.
  • Perf: Zero CPU overhead per object. 1 Draw Call total.

Bonus: Graphics.DrawProcedural
Similar to Indirect, but doesn't require a Mesh asset. It generates geometry (like a full-screen quad or points) entirely in the Vertex Shader using `SV_VertexID`. Great for avoiding backface culling overhead if you are drawing points/billboards.

Memory Architecture: CPU vs. GPU

Understanding where your data lives is critical.

[ CPU RAM (System Memory) ]        [ PCIe BUS ]        [ GPU VRAM (Video Memory) ]
       Array       ---------- SLOW ---------->   ComputeBuffer
       Matrix4x4[]        ---------- SLOW ---------->   StructuredBuffer

Golden Rule: Crossing the PCIe Bus is the bottleneck. DrawMeshInstanced crosses the bus every frame (uploading arrays). Indirect crosses the bus only once (initialization), then the data stays on the GPU.

Implementation: DrawMeshInstancedIndirect

This is the VRify standard for block models. Here is the exact initialization order.

1. Define Your Data Layout

The C# struct and HLSL struct must match byte-perfectly.

// C#
struct BlockData {
    public Vector3 position; // 12 bytes
    public uint colorPacked; // 4 bytes 
    // Total Stride: 16 bytes
}

// HLSL
struct BlockData {
    float3 position;
    uint colorPacked;
};

2. Create the ComputeBuffer (Initialization)

Order matters. Do this in Start() or OnEnable(), never Update.

// Parameters: Count (125k) and Stride (16 bytes)
dataBuffer = new ComputeBuffer(125000, 16, ComputeBufferType.Structured);

// Fill with initial data (Crossing the Bus!)
dataBuffer.SetData(myCpuArray);

3. The "Args Buffer" (The Holiest of Buffers)

For Indirect rendering, the CPU doesn't know how many things to draw. It reads a special buffer containing 5 integers.

// 5 uints: IndexCount, InstanceCount, StartIndex, BaseVertex, StartInstance
uint[] args = new uint[5] { 0, 0, 0, 0, 0 };

// Get the index count from the mesh we want to draw
args[0] = (uint)mesh.GetIndexCount(0);
args[1] = (uint)instanceCount; // 125,000

argsBuffer = new ComputeBuffer(1, args.Length * sizeof(uint), ComputeBufferType.IndirectArguments);
argsBuffer.SetData(args);

4. Material Setup

You must bind the buffer to the material so the shader can find it.

// "blockDataBuffer" must match the name in your Shader
myMaterial.SetBuffer("blockDataBuffer", dataBuffer);

5. The Draw Loop (Update)

Now, inside Update(), we issue the command.

// Bounds are required for Frustum Culling. 
// If the camera doesn't see 'bounds', the GPU command is skipped.
Bounds bounds = new Bounds(Vector3.zero, new Vector3(1000, 1000, 1000));

Graphics.DrawMeshInstancedIndirect(
    mesh, 
    0, 
    myMaterial, 
    bounds, 
    argsBuffer
);

Note: We pass argsBuffer, not dataBuffer. The material already holds the reference to the data.

Optimization: The Sparse Grid Problem

Let's say you have a 3D grid, but 90% is empty air. Drawing 1 million invisible "Air" blocks is a waste of vertex processing.

Packing Data

If you have 3 block types (Dirt, Gold, Rock), do not send a float4 color (16 bytes). That's massive waste.

Instead, pack it into a single uint (4 bytes) or even compress it into the position struct if you use float4 positionAndType.

Optimization Example:

  • Bad: float4 color (16 bytes) * 1 million = 16 MB bandwidth.
  • Good: uint typeID (4 bytes). Construct the color in the Shader logic.

Avoiding Branch Divergence

In the shader, avoid this:

// BAD: Divergence
if (type == 0) color = red;
else if (type == 1) color = gold;
else color = grey;

Instead, use a Global Color Array or Texture Map.

// GOOD: Uniform lookup
color = _ColorPalette[type];

This ensures all threads run the exact same instruction, fetching from an array, rather than branching into different code paths.

Key Implementation Details

  1. Stride Calculation: Must match exactly. Float=4, Vector3=12, Matrix=64 bytes. Mismatch = crash or glitches.
  2. Initialization Order: Create Buffers -> Set Data -> Bind to Material. Do not re-create buffers in Update.
  3. Release Memory: ComputeBuffer is unmanaged C++ memory. You MUST pair every new with a Release() in OnDisable() or your GPU VRAM will leak.
  4. Args Buffer: 5 integers. The second one (Instance Count) is the most important one to update if your count changes dynamically.
