Lesson 04

Forward vs Deferred & Forward+

Understanding rendering paths and when to use each

The Core Problem: Lights Are Expensive

The fundamental difference between forward and deferred rendering is how they handle many lights. Understanding this will tell you when to use each approach.

🧠 The Light-Geometry Relationship

Every light that affects an object requires work. In a naive approach: Cost = Objects × Lights. With 100 objects and 50 lights, that's 5,000 lighting calculations. Rendering approaches differ in how they tackle this multiplication.

Forward Rendering

The traditional approach used by URP's default mode. Objects are rendered one by one, calculating lighting during the fragment shader.

FORWARD RENDERING: For each object: For each light affecting this object: └── Calculate lighting in fragment shader └── Output color to framebuffer Object 1 → [Vertex] → [Fragment + All Lights] → Framebuffer Object 2 → [Vertex] → [Fragment + All Lights] → Framebuffer Object 3 → [Vertex] → [Fragment + All Lights] → Framebuffer

Pros

  • Transparency works naturally — alpha blending is straightforward
  • MSAA support — hardware anti-aliasing works efficiently
  • Lower memory usage — no GBuffer required
  • Simple shader workflow — everything happens in one pass
  • Better for mobile/browser — fewer GPU features required

Cons

  • Light count limited — URP limits per-object lights (typically 4-8 additional)
  • Overdraw re-calculates lighting — if pixels are overwritten, lighting work is wasted
  • Per-object light loops — more lights = more passes or shader complexity
💚 VRify Relevance

For browser targets, Forward rendering is typically your best choice. WebGL has memory constraints, MSAA support varies, and deferred's GBuffer can be problematic. Forward is simpler and more compatible.

Deferred Rendering

Available in HDRP and as a URP option. Separates geometry rendering from lighting by using a GBuffer (Geometry Buffer).

DEFERRED RENDERING: PASS 1: Geometry Pass (write to GBuffer) ┌─────────────────────────────────────────────────────────────────┐ │ For each object: │ │ └── Render geometry data into multiple textures: │ │ ├── Albedo (base color) │ │ ├── Normal │ │ ├── Depth │ │ ├── Metallic/Smoothness │ │ └── Emission │ └─────────────────────────────────────────────────────────────────┘ ↓ PASS 2: Lighting Pass (read from GBuffer) ┌─────────────────────────────────────────────────────────────────┐ │ For each light: │ │ └── Sample GBuffer, calculate lighting for affected pixels │ │ └── Accumulate into final image │ └─────────────────────────────────────────────────────────────────┘

The GBuffer

The GBuffer is a set of render textures that store surface properties for every visible pixel. Typical contents:

Texture Contents Why Stored
Albedo RGB color + Alpha Base material color for lighting
Normals World-space normal Light direction calculations
Depth Distance from camera Position reconstruction
Specular/Roughness Material properties Reflection calculations
🧠 Why Deferred Is Efficient with Many Lights

In deferred, geometry is rendered once regardless of light count. Each light then only processes the pixels it affects (using its volume/range). Cost becomes Objects + (Lights × affected pixels) instead of Objects × Lights.

Pros

  • Many lights efficiently — hundreds of lights with minimal cost increase
  • Pixel-perfect lighting — no per-object light limits
  • Screen-space effects are easy — all data is in textures (SSAO, SSR)

Cons

  • High memory usage — GBuffer can be 160+ bytes per pixel at 4K
  • Transparency is hard — GBuffer only stores one surface per pixel; transparent objects need forward pass
  • No MSAA — must use post-process AA (FXAA, TAA, SMAA)
  • Limited material variety — all materials must fit in GBuffer format

Forward+ (Clustered Forward / Tiled Forward)

The best of both worlds. This is what URP calls "Forward+" when you enable it in renderer settings.

FORWARD+ RENDERING: PREPASS: Build Light Clusters ┌─────────────────────────────────────────────────────────────────┐ │ Divide screen into tiles (e.g., 16×16 pixel tiles) │ │ Divide depth range into slices (creating 3D clusters) │ │ For each light: determine which clusters it overlaps │ │ Create lookup table: cluster → list of affecting lights │ └─────────────────────────────────────────────────────────────────┘ ↓ RENDER: Forward with Efficient Light Access ┌─────────────────────────────────────────────────────────────────┐ │ For each object: │ │ In fragment shader: │ │ └── Look up which cluster this pixel is in │ │ └── Only iterate lights in that cluster (not ALL lights) │ │ └── Calculate lighting, output color │ └─────────────────────────────────────────────────────────────────┘

Why Forward+ Works

The key insight is lights have limited range. A point light in the corner of your scene doesn't affect pixels on the opposite side. By pre-computing which lights affect which screen regions, each pixel only loops through relevant lights.

🧠 Clusters vs Tiles

Tiled Forward: 2D tiles based on screen position only. Works but doesn't handle depth well.

Clustered Forward (Forward+): 3D clusters that also consider depth. Much better at culling lights that are near but behind geometry.

Pros

  • Many lights like deferred — hundreds of local lights efficiently
  • MSAA support — it's still forward rendering
  • Transparency works — same as regular forward
  • Less memory than deferred — no full GBuffer
  • Material flexibility — not limited by GBuffer format

Cons

  • Depth prepass overhead — needs to know depth before lighting
  • Cluster computation — some CPU/GPU work to build light lists
  • Not all platforms — needs compute shader support

Comparison Summary

Feature Forward Deferred Forward+
Many Lights ❌ Limited ✅ Excellent ✅ Excellent
MSAA ✅ Yes ❌ No ✅ Yes
Transparency ✅ Easy ⚠️ Hard ✅ Easy
Memory ✅ Low ❌ High ⚠️ Medium
WebGL ✅ Best choice ❌ Not recommended ⚠️ Limited support

What to Use at VRify

Given browser targets and geospatial visualization:

  • Primary: Forward rendering — most compatible with WebGL, sufficient for visualization scenarios
  • Consider Forward+ if: You have many local lights and can target WebGL 2.0 with compute support (check browser compatibility)
  • Avoid Deferred for browser: Memory constraints and transparency complexity make it problematic

Key Takeaways

  • Forward: Simple, MSAA-friendly, limited lights — best for mobile/browser
  • Deferred: Many lights via GBuffer, high memory, no MSAA — best for AAA desktop
  • Forward+: Hybrid approach with clustered light culling — best of both worlds with compute support
  • GBuffer: Stores surface properties per-pixel (albedo, normals, depth, roughness)
  • For VRify: Forward rendering is your safest bet for browser compatibility
