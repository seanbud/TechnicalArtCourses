Lesson 16: Procedural Drawing & GPU Instancing

The "Phantom" Geometry

At VRify, you might be asked to render a dataset with 125,000 distinct hexagon tiles—a 50x50x50 grid. If you try to spawn 125,000 GameObjects with MeshRenderers, Unity will likely crawl to 2 FPS or crash entirely.

GameObjects are heavy. They have Transforms, Colliders, Editor overhead, and serialization costs. For massive scale, we bypass GameObjects entirely and talk directly to the GPU using the Graphics API.

We call this "Phantom Geometry" because these objects don't exist in the Scene Hierarchy. You can't click them in the Scene view. They exist only as mathematical commands sent to the render pipeline.

DrawMesh vs. GameObjects

❌ The GameObject Way

Instantiate(prefab) 125,000 times.

  • Memory: Massive overhead per object
  • CPU: Updating 125k Transforms is slow
  • Hierarchy: Editor becomes unusable
  • Garbage Collection: High pressure

✅ The Graphics API Way

Graphics.DrawMeshInstanced(...)

  • Memory: Just arrays of matrices (floats)
  • CPU: Minimal overhead (batching)
  • Hierarchy: Empty (clean)
  • Garbage Collection: Zero (if using buffers)

Technique 1: Graphics.DrawMeshInstanced

This is the standard way to draw up to 1023 instances per draw call. Unity automatically batches these calls for you.

The limitations

  • Max 1023 instances per batch (you must split your list)
  • Requires MaterialPropertyBlock for per-instance data
  • Shader must support instancing (#pragma multi_compile_instancing)

Code Example: The 1023 Batcher

void Update() {
    // We have 125,000 matrices to draw
    // We must submit them in batches of 1023
    
    RenderParams rp = new RenderParams(myMaterial);
    
    for (int i = 0; i < allMatrices.Count; i += 1023) {
        int count = Mathf.Min(1023, allMatrices.Count - i);
        // Create a slice or pass the sub-list (simplified here)
        var batch = allMatrices.GetRange(i, count); 
        
        Graphics.DrawMeshInstanced(
            mesh, 
            0, 
            myMaterial, 
            batch, 
            null, // MaterialPropertyBlock
            ShadowCastingMode.On
        );
    }
}

Technique 2: DrawMeshInstancedIndirect (The Holy Grail)

What if 1023 isn't enough? DrawMeshInstancedIndirect (or DrawProcedural) moves the data management effectively entirely to the GPU.

Instead of passing C# arrays, you pass a ComputeBuffer containing all your object data. The limits become "how much VRAM do you have?"

How it works

CPU SIDE:
1. Create ComputeBuffer with 125,000 transform matrices
2. Create "Args Buffer" (Index count, Instance count, Start Index...)
3. Send Graphics.DrawMeshInstancedIndirect(mesh, submesh, material, bounds, argsBuffer)

GPU SIDE (Shader):
1. Use SV_InstanceID to fetch matrix from the StructuredBuffer
2. Apply transform
3. Draw

This is how you render millions of grass blades or particles. The CPU issues one single command.

Per-Instance Properties (Color, ID, etc.)

You can't instantiate a new Material for every hex (that breaks batching). Instead, we use MaterialPropertyBlocks (MPB).

// C# Side
MaterialPropertyBlock props = new MaterialPropertyBlock();
List colors = new List(); // 125k colors

// Fill colors...

// Pass to DrawMeshInstanced
Graphics.DrawMeshInstanced(mesh, 0, mat, matrices, props);

// Shader Side
UNITY_INSTANCING_BUFFER_START(Props)
    UNITY_DEFINE_INSTANCED_PROP(float4, _Color)
UNITY_INSTANCING_BUFFER_END(Props)

// Vertex Shader
float4 color = UNITY_ACCESS_INSTANCED_PROP(Props, _Color);

For Indirect drawing, you don't use MPB. You use another StructuredBuffer in the shader containing your colors, indexed by Instance ID.

The "Phantom" Problem: Interaction & Physics

Since these aren't GameObjects, Collider components don't work. You can't just add a `BoxCollider`. You can't use `OnMouseDown`. So how do you click a hex?

Solution A: Math Raycasting (Fastest)

If your grid is regular (like a hex grid), you don't need physics. You can calculate mathematically which cell the mouse ray hits.

  • Raycast against the ground plane (y=0)
  • Convert world hit point -> Grid Coordinate (q, r)
  • Look up data in your 2D/3D array

Solution B: Proxy Colliders (Hackier)

If you only need to click one thing at a time, spawn a single invisible Collider and move it to track the mouse. Or, if you have few objects (1000s), you can spawn "Entity" generic gameobjects with just colliders but no renderers.

Solution C: GPU Raycasting (Advanced)

Send the mouse ray to a Compute Shader. Have each of the 125,000 threads check "did the ray hit me?" Write the ID of the hit object to a buffer. Read it back on CPU.

Animation Strategy

⚠️ BAD: Calculating 125,000 generic Mathf.Sin() animations on CPU and updating 125,000 matrices in the list every frame.

✅ GOOD: Pass Time.time to the shader. Use vertex displacement in the shader to animate position/scale/rotation based on Instance ID.

Example: To make hexes "cascade" in:

// Vertex Shader
float id = UNITY_GET_INSTANCE_ID(input);
float noise = snoise(worldPos * 0.1);
float triggerTime = _StartTime + noise * 2.0;
float scale = smoothstep(triggerTime, triggerTime + 0.5, _Time.y);

// Scale the vertex position (effectively scaling the object)
v.vertex.xyz *= scale;

Zero CPU cost. 125,000 animating objects at 60 FPS.

Key Takeaways

  1. Scale kills GameObjects. Use Graphics.DrawMeshInstanced for >5k similar objects.
  2. DrawMeshInstanced uses batches of 1023 matrices.
  3. DrawMeshInstancedIndirect uses GPU buffers (`ComputeBuffer`) for effectively infinite scale.
  4. Interaction requires math-based raycasting (Grid coords) or proxy colliders, since there are no GameObjects.
  5. Animation should happen in the **Vertex Shader** using Instance ID/Position, not by updating matrices on the CPU.
  6. MaterialPropertyBlock breaks instances into batches if you aren't careful, but allows per-instance colors for standard instancing.
