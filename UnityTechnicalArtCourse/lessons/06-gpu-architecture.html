GPU Architecture

Parallel processing, branching costs, and Texture2D arrays

Why GPUs Are Different

CPUs are designed for complex, sequential tasks. GPUs are designed for simple, massively parallel tasks. Understanding this difference explains many graphics optimization principles.

CPU vs GPU ARCHITECTURE: CPU (few powerful cores): ┌─────────────────┐ │ Core 0 │ Each core: complex, independent │ (very smart) │ Good at: branching, varied workloads ├─────────────────┤ │ Core 1 │ │ (very smart) │ ├─────────────────┤ │ ... (4-16) │ └─────────────────┘ GPU (many simple cores): ┌──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──┐ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ Each core: simple ├──┼──┼──┼──┼──┼──┼──┼──┼──┼──┼──┼──┼──┼──┼──┼──┤ Thousands of them │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ All do SAME instruction ├──┼──┼──┼──┼──┼──┼──┼──┼──┼──┼──┼──┼──┼──┼──┼──┤ on different data │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ └──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──┘ (thousands of simple cores)

SIMD: Same Instruction, Multiple Data

GPUs use SIMD (Single Instruction, Multiple Data) execution. Groups of threads execute the same instruction simultaneously on different data.

🧠 The Key Insight

A GPU "core" runs the same shader instruction across 32-64 pixels at once (called a warp on NVIDIA, wavefront on AMD). Every pixel in the warp executes the same line of shader code at the same time, just with different input values.

SIMD EXECUTION (simplified warp of 8 pixels): Shader line: color = texture.Sample(uv); Pixel 0: uv=(0.0, 0.0) ─┐ Pixel 1: uv=(0.1, 0.0) │ Pixel 2: uv=(0.2, 0.0) │ All 8 execute the Pixel 3: uv=(0.3, 0.0) ├─ SAME instruction Pixel 4: uv=(0.0, 0.1) │ at the SAME time Pixel 5: uv=(0.1, 0.1) │ with DIFFERENT data Pixel 6: uv=(0.2, 0.1) │ Pixel 7: uv=(0.3, 0.1) ─┘

Why Branching Hurts: Warp Divergence

This is critical to understand for interview questions. What happens when you write if/else in a shader?

BRANCHING PROBLEM: Shader code: if (someCondition) { doExpensiveA(); // 10 instructions } else { doExpensiveB(); // 10 instructions } Warp execution when HALF take each branch: Pixel 0-3: condition = true Pixel 4-7: condition = false Step 1: Pixels 0-3 run doExpensiveA() (10 instructions) Pixels 4-7 sit IDLE and wait Step 2: Pixels 4-7 run doExpensiveB() (10 instructions) Pixels 0-3 sit IDLE and wait Total time: 20 instructions (worst case: BOTH branches run!)
⚠️ Warp Divergence

When threads in a warp take different branches, the warp must execute both branches and mask out threads that shouldn't run each branch. This can effectively halve your shader performance or worse.

When Branching Is Okay

  • Uniform branches: If all pixels in a warp take the same branch, no divergence occurs
  • Early-out branches: Skipping work entirely can still be a win if most pixels bail early
  • Simple branches: Very short branches may be worth it anyway
// OKAY: Same value for entire warp (uniform)
if (_GlobalFeatureEnabled) { ... }

// PROBLEMATIC: Different per-pixel
if (worldPosition.y > 5.0) { ... }

// SOMETIMES OKAY: Early-out
if (alpha < 0.001) discard;  // Skips remaining work

GPU Memory Hierarchy

Memory access patterns matter a lot for GPU performance:

GPU MEMORY (fastest to slowest): ┌─────────────────────────────────────────┐ │ Registers (per-thread) │ ← Fastest, very limited │ ~256 registers per thread │ ├─────────────────────────────────────────┤ │ Shared Memory (per block/warp) │ ← Fast, limited (~32-48KB) │ Shared between threads in a group │ ├─────────────────────────────────────────┤ │ L1/L2 Cache │ ← Fast, automatic ├─────────────────────────────────────────┤ │ Texture Memory (cached differently) │ ← Optimized for 2D access ├─────────────────────────────────────────┤ │ Global/VRAM (main GPU memory) │ ← Slowest, abundant │ This is your textures, meshes, buffers │ └─────────────────────────────────────────┘
🧠 Texture Cache Optimization

GPUs have specialized texture caches that exploit spatial locality — neighboring pixels often sample nearby UV coordinates. This is why texture filtering is fast and why texture atlases work well.

Texture2D Arrays: Why They Exist

You mentioned wanting to understand Texture2DArray. Here's the problem they solve:

PROBLEM: Material Variety Scenario: 100 rocks with 10 different textures Traditional approach: ├── 10 different materials (one per texture) ├── 10 different draw calls (can't batch) └── Each rock uses its assigned material With Texture2D Array: ├── 1 Texture2DArray containing all 10 textures as layers ├── 1 material that samples from array using an index ├── Rocks pass texture index as vertex attribute └── All rocks can be batched together!

How Texture2DArray Works

A Texture2DArray is a stack of 2D textures with identical dimensions and format. You sample using (u, v, layer_index).

// Conceptually in shader:
TEXTURE2D_ARRAY(_TextureArray);

// Sampling:
float4 color = SAMPLE_TEXTURE2D_ARRAY(
    _TextureArray, sampler,
    uv,           // u, v coordinates
    textureIndex  // which layer (0, 1, 2, ...)
);

Use Cases

  • Terrain textures: Blend between grass/dirt/rock stored in array
  • Character variations: Same model, different skins
  • Decal systems: Many decal textures in one array
  • Batching enabler: Reduce material count to reduce draw calls
💚 VRify Relevance

For geospatial visualization, texture arrays could unify materials for similar features (mine layers, elevation bands, different scan types) without breaking batching. Great for point cloud colorization or terrain blending.

Compute Shaders Revisited

Since you mentioned wanting to understand compute shaders:

Compute shaders run outside the graphics pipeline. They're general-purpose GPU programs that read/write to buffers and textures.

COMPUTE SHADER EXECUTION: Input: Compute Shader: Output: ───────────────── ───────────────────── ───────────────── Buffers Dispatch(X, Y, Z) Modified buffers Textures └── Thread groups New textures └── Threads (64-256 per group) Example: 1024 particles Dispatch(16, 1, 1) with 64 threads/group = 16 groups × 64 threads = 1024 parallel threads = Each thread updates one particle

Thread Groups

Compute work is dispatched in thread groups. Each group has a fixed number of threads (commonly 64-256). You decide how many groups to launch.

  • [numthreads(64, 1, 1)] — defines group size (64 threads)
  • Dispatch(16, 1, 1) — launches 16 groups
  • Total threads: 64 × 16 = 1024

Compute vs Graphics

Graphics Shaders Compute Shaders
Fixed input (vertices/pixels) Arbitrary input (any buffer)
Fixed output (framebuffer) Arbitrary output (any buffer)
Part of render pipeline Separate from rendering
Great for drawing Great for simulation, processing
⚠️ WebGL Compute Support

WebGL 2.0 does NOT support compute shaders. WebGPU will, but it's not yet widely deployed. For browser targets at VRify, compute-heavy workflows may need to run on the server or fall back to CPU.

Key Takeaways

  • GPUs use SIMD — thousands of threads execute the same instruction on different data
  • Warp divergence from branching forces both branches to run, devastating performance
  • Uniform branches (same value for all pixels) don't cause divergence
  • GPU memory has hierarchy: registers → shared → cache → global VRAM
  • Texture2DArray bundles multiple textures, enabling batching with variety
  • Compute shaders are general-purpose GPU programs outside the graphics pipeline
  • WebGL 2.0 lacks compute support — consider this for VRify browser targets
