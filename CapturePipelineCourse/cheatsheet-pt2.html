The Three Architecture Layers

Config Registry

Scans /config/clients/*.json at startup. Each profile specifies skeleton, naming pattern, export format, and delivery destination. New client = new JSON file, zero code changes.

Export Adapters (Factory)

FBX, glTF, USD — each has its own adapter class. A factory picks the right one from the client config. Adding a format = one new class.

Delivery Adapters (Factory)

Perforce, NAS/SMB, S3, SFTP — same factory pattern. The pipeline calls deliver() and doesn't know where the file goes.

Plugin Architecture

When a client needs genuinely unique behavior (not configurable), isolate it in a plugin with defined hook points: pre_export, post_export, custom_validate.

5 Design Pattern Quick-Reference

Patterns The four patterns that answer 80% of interview architecture questions.

🔌 Adapter Pattern

Problem: Vendor lock-in
Solution: Wrap each vendor's SDK behind a common interface
Result: Swap vendors without changing downstream code
Used in: Ingest (Vicon vs OptiTrack), Export (FBX vs glTF), Delivery (P4 vs NAS vs S3)

🏭 Factory Pattern

Problem: Choosing the right adapter at runtime
Solution: A factory reads config and returns the correct adapter
Result: get_export_adapter(profile) — one line, any format
Used in: Export factory, Delivery factory, Ingest factory

♟️ Strategy Pattern

Problem: Marker vs markerless need different processing
Solution: Abstract pipeline stages with swappable implementations
Result: Technology choice is config, not code branches
Used in: Universal pipeline (ingest + cleanup strategies)

🔒 Circuit Breaker

Problem: Hammering a dead service backs up the entire queue
Solution: After N failures, fail ALL requests instantly. Auto-retry after cooldown
Result: Cascading failures become instant failure + alert
Used in: NAS writes, P4 submits, API calls
