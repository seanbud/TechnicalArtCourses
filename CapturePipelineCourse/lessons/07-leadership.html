Lesson 07

System Design & Leadership

Thinking like an Architect: Decoupling, Failure Modes, and Cultural Engineering.

The Whiteboard Interview

In a Senior TA interview, they will hand you a marker and say: "Design a pipeline to ingest 1TB of photos per day." Do not write code. Draw boxes. They are testing if you understand Latency vs Throughput.

Shoot ruined. │ │ *Issue:* Tightly Coupled. │ ├─────────────────────────────────────────────────────────────────┤ │ ROBUST DESIGN (The Architect) │ │ [Camera] ──▶ [Local SSD Buffer] ──▶ [Uploader Service] │ │ │ │ │ │ ▼ ▼ │ │ Safe even if Net Retries until Success │ │ is down. 1. The Principle of Decoupling (Queues)

Never connect a critical producer (Camera) directly to a flaky consumer (Network). Always use a Message Queue.

  • Producer: Writes to Queue (Fast).
  • Consumer: Reads from Queue (Slow).
  • Benefit: The Producer can work even if the Consumer is offline.

2. Technical Drill: Design Prompts

Practice these 3 scenarios. Draw them out.

Scenario A: The Multi-Site Sync

Problem: "We have a studio in Vancouver and one in Guildford (UK). How do we keep 50TB of data in sync so both teams can work?"

Solution: The Proxy Pattern.

  • Central Source: Perforce Server in Vancouver (Master).
  • Local Caching: Perforce Proxy server in Guildford.
  • Pre-Fetch: A script runs at 3AM to force-sync the Proxy with yesterday's dailies, so when Guildford artists arrive at 9AM, the data is local (Fast).
Scenario B: The Global Asset ID

Problem: "We have assets in Perforce, Shotgun, and Jira. How do we link them?"

Solution: The UUID/manifest.

  • The Key: Use a GUID (Globally Unique ID) generated at asset creation.
  • The Sidecar: Every asset gets a `.json` sidecar file containing this ID.
  • The Index: A central DB (Lesson 06) maps ID -> P4 Path -> Shotgun ID.
Scenario C: The "Crash Reporter"

Problem: "Maya is crashing randomly. We don't know why."

Solution: Telemetry.

  • Hook: Write a `userSetup.py` script that hooks into `maya.cmds.scriptJob(event="quitApplication")`.
  • Payload: Capture RAM usage, Loaded Plugins, VRAM, and Last Command.
  • Transport: Send JSON payload to an ELK Stack (Elasticsearch, Logstash, Kibana) for visualization. "Oh look, everyone crashing has the 'BetaTools' plugin loaded."

3. Cultural Engineering (Code Reviews)

You are the gatekeeper. Your job is to improve the team, not just the code.

🧠 Concept: The "Why" Rule

Never just say "Fix this." Explain The Constraint.
Bad: "Don't use `Z:\` path." (Author feels stupid).
Good: "Linux render nodes don't have drive letters, so `Z:\` will cause a farm failure. Please use `os.getenv('PROJECT_ROOT')` instead." (Author learns system architecture).

4. Behavioral Prep (The Bar Raiser)

Finally, they will ask: "Tell me about a time you failed." Use the STAR Method.

Example Answer: The Broken Build

  • Situation: "We were shipping the E3 demo. The farm was crashing on every 10th frame."
  • Task: "I needed to fix the stability without stopping the artists."
  • Action: "I identified a memory leak in the shader compiler. I wrote a 'Chunking' script (Lesson 05) to restart Maya every 5 frames, bypassing the leak."
  • Result: "The farm stability went from 50% to 100%. We shipped the demo on time."

Key Takeaways

  • Decoupling: Use queues to separate Producers (Camera) from Consumers (Network). Never couple tightly.
  • Design for Failure: Assume the network will fail, the disk will fill up, and the renderer will crash. Plan for the "Sad Path".
  • Latency vs Throughput: Understand the difference. Optimization requires knowing which one is the bottleneck.
  • Code Review is Culture: Use reviews to teach architecture, not just to fix bugs. Explain the "Why".
  • Telemetry: You can't fix what you can't measure. Log everything to a central server (ELK Stack).
🎬 EA Studio Relevance: Need for Speed

In Need for Speed, the "car usage" telemetry tells designers which cars are popular. In Tech Art, "tool usage" telemetry tells us which tools are broken. If we see the "Import Car" tool throwing errors for 50% of users, we prioritize fixing it immediately. We don't wait for artists to report bugs—the system reports them for us.
