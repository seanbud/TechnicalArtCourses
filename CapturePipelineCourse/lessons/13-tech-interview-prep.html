2. Question Bank: Cross-Departmental Data Flow

Q: "How would you design a system that gets mocap data from the capture stage to the game team in under 30 minutes?"

Your Answer:

Context: In a marker-based capture pipeline, there are 6 stages between the raw capture and the game team receiving an FBX: ingest, solve, retarget, validate, export, and deliver. Each stage currently has manual handoffs with waiting time.

Architecture: I'd build an event-driven pipeline using a file-system watcher (Python's watchdog) that monitors the stage output directory. Each stage is a discrete step that triggers the next automatically. No human hands-off required between stages. This is the "sweeper" pattern — a daemon that sweeps new files and pushes them through the pipeline.

Implementation: The watcher detects a new .c3d file, validates it, and submits a Deadline farm job for the solve+retarget. When the farm job completes, a callback triggers the export adapter which writes FBX in the client's format. The delivery adapter (Perforce or NAS copy) then publishes the file. A Slack notification pings the game team's channel with the asset link.

Tradeoff: Full automation vs. human QA. For hero performances, I'd add a "hold for review" gate where an animator approves the solve before delivery. For background animations, fully automated.

Metric: This reduces the stage-to-artist time from "whenever someone remembers to process it" — often hours or next-day — to a consistent 15-25 minutes.

Q: "How do you handle versioning for mocap data?"

Your Answer:

Context: Mocap data goes through multiple iterations — raw capture, first solve, manual cleanup, re-solve, final delivery. Each version needs to be tracked, and artists need to be able to access any previous version.

Architecture: I'd use a three-tier versioning approach. Perforce for the source-of-truth version control. A semantic version suffix on every file (_v001, _v002) so the naming itself communicates the version. And a metadata JSON sidecar file per take that records the full history.

Implementation:

Python: Version-Tracked Delivery

"""
Version-tracked delivery: every file gets a version, every version
gets a sidecar with metadata history.
"""
import json
import os
import hashlib
from datetime import datetime

def get_next_version(output_dir, take_name, extension="fbx"):
    """Find the next available version number for a take."""
    version = 1
    while True:
        filename = f"{take_name}_v{version:03d}.{extension}"
        if not os.path.exists(os.path.join(output_dir, filename)):
            return version, filename
        version += 1

def create_sidecar(output_path, metadata):
    """Create a JSON sidecar file with version metadata."""
    sidecar_path = output_path.replace('.fbx', '_meta.json')
    
    # Load existing sidecar if it exists
    history = []
    if os.path.exists(sidecar_path):
        with open(sidecar_path) as f:
            existing = json.load(f)
            history = existing.get("version_history", [])
    
    # Add this version's entry
    with open(output_path, 'rb') as f:
        md5 = hashlib.md5(f.read()).hexdigest()
    
    history.append({
        "version": metadata["version"],
        "timestamp": datetime.now().isoformat(),
        "source_file": metadata["source_file"],
        "technology": metadata.get("technology", "marker"),
        "operator": metadata.get("operator", "automated"),
        "md5": md5,
        "notes": metadata.get("notes", ""),
    })
    
    sidecar = {
        "take_name": metadata["take_name"],
        "latest_version": metadata["version"],
        "version_history": history,
    }
    
    with open(sidecar_path, 'w') as f:
        json.dump(sidecar, f, indent=2)
    
    return sidecar_path


# Usage:
version_num, filename = get_next_version("/output/fc/", "FC_20260228_sc01_sh05")
# → (3, "FC_20260228_sc01_sh05_v003.fbx")

create_sidecar(output_path, {
    "take_name": "FC_20260228_sc01_sh05",
    "version": version_num,
    "source_file": "/captures/raw/sc01_sh05.c3d",
    "technology": "marker",
    "notes": "Re-solved after marker swap fix on frame 1205",
})

3. Question Bank: Automation & Reduced Processing Time

Q: "Give me an example of tooling you'd build to reduce processing time."

Your Answer:

Context: The biggest time sink in mocap post-production is manual bone naming fixes. After Vicon's solver outputs joints, they use Vicon's naming convention. The game team uses a different naming convention. Someone manually renames every joint — 50+ joints per take, thousands of takes per year.

Architecture: A config-driven batch renaming tool. A JSON file maps Vicon joint names to game skeleton names. A Python script reads the FBX, renames all joints in a single pass, validates the result, and writes the output. CLI-first for batch processing, with a Maya/MotionBuilder UI wrapper for one-off fixes.

Implementation: Python with the FBX SDK for direct FBX manipulation, or Maya's cmds.rename() for in-scene renaming. The JSON config makes it client-portable — different game teams have different skeleton templates, so you just swap the config file.

Metric: Manual renaming takes 10-15 minutes per take. With the tool, it takes 2 seconds. Across 5,000 takes per year, that's ~1,000 hours of artist time recovered.

Q: "How would you validate that a processed mocap file is correct before delivery?"

Your Answer:

Architecture: A validation pipeline with pluggable checkers. Each checker tests one thing and returns pass/fail with a reason. The validation runner executes all checkers and generates a report.

Python: Pluggable Validation Pipeline

"""
Validation Pipeline — Pluggable checkers that verify mocap data quality.
"""
import re
import os

class ValidationResult:
    def __init__(self, checker_name, passed, message=""):
        self.checker_name = checker_name
        self.passed = passed
        self.message = message
    
    def __repr__(self):
        status = "✅" if self.passed else "❌"
        return f"{status} [{self.checker_name}] {self.message}"


# === Individual Checkers ===

def check_naming(filepath, profile):
    """Verify filename matches client naming convention."""
    basename = os.path.basename(filepath)
    pattern = profile.naming_pattern
    
    if re.match(pattern, os.path.splitext(basename)[0]):
        return ValidationResult("naming", True, f"Matches pattern: {pattern}")
    return ValidationResult("naming", False, 
        f"'{basename}' does not match pattern: {pattern}")

def check_skeleton_completeness(scene_joints, profile):
    """Verify all required joints exist."""
    required = set(profile.required_joints)
    present = set(scene_joints)
    missing = required - present
    
    if not missing:
        return ValidationResult("skeleton", True, 
            f"All {len(required)} required joints present")
    return ValidationResult("skeleton", False, 
        f"Missing joints: {missing}")

def check_frame_range(frame_count, profile):
    """Verify frame count is within acceptable range."""
    min_f = profile.get("validation", "min_frame_count", default=10)
    max_f = profile.get("validation", "max_frame_count", default=50000)
    
    if min_f <= frame_count <= max_f:
        return ValidationResult("frame_range", True, 
            f"Frame count {frame_count} is within [{min_f}, {max_f}]")
    return ValidationResult("frame_range", False, 
        f"Frame count {frame_count} outside range [{min_f}, {max_f}]")

def check_root_at_origin(root_position, tolerance=5.0):
    """Verify root joint is near world origin at frame 0."""
    import math
    dist = math.sqrt(sum(c**2 for c in root_position))
    
    if dist < tolerance:
        return ValidationResult("root_origin", True, 
            f"Root is {dist:.2f}cm from origin (tolerance: {tolerance}cm)")
    return ValidationResult("root_origin", False, 
        f"Root is {dist:.2f}cm from origin (tolerance: {tolerance}cm)")

def check_file_integrity(filepath):
    """Verify the file is a valid FBX (not truncated/corrupt)."""
    # Quick check: FBX files start with "Kaydara FBX Binary" or
    # ";" (ASCII FBX comment) in the first bytes
    with open(filepath, 'rb') as f:
        header = f.read(20)
    
    if b"Kaydara" in header or header[:1] == b";":
        return ValidationResult("integrity", True, "Valid FBX header")
    return ValidationResult("integrity", False, "Invalid FBX header — file may be corrupt")


# === Runner ===
def validate_all(filepath, scene_joints, frame_count, root_position, profile):
    """Run all validators and return results."""
    results = [
        check_naming(filepath, profile),
        check_skeleton_completeness(scene_joints, profile),
        check_frame_range(frame_count, profile),
        check_root_at_origin(root_position),
        check_file_integrity(filepath),
    ]
    
    for r in results:
        print(r)
    
    passed = all(r.passed for r in results)
    print(f"\n{'✅ ALL PASSED' if passed else '❌ VALIDATION FAILED'}")
    return results

4. Question Bank: System Design & Architecture

Q: "How would you design a pipeline that works for 15 different clients without maintaining 15 codebases?"

Your Answer: (This is Lesson 10, condensed into an interview answer.)

Architecture: One pipeline engine, many configuration profiles. Each client gets a JSON config file specifying their skeleton template, naming pattern, export format, and delivery destination. The core code reads the config and acts generically. The pipeline uses three patterns: Config Registry for client profiles, Adapter + Factory for format-specific export and delivery, and a Plugin Architecture for genuinely unique client requirements that can't be expressed as config.

Litmus test: "If I can grep my pipeline source code and find the word 'FIFA' or any specific client name, I have a design flaw. Client names belong only in config files."

Metric: Adding a new client goes from "weeks of custom development" to "create a JSON file and test."

Q: "How do you ensure zero downtime during a pipeline deployment?"

Your Answer: (This is Lesson 11, condensed.)

Architecture: Symlink-based deployment. The live pipeline path /pipeline/current is a symlink. Deploy the new version to its own directory, run smoke tests, then atomically swap the symlink. Rollback is one command: re-point the symlink to the previous version. For higher confidence, use canary deployment — route 10% of captures through the new version, monitor error rates for 24 hours, then roll out to 100%.

Defense layers: Retry with exponential backoff for transient failures. Circuit breaker to stop hammering dead services. Graceful degradation — if the NAS dies, capture data queues to local SSD and auto-flushes when NAS recovers. Health monitor daemon checking every critical system every 30 seconds via heartbeat. Slack alerts with severity classification.

Q: "How would you unify marker-based and markerless capture into a single pipeline?"

Your Answer: (This is Lesson 12, condensed.)

Key insight: The input and solving are completely different, but the output is identical — joint rotations per frame on a skeleton. That convergence point is where unification happens.

Architecture: Strategy Pattern. Define a universal internal format (CaptureResult) as the convergence point. Technology-specific code (ingest, cleanup) sits above. Universal code (retarget, validate, export, deliver) sits below. The pipeline reads the technology type from config and assembles the right strategies.

Roadmap: Phased over 2-3 years — foundation, parallel testing, validation parity, canary production on low-risk content, then full unification. Never ship unproven technology to hero performances.

5. Question Bank: Problem Solving & Debugging

Q: "You get a report that 30% of mocap takes from yesterday have broken hand poses. How do you investigate?"

Your Answer:

  1. Triage: Pull the error logs for yesterday's batch. Filter for hand-related validation failures. Are they all from the same capture session? Same actor? Same marker set?
  2. Isolate: Look at the raw .c3d data for a failing take. Are the hand markers present? Are they swapped? The most common cause of broken hand poses is marker occlusion during fast hand movements — grabbing, catching, pointing.
  3. Root cause: Check if the solver filled the gap with interpolation that went wrong. If so, the fix is to improve the gap-filling algorithm with spline interpolation instead of linear. If the markers were physically missing (fell off), the fix is a process change: better marker adhesion, or a camera arrangement that reduces hand occlusion.
  4. Systemic fix: Add a hand-specific validation checker that detects impossible finger angles (hyperextension, interpenetration) and flags them before delivery. This prevents the 30% error from ever reaching the game team.
  5. Metric: "We went from 30% failure rate to <2% after adding the finger angle validator."
Q: "The pipeline is running 3x slower than last week. How do you diagnose it?"

Your Answer:

  1. Check the dashboard: Look at processing time per take over the last 7 days. Is the slowdown gradual or sudden? Gradual suggests resource exhaustion (disk filling up, memory leak). Sudden suggests a change (software update, config change, new vendor data).
  2. Check recent changes: What was deployed last week? Was there a Maya update? A new Python library? A config change? git log --since="7 days ago" on the pipeline repo.
  3. Profile: Run Python's cProfile on a single take. Where is the time being spent? Is it I/O (disk/network), CPU (solving), or waiting (dependencies)?
  4. Check I/O: Is the NAS slow? Run dd or iperf to benchmark network throughput. Is another team flooding the NAS with large file copies?
  5. Check the farm: Are Deadline render nodes healthy? Are they queueing because other jobs are hogging capacity?
  6. Fix: The specific fix depends on the root cause, but the diagnostic methodology is what matters. Show the interviewer you have a systematic approach: metrics → recent changes → profiling → resource checks.

Key Takeaways

  • Structure every technical answer: Context → Architecture → Implementation → Tradeoff → Metric. This demonstrates both depth and strategic thinking.
  • Always mention metrics. "Reduced from X hours to Y minutes" is infinitely more compelling than "made it faster."
  • Config over code: This single principle answers half the interview questions. Client names in config, never in code.
  • Name the patterns: Adapter, Strategy, Factory, Circuit Breaker. Knowing the names gives weight to your answers.
  • Honesty + strategy: Acknowledge beginner production experience, but reframe it as "fresh architectural perspective with strong fundamentals."
  • Demonstrate systematic thinking: For debugging questions, show a methodology: metrics → changes → profiling → resource checks. Don't guess.
