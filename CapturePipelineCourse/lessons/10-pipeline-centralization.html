Pipeline Centralization: One Pipeline For 15 Vendor-Clients

How to serve 15 different clients who all want different things — without maintaining 15 different pipelines.

⚠️ The Core Problem

EA's capture studio doesn't serve one team. It serves ~15 different vendor-clients — different game teams, different external partners, different projects. Each wants output in a specific format, with specific naming, specific skeletons, delivered to a specific location. Without centralization, you end up maintaining 15 separate pipelines. That's a staffing and maintenance nightmare.

1. The Problem: N Clients × M Requirements = Chaos

Imagine this scenario:

Client Skeleton Naming Format Delivery
FC (FIFA) Team EA_Sports_v3 FC_YYYYMMDD_scNN_shNN FBX 2020 Perforce //depot/fc/mocap/
Madden Team Madden_v2 MN_shotName_take## FBX 2020 NAS \\nas02\madden\
Battlefield Frostbite_Generic BF_actorID_actionType FBX 2019 Perforce //depot/bf/anim/
Metaverse Partner MetaHuman_compat MV_avatarID_motion glTF / USD S3 bucket

If you write a custom script for each client, you end up with 15 export scripts, 15 naming validators, 15 delivery scripts. When you change the core pipeline (say, upgrading the FBX exporter), you have to update all 15. This doesn't scale.

2. The Solution: Shared Core + Per-Client Config

The architecture pattern is: one pipeline engine, many configuration profiles.

🧠 The Golden Rule

If you grep the core pipeline code and find the word "FIFA" or "Madden" or any client name, you have a design flaw. Client specifics belong ONLY in configuration files. The core code reads the config and acts generically.

Python: The Client Profile Registry

"""
Client Profile Registry — Loads and manages per-client configurations.
The core pipeline asks: "What does this client need?" and the registry answers.
"""
import json
import os
import logging
from pathlib import Path

log = logging.getLogger("client_registry")

class ClientProfile:
    """Represents one client's configuration."""
    
    def __init__(self, config_path):
        with open(config_path) as f:
            self._config = json.load(f)
        self.client_id = self._config["client_id"]
        self.display_name = self._config["display_name"]
        log.info(f"Loaded client profile: {self.display_name}")
    
    @property
    def skeleton_template(self):
        return self._config["skeleton"]["template"]
    
    @property
    def required_joints(self):
        return self._config["skeleton"]["required_joints"]
    
    @property
    def naming_pattern(self):
        return self._config["naming"]["pattern"]
    
    @property
    def export_format(self):
        return self._config["export"]["format"]
    
    @property
    def delivery_method(self):
        return self._config["delivery"]["method"]
    
    @property
    def delivery_destination(self):
        method = self.delivery_method
        if method == "perforce":
            return self._config["delivery"]["depot_path"]
        elif method == "nas":
            return self._config["delivery"]["nas_path"]
        elif method == "s3":
            return self._config["delivery"]["s3_bucket"]
        elif method == "sftp":
            return self._config["delivery"]["sftp_host"]
    
    def get(self, *keys, default=None):
        """Dot-path access into the config."""
        obj = self._config
        for key in keys:
            if isinstance(obj, dict) and key in obj:
                obj = obj[key]
            else:
                return default
        return obj


class ClientRegistry:
    """Discovers and manages all client profiles."""
    
    def __init__(self, config_dir):
        self.config_dir = Path(config_dir)
        self._profiles = {}
        self._load_all()
    
    def _load_all(self):
        """Scan the config directory for .json profiles."""
        for config_file in self.config_dir.glob("*.json"):
            try:
                profile = ClientProfile(config_file)
                self._profiles[profile.client_id] = profile
                log.info(f"Registered client: {profile.client_id}")
            except Exception as e:
                log.error(f"Failed to load {config_file}: {e}")
    
    def get_profile(self, client_id) -> ClientProfile:
        """Get a specific client's profile."""
        if client_id not in self._profiles:
            raise ValueError(
                f"Unknown client: '{client_id}'. "
                f"Available: {list(self._profiles.keys())}"
            )
        return self._profiles[client_id]
    
    def list_clients(self):
        """Return all registered client IDs."""
        return list(self._profiles.keys())

4. Python: The Output Adapter Factory

Different clients need different output formats. The Adapter + Factory pattern from Lesson 08 applies here too — but now for export instead of import:

Python: Per-Client Export Adapters

"""
Output Adapter Factory — Exports processed data in the client's required format.
"""
from abc import ABC, abstractmethod
import logging

log = logging.getLogger("export_adapter")

class BaseExportAdapter(ABC):
    """All export adapters implement export()."""
    
    @abstractmethod
    def export(self, scene_data, output_path, profile):
        pass

class FBXExportAdapter(BaseExportAdapter):
    """Export to FBX with client-specific settings."""
    
    def export(self, scene_data, output_path, profile):
        import maya.cmds as cmds
        import maya.mel as mel
        
        # Apply client-specific FBX settings
        fbx_version = profile.get("export", "fbx_version", default="FBX202000")
        mel.eval(f'FBXExportFileVersion -v "{fbx_version}"')
        
        if profile.get("export", "bake_animation", default=True):
            mel.eval('FBXExportBakeComplexAnimation -v true')
        
        if profile.get("export", "remove_namespaces", default=False):
            self._strip_namespaces()
        
        cmds.file(output_path, force=True, type="FBX export", exportAll=True)
        log.info(f"FBX exported: {output_path} (version: {fbx_version})")
    
    def _strip_namespaces(self):
        """Remove all namespaces — common client requirement."""
        import maya.cmds as cmds
        namespaces = cmds.namespaceInfo(lon=True, recurse=True)
        # Remove in reverse order (children first)
        for ns in reversed(namespaces):
            if ns not in ('UI', 'shared'):
                try:
                    cmds.namespace(removeNamespace=ns, mergeNamespaceWithRoot=True)
                except:
                    pass

class GLTFExportAdapter(BaseExportAdapter):
    """Export to glTF — used by metaverse/real-time clients."""
    
    def export(self, scene_data, output_path, profile):
        # glTF export might use a plugin or external converter
        import subprocess
        
        # First export FBX, then convert
        temp_fbx = output_path.replace('.gltf', '_temp.fbx')
        FBXExportAdapter().export(scene_data, temp_fbx, profile)
        
        # Convert using FBX2glTF (Facebook's open-source converter)
        subprocess.run([
            "FBX2glTF",
            "--input", temp_fbx,
            "--output", output_path,
            "--binary",  # .glb for smaller files
        ], check=True)
        
        import os
        os.remove(temp_fbx)
        log.info(f"glTF exported: {output_path}")

class USDExportAdapter(BaseExportAdapter):
    """Export to USD — for Pixar/Apple/NVIDIA ecosystem clients."""
    
    def export(self, scene_data, output_path, profile):
        import maya.cmds as cmds
        
        # Maya 2022+ has built-in USD support
        cmds.loadPlugin("mayaUsdPlugin", quiet=True)
        cmds.file(
            output_path,
            force=True,
            type="USD Export",
            exportAll=True,
            options="shadingMode=none;exportSkels=auto"
        )
        log.info(f"USD exported: {output_path}")


# === Factory: Pick adapter based on client config ===
EXPORT_ADAPTERS = {
    "fbx":  FBXExportAdapter,
    "gltf": GLTFExportAdapter,
    "glb":  GLTFExportAdapter,
    "usd":  USDExportAdapter,
    "usda": USDExportAdapter,
}

def get_export_adapter(profile):
    """Factory: returns the right exporter for a client."""
    fmt = profile.export_format.lower()
    adapter_class = EXPORT_ADAPTERS.get(fmt)
    if not adapter_class:
        raise ValueError(f"No export adapter for format: '{fmt}'")
    return adapter_class()

5. Custom Tools for Specific Clients

Beyond config-driven output, some clients need genuinely custom tooling. Here's what that looks like in practice:

Example: The "Metaverse Client"

🧠 What is "Metaverse Stuff"?

When EA mentioned building tools for a "metaverse client," this likely means:

  • Real-time avatar animation: Mocap driving a 3D avatar live (think VTubers, virtual concerts, or Meta's Horizon Worlds).
  • Lightweight skeletons: Metaverse avatars have simpler rigs than AAA game characters — maybe 40 joints instead of 200. You need a retarget template that maps your full mocap solve down to a simplified skeleton.
  • glTF/USD output: The metaverse ecosystem runs on web-friendly formats, not FBX. glTF (GL Transmission Format) is the "JPEG of 3D" — small, fast to load, browser-compatible.
  • Streaming-ready data: Instead of delivering a complete FBX file, you might need to stream joint rotations in real-time via WebSockets or a service like LiveLink.
  • Custom LOD requirements: Metaverse has aggressive performance constraints. Meshes might need to be under 5K triangles with aggressive texture compression.

When Custom Tools Are Justified

Scenario Config-Driven? Custom Tool?
Different naming convention ✅ Just a regex in JSON ❌ No
Different skeleton template ✅ Joint map in JSON ❌ No
Different export format (FBX vs glTF) ✅ Export adapter ❌ No
Real-time mocap streaming to avatars ❌ ✅ Custom LiveLink bridge
Custom facial blendshape mapping ❌ ✅ Client-specific retarget tool
Auto-LOD generation for mobile metaverse ❌ ✅ Custom decimation pipeline
Custom QA for client-specific constraints Partially ✅ Custom validator plugin
🎬 EA Studio Relevance

The interview mentioned that building custom solutions for specific vendor-clients is "heavy on the pipeline." This is why centralization is critical — you absorb as much as possible into configurable core behavior, and only build custom tools when the requirement is genuinely unique. Every custom tool is technical debt that someone has to maintain.

6. The Plugin Architecture for Client-Specific Extensions

When you DO need custom behavior, use a plugin architecture so it's isolated from the core:

Python: Client Plugin System

"""
Plugin Architecture — Client-specific extensions that plug into the core pipeline.
Each plugin is a Python module in the plugins/ directory.
"""
import importlib
import os
import logging
from pathlib import Path

log = logging.getLogger("plugin_system")

class PluginManager:
    """Discovers and loads client-specific plugins at runtime."""
    
    def __init__(self, plugin_dir):
        self.plugin_dir = Path(plugin_dir)
        self._plugins = {}
    
    def discover(self):
        """Scan for plugin modules (each is a .py file with a register() function)."""
        for py_file in self.plugin_dir.glob("*.py"):
            if py_file.name.startswith("_"):
                continue
            
            module_name = py_file.stem
            try:
                spec = importlib.util.spec_from_file_location(module_name, py_file)
                module = importlib.util.module_from_spec(spec)
                spec.loader.exec_module(module)
                
                if hasattr(module, 'register'):
                    plugin_info = module.register()
                    self._plugins[plugin_info['client_id']] = module
                    log.info(f"Loaded plugin: {module_name} → {plugin_info['client_id']}")
                else:
                    log.warning(f"Plugin {module_name} has no register() function, skipping")
            except Exception as e:
                log.error(f"Failed to load plugin {module_name}: {e}")
    
    def get_hooks(self, client_id):
        """Get all hook functions for a client."""
        module = self._plugins.get(client_id)
        if not module:
            return {}
        
        hooks = {}
        for hook_name in ['pre_export', 'post_export', 'custom_validate',
                          'custom_retarget', 'custom_delivery']:
            if hasattr(module, hook_name):
                hooks[hook_name] = getattr(module, hook_name)
        
        return hooks

7. The Centralized Pipeline Runner

Putting it all together — the core pipeline that reads a client config and runs the appropriate steps:

Python: The Unified Pipeline Runner

"""
Unified Pipeline Runner — The single entry point for all clients.
Takes capture data + client_id, does everything else automatically.
"""
import logging

log = logging.getLogger("pipeline_runner")

class PipelineRunner:
    def __init__(self, registry, plugin_manager, delivery_bot):
        self.registry = registry
        self.plugins = plugin_manager
        self.delivery = delivery_bot
    
    def process(self, input_path, client_id, output_dir):
        """Process one capture file for a specific client."""
        
        # 1. Load client profile
        profile = self.registry.get_profile(client_id)
        log.info(f"Processing for: {profile.display_name}")
        
        # 2. Load capture data (vendor-agnostic via Adapter Pattern)
        from pipeline.adapters import load_capture
        capture_data = load_capture(input_path)
        
        # 3. Run pre-export plugin hooks (if any)
        hooks = self.plugins.get_hooks(client_id)
        if 'pre_export' in hooks:
            hooks['pre_export'](capture_data, profile)
        
        # 4. Validate (core + optional custom)
        from pipeline.validation import CaptureValidator
        validator = CaptureValidator(profile)
        results = validator.validate_all(input_path)
        
        if 'custom_validate' in hooks:
            ok, msg = hooks['custom_validate'](capture_data, profile)
            if not ok:
                log.error(f"Custom validation failed: {msg}")
                return False
        
        # 5. Export in client's format
        from pipeline.export import get_export_adapter
        adapter = get_export_adapter(profile)
        
        output_filename = self._build_output_name(input_path, profile)
        output_path = os.path.join(output_dir, output_filename)
        adapter.export(capture_data, output_path, profile)
        
        # 6. Run post-export hooks
        if 'post_export' in hooks:
            hooks['post_export'](output_path, profile)
        
        # 7. Deliver
        self.delivery.deliver(output_path, profile)
        
        log.info(f"Pipeline complete: {output_path}")
        return True
    
    def _build_output_name(self, input_path, profile):
        """Generate output filename from naming convention."""
        import os, re
        from datetime import datetime
        
        base = os.path.splitext(os.path.basename(input_path))[0]
        ext = profile.export_format
        
        # Apply client naming pattern if specified
        return f"{base}.{ext}"

Just as export format varies by client, so does the delivery destination:

"""
Delivery Adapters — Each client's output goes to a different place.
"""
from abc import ABC, abstractmethod

class BaseDeliveryAdapter(ABC):
    @abstractmethod
    def deliver(self, file_path, profile):
        pass

class PerforceDelivery(BaseDeliveryAdapter):
    def deliver(self, file_path, profile):
        depot = profile.get("delivery", "depot_path")
        # P4Python submit (from Lesson 08)
        publish_to_perforce(file_path, depot)

class NASDelivery(BaseDeliveryAdapter):
    def deliver(self, file_path, profile):
        import shutil
        dest = profile.get("delivery", "nas_path")
        shutil.copy2(file_path, dest)

class S3Delivery(BaseDeliveryAdapter):
    def deliver(self, file_path, profile):
        import boto3
        bucket = profile.get("delivery", "s3_bucket")
        s3 = boto3.client('s3')

Key Takeaways

  • One core pipeline, many configs: Client specifics belong in JSON configuration files, never in the core code. If you grep for "FIFA" in your pipeline source, you have a design flaw.
  • Adapter + Factory for export: Each format (FBX, glTF, USD) has its own exporter class. A factory picks the right one based on the client config.
  • Plugin architecture for custom tools: When a client needs genuinely unique behavior, isolate it in a plugin module. The core pipeline calls plugin hooks at defined stages.
  • "Metaverse" = real-time avatars, lightweight rigs, web-friendly formats: glTF instead of FBX, 40 joints instead of 200, streaming instead of file delivery.
  • Multi-destination delivery: Perforce, NAS, S3, SFTP — each client gets their output delivered where they need it, driven by config.
  • Centralization ≠ one-size-fits-all: It means having a unified architecture that flexes to accommodate differences without forking the codebase.
