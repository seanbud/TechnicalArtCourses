Lesson 08

Lag Compensation

Collider rollback for instant-hit weapons, state synchronization for server-only events, and predicted projectiles that align across all clients.

🧠 Where We Are

Prediction (Lessons 05–07) keeps the local player feeling responsive. Lag compensation is the other side of the coin — it ensures that actions aimed at other players (shooting, abilities, status effects) feel fair despite network latency. This lesson covers all three of FishNet's lag compensation systems.

What You'll Learn

  • What lag compensation (collider rollback) is and why it matters
  • How to synchronize server-only states (stuns, EMPs, debuffs) with prediction using tick-based scheduling
  • How to implement raycast-based lag compensation with the RollbackManager (Pro Feature)
  • How to build predicted projectiles that visually align across all clients — without networking them

1. The Problem: Time Disagreement

In a server-authoritative architecture, there's a fundamental timing problem. When a player fires a shot, by the time the input reaches the server, the target has already moved. Without compensation, the server would check the raycast against the target's current position — not where the target was when the player actually fired. The result: shots that clearly hit on the client's screen miss on the server.

FishNet addresses this with three complementary systems:

System Use Case Requires Pro?
States Synchronizing server-only effects (stuns, EMPs, debuffs) so they fire at the same tick on all clients No
Raycast Rollback Rewinding colliders to historical positions for accurate hitscan weapons Yes
Predicted Projectiles Non-networked projectiles that visually align using time-based acceleration No

2. State Synchronization

Official Docs: States

Sometimes only the server knows when a state change should happen — an EMP that disables movement, a stun from a trap, or a speed debuff from a zone. If the server applies the effect immediately too, the owning client won't know about it until the next reconcile, causing a desync.

⚠️ Ideal Approach vs. This Workaround

FishNet's official guidance: ideally clients and server should run the same simulation. If both sides can detect the same condition (e.g., a trigger zone), do that instead — it's simpler and more robust. The tick-scheduling pattern below is for cases where only the server knows about the event.

The Pattern: Schedule a Future Tick

Instead of applying the state immediately, the server picks a tick slightly in the future (giving the RPC time to reach clients) and broadcasts it. All clients — and the server — then activate the state at that exact tick.

Step 1: Server Detects and Broadcasts

private void TimeManager_OnTick()
{
    // Server-only check — pretend ApplyEMP() is some
    // server-side condition detection.
    if (base.IsServer && ApplyEMP())
    {
        // Schedule 50ms into the future — enough time for
        // the RPC to reach clients before the tick fires.
        uint futureTicks = base.TimeManager.TimeToTicks(
            0.05f, TickRounding.RoundUp);

        ObserversSetEMPTick(
            base.TimeManager.LocalTick + futureTicks);
    }
}

// RunLocally = true → server also sets its own EMP tick.
[ObserversRpc(RunLocally = true)]
private void ObserversSetEMPTick(uint serverTick)
{
    // Convert server tick to this client's local tick.
    _empStartTick = base.TimeManager.TickToLocalTick(serverTick);
    // EMP lasts 1 second.
    _empEndTick = _empStartTick + base.TimeManager.TimeToTicks(
        1f, TickRounding.RoundUp);
}

Step 2: Replicate Checks the Tick Range

private uint _empStartTick = uint.MaxValue;
private uint _empEndTick = uint.MaxValue;

[Replicate]
private void Replicate(MotorData md, ReplicateState state = ...)
{
    uint localTick = base.TimeManager.LocalTick;

    // If we're inside the EMP window, block movement.
    if (localTick >= _empStartTick && localTick <= _empEndTick)
    {
        // Don't reset _empStartTick/_empEndTick here!
        // These values must persist through replays.
        return;
    }

    // Normal motor logic here...
}
🚨 Critical: Don't Reset Tick Values During Replay

If you clear _empStartTick after the EMP ends, replays won't see the correct values and the EMP will be skipped during reconcile → desync. Tick-based ranges naturally expire — they don't need cleanup.

💡 The Trade-Off

This pattern introduces a small delay (~50ms) before the state takes effect. In practice this is acceptable — the alternative (immediate application) causes visible desync. The delay is typically less than one frame at 30fps.

3. Raycast Lag Compensation (Pro Feature)

Official Docs: Raycast

For instant-hit weapons (hitscan guns, melee swings, beam attacks), FishNet Pro provides the RollbackManager — a system that rewinds all registered colliders to their historical positions, performs a standard Physics.Raycast(), and then restores them. From your code's perspective, raycasting "just works" against where targets were when the client fired.

Setup Checklist

  1. Add hitbox colliders as children of animated bones (head, body, etc.). Each hitbox must be on its own child GameObject with only a collider — any collider type works.
  2. Add ColliderRollback to the NetworkObject (typically root). Drag all hitbox GameObjects into its Collider Parents collection.

    Tip: Name hitbox objects with a "Hitbox" suffix, lock the inspector on ColliderRollback, search "Hitbox", and drag them all in at once.

  3. Add RollbackManager to your NetworkManager GameObject. Configure its settings (rollback frames, additional ticks for TickSmoother alignment).
⚠️ TickSmoother Compatibility

If using a Tick Smoother, place hitbox colliders on the target objects — not on smoothed objects (the object with the smoother or its children). The RollbackManager handles its own interpolation during rollback, so local smoothing on hitboxes is unnecessary. Set the smoother's Flat Interpolation value to match Additional Ticks in the RollbackManager.

Performing the Rollback

[Client]
private void Fire()
{
    // Capture the precise tick at the moment of firing.
    PreciseTick pt = base.TimeManager.GetPreciseTick(
        TickType.LastPacketTick);

    // Send to server with the tick.
    ServerFire(pt);
}

[ServerRpc]
private void ServerFire(PreciseTick pt)
{
    // Rewind all registered colliders to where they were
    // at the client's firing tick.
    base.RollbackManager.Rollback(
        pt,
        RollbackManager.PhysicsType.Physics,
        base.IsOwner  // Whether to also rollback self
    );

    // Perform a standard raycast — colliders are now
    // in their historical positions.
    if (Physics.Raycast(transform.position,
        transform.forward, out RaycastHit hit))
    {
        // Process hit normally.
        var target = hit.collider.GetComponentInParent<Health>();
        target?.TakeDamage(25f);
    }

    // CRITICAL: Restore colliders to current positions.
    base.RollbackManager.Return();
}
🧠 What's Happening Under the Hood

The RollbackManager stores position snapshots for every registered hitbox at every tick. When Rollback() is called, it moves all hitbox colliders back in time, interpolating between snapshots for sub-tick accuracy using the PreciseTick. After your raycast, Return() moves everything back. This is why you use PreciseTick instead of a raw tick — it captures the fractional position between ticks.

4. Predicted Projectiles

Official Docs: Projectiles

For weapons that fire visible, traveling projectiles (rockets, arrows, plasma bolts), FishNet provides a pattern that does not require networking the projectile at all. Instead, each peer spawns the projectile locally and accelerates it to compensate for latency — resulting in visual alignment across all clients.

💡 Why Not Network the Projectile?

A networked projectile would require a NetworkObject + spawn/despawn calls + per-tick state synchronization. For fast-moving projectiles, that's enormous bandwidth and CPU cost for minimal gameplay benefit. Instead, each client runs the projectile locally with a slight speed boost, and the server resolves hits authoritatively.

The Flow

  1. Owner fires: Spawns the projectile locally with 0 passed time (no catch-up needed). Sends an RPC to the server with the current tick.
  2. Server receives: Calculates how much time has passed since the client ticked, spawns the projectile with that passed time, then forwards to spectators.
  3. Spectators receive: Calculate passed time from the original tick, spawn and accelerate.

Step 1: Client Fires + Server Receives

[SerializeField] private PredictedProjectile _projectile;

// Max latency we'll compensate for — limits griefing from
// extremely laggy players.
private const float MAX_PASSED_TIME = 0.3f;

// Called locally when the player clicks fire.
private void ClientFire()
{
    Vector3 position = transform.position;
    Vector3 direction = transform.forward;

    // Spawn locally with 0 passed time — this is the
    // firing client, no catch-up needed.
    SpawnProjectile(position, direction, 0f);

    // Tell the server, including our current tick.
    ServerFire(position, direction, base.TimeManager.Tick);
}

[ServerRpc]
private void ServerFire(Vector3 position,
    Vector3 direction, uint tick)
{
    // Calculate how long it took for this RPC to arrive.
    float passedTime = (float)base.TimeManager
        .TimePassed(tick, false);

    // Cap at half the max — server's projectile doesn't need
    // as much catch-up as spectators.
    passedTime = Mathf.Min(MAX_PASSED_TIME / 2f, passedTime);

    SpawnProjectile(position, direction, passedTime);

    // Forward to spectators (everyone except the owner).
    ObserversFire(position, direction, tick);
}

[ObserversRpc(ExcludeOwner = true)]
private void ObserversFire(Vector3 position,
    Vector3 direction, uint tick)
{
    float passedTime = (float)base.TimeManager
        .TimePassed(tick, false);

    // Spectators get the full max — they're the farthest
    // behind the original firing moment.
    passedTime = Mathf.Min(MAX_PASSED_TIME, passedTime);

    SpawnProjectile(position, direction, passedTime);
}

private void SpawnProjectile(Vector3 position,
    Vector3 direction, float passedTime)
{
    PredictedProjectile pp = Instantiate(
        _projectile, position, Quaternion.identity);
    pp.Initialize(direction, passedTime);
}

Step 2: The PredictedProjectile Script

This is a plain MonoBehaviour — not a NetworkBehaviour! It has no NetworkObject. It's instantiated locally on each peer and destroyed independently.

// NOT a NetworkBehaviour — just MonoBehaviour.
public class PredictedProjectile : MonoBehaviour
{
    private Vector3 _direction;
    private float _passedTime = 0f;
    private const float MOVE_RATE = 5f;

    public void Initialize(Vector3 direction, float passedTime)
    {
        _direction = direction;
        _passedTime = passedTime;
    }

    private void Update()
    {
        Move();
    }

    private void Move()
    {
        float delta = Time.deltaTime;
        float passedTimeDelta = 0f;

        if (_passedTime > 0f)
        {
            // Exponential catch-up — fast at start, slow at end.
            // 8% of remaining per frame. Feels like smooth damp.
            float step = _passedTime * 0.08f;
            _passedTime -= step;

            // If remaining time is negligible, consume it all.
            if (_passedTime <= delta / 2f)
            {
                step += _passedTime;
                _passedTime = 0f;
            }

            passedTimeDelta = step;
        }

        // Move using base rate + catch-up delta.
        transform.position += _direction *
            (MOVE_RATE * (delta + passedTimeDelta));
    }
}
🧠 The Acceleration Curve

Using passedTime * 0.08f per frame creates an exponential decay curve — the projectile moves fastest right after spawning (when it has the most time to make up) and gradually returns to normal speed. This mimics a smooth damp and prevents the jarring "teleport then slow down" effect a flat catch-up rate would cause.

Step 3: Collision Handling

Since projectiles are local-only, collision logic must account for which peer is running:

private void OnCollisionEnter(Collision collision)
{
    // Client: show VFX, play audio.
    if (InstanceFinder.IsClient)
    {
        // Spawn impact particles, play hit sound
    }

    // Server: apply damage (authoritative).
    if (InstanceFinder.IsServer)
    {
        var target = collision.gameObject
            .GetComponent<Health>();
        if (target != null)
            target.TakeDamage(50f);
    }

    // Pool or destroy (both sides).
    Destroy(gameObject);
}
🚨 Accuracy Caveat

Because projectiles are not networked, there's a very small chance of alignment divergence between peers — especially at extremely high latencies or with fast-moving targets. In practice, these differences are visually insignificant and don't affect authoritative hit resolution (which happens on the server).

5. Choosing the Right Approach

Scenario Approach Why
Hitscan weapons (rifles, beams) Raycast Rollback (Pro) Instant hits need collider rewind — no visible projectile to accelerate.
Visible projectiles (rockets, arrows) Predicted Projectiles No networking overhead, visual alignment via acceleration. Works with free FishNet.
Server-only effects (stuns, EMPs) State Synchronization Tick-based scheduling ensures all peers activate the effect at the same simulation time.
Trigger-based effects (zones, AoE) Shared simulation (best) or States (fallback) If both sides can detect the same condition, let them — no special compensation needed.

Key Takeaways

  • State synchronization uses tick-based scheduling to align server-only events across all peers. Schedule effects slightly in the future and never reset tick values during replay.
  • Raycast rollback (Pro) rewinds hitbox colliders to historical positions using PreciseTick. Always call Return() after your raycast.
  • Predicted projectiles are non-networked GameObjects that use time-based acceleration to visually align across peers. Server handles hit resolution authoritatively.
  • The exponential catch-up pattern (passedTime * 0.08f per frame) produces smooth acceleration that decays naturally — no abrupt speed changes.
  • When possible, run the same simulation on both client and server rather than using lag compensation. Compensation is the fallback for server-only knowledge.
