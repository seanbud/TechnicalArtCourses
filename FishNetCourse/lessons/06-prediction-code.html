Lesson 06

Writing Prediction Code

From data structures to the full Replicate → Reconcile loop — the complete code walkthrough for controlled objects, non-controlled objects, ReplicateState, and advanced controls.

🧠 Where We Are

In Lesson 05 you configured PredictionManager, TimeManager, and NetworkObject — the inspector side. Now it's time to write the C# code that runs the prediction loop. Everything in this lesson takes place inside classes that extend NetworkBehaviour.

What You'll Learn

  • How to define ReplicateData and ReconcileData structures
  • The full flow: OnTick → [Replicate] → OnPostTick → CreateReconcile → [Reconcile]
  • How to sync non-controlled physics objects with the prediction system
  • The ReplicateState enum — what each flag means and when each fires
  • How to use states to prevent future movement, play one-time effects, and predict spectated inputs
  • How to add advanced controls (sprinting, ground checks, stamina) and reconcile all affected state

1. The Two Data Structures

FishNet's prediction system revolves around two structures that you define yourself. They must implement specific interfaces and carry a tick field.

Official Docs: Controlling an Object

ReplicateData (IReplicateData)

This struct holds everything the simulation needs to execute one tick of gameplay — the inputs. Think of it as a snapshot of what the player is doing at a given moment.

public struct ReplicateData : IReplicateData
{
    public bool Jump;
    public float Horizontal;
    public float Vertical;

    public ReplicateData(bool jump, float horizontal, float vertical) : this()
    {
        Jump = jump;
        Horizontal = horizontal;
        Vertical = vertical;
    }

    // Required boilerplate — FishNet uses these internally.
    private uint _tick;
    public void Dispose() { }
    public uint GetTick() => _tick;
    public void SetTick(uint value) => _tick = value;
}
💡 What Belongs in ReplicateData?

Anything that drives the simulation: movement axes, jump flags, fire buttons, ability selections. Do not include derived state (position, velocity) — that's what ReconcileData is for. Keep this struct small; it's sent over the network every tick.

ReconcileData (IReconcileData)

This struct holds the authoritative result after the server runs a replicate tick — the state snapshot the client rolls back to if its prediction was wrong.

public struct ReconcileData : IReconcileData
{
    public PredictionRigidbody PredictionRigidbody;

    public ReconcileData(PredictionRigidbody pr) : this()
    {
        PredictionRigidbody = pr;
    }

    private uint _tick;
    public void Dispose() { }
    public uint GetTick() => _tick;
    public void SetTick(uint value) => _tick = value;
}
🚨 The Golden Rule of ReconcileData

Include every piece of state that could possibly diverge between server and client. If your replicate method modifies a _stamina float, that float must be in your ReconcileData. If it reads from a _grounded bool computed by a raycast, that bool should be reconciled. Miss one field and you'll get "random" jitter that works in editor but fails with latency.

2. The Prediction Loop: Step by Step

Here's the complete lifecycle — from wiring up callbacks to sending the reconcile.

Step 1: Set Up Tick Callbacks

// PredictionRigidbody wraps a Rigidbody and handles
// velocity/force queuing for deterministic replay.
public PredictionRigidbody PredictionRigidbody;
private bool _jump;

private void Awake()
{
    PredictionRigidbody = ObjectCaches<PredictionRigidbody>.Retrieve();
    PredictionRigidbody.Initialize(GetComponent<Rigidbody>());
}

private void OnDestroy()
{
    ObjectCaches<PredictionRigidbody>.StoreAndDefault(ref PredictionRigidbody);
}

public override void OnStartNetwork()
{
    base.TimeManager.OnTick += TimeManager_OnTick;
    base.TimeManager.OnPostTick += TimeManager_OnPostTick;
}

public override void OnStopNetwork()
{
    base.TimeManager.OnTick -= TimeManager_OnTick;
    base.TimeManager.OnPostTick -= TimeManager_OnPostTick;
}
⚠️ OnTick vs OnPostTick

OnTick is where you call your [Replicate] method. OnPostTick is where you call CreateReconcile(). The reason: for physics-based prediction, you need the physics step to finish (which happens between OnTick and OnPostTick) before capturing the authoritative state. Non-physics controllers can do both in OnTick.

Step 2: Gather Input + Call Replicate

private void Update()
{
    // Capture single-frame inputs in Update — ticks don't
    // fire every frame, so GetKeyDown would be missed.
    if (base.IsOwner)
    {
        if (Input.GetKeyDown(KeyCode.Space))
            _jump = true;
    }
}

private void TimeManager_OnTick()
{
    RunInputs(CreateReplicateData());
}

private ReplicateData CreateReplicateData()
{
    if (!base.IsOwner)
        return default;

    float horizontal = Input.GetAxisRaw("Horizontal");
    float vertical = Input.GetAxisRaw("Vertical");
    ReplicateData md = new ReplicateData(_jump, horizontal, vertical);
    _jump = false;
    return md;
}
💡 Why Return default for Non-Owners?

The server receives inputs from the owner via the network, so it doesn't need to create its own. Non-owner clients receive forwarded inputs via state forwarding. Returning default signals to FishNet that this client has no local input to contribute.

Step 3: Implement the [Replicate] Method

[Replicate]
private void RunInputs(ReplicateData data,
    ReplicateState state = ReplicateState.Invalid,
    Channel channel = Channel.Unreliable)
{
    // Always use PredictionRigidbody for forces — never
    // modify Rigidbody.velocity directly.
    Vector3 forces = new Vector3(data.Horizontal, 0f, data.Vertical) * _moveRate;
    PredictionRigidbody.AddForce(forces);

    if (data.Jump)
    {
        PredictionRigidbody.AddForce(
            new Vector3(0f, _jumpForce, 0f), ForceMode.Impulse);
    }

    // Optional: extra gravity for snappier feel
    PredictionRigidbody.AddForce(Physics.gravity * 3f);

    // MUST call Simulate() at the end — this processes
    // all queued forces for this tick.
    PredictionRigidbody.Simulate();
}

Step 4: CreateReconcile + [Reconcile]

private void TimeManager_OnPostTick()
{
    // Always call CreateReconcile regardless of owner/server status.
    CreateReconcile();
}

public override void CreateReconcile()
{
    ReconcileData rd = new ReconcileData(PredictionRigidbody);
    ReconcileState(rd);
}

[Reconcile]
private void ReconcileState(ReconcileData data,
    Channel channel = Channel.Unreliable)
{
    PredictionRigidbody.Reconcile(data.PredictionRigidbody);
}
⚠️ CharacterController Special Case

If you're using a CharacterController instead of a Rigidbody, you must disable it before setting position and re-enable it after in your Reconcile method. The CharacterController uses a custom physics engine that doesn't refresh its internal state from transform changes unless toggled:

[Reconcile]
private void ReconcileState(ReconcileData data, Channel channel = Channel.Unreliable)
{
    _characterController.enabled = false;
    transform.localPosition = data.Position;
    _characterController.enabled = true;
}

3. Non-Controlled Objects

Not every networked physics object is player-controlled. Crates, balls, doors — anything with a Rigidbody that reacts to the world but has no controller input — still needs to participate in the prediction system to avoid desync during reconciliation replay.

Official Docs: Non-Controlled Object

The pattern is nearly identical to controlled objects, but simpler: no input polling, and default is always passed to the replicate method.

public class RigidbodySync : NetworkBehaviour
{
    public struct ReplicateData : IReplicateData
    {
        public ReplicateData(uint unused) : this() { }
        private uint _tick;
        public void Dispose() { }
        public uint GetTick() => _tick;
        public void SetTick(uint value) => _tick = value;
    }

    public struct ReconcileData : IReconcileData
    {
        public PredictionRigidbody PredictionRigidbody;
        public ReconcileData(PredictionRigidbody pr) : this()
        {
            PredictionRigidbody = pr;
        }
        private uint _tick;
        public void Dispose() { }
        public uint GetTick() => _tick;
        public void SetTick(uint value) => _tick = value;
    }

    private PredictionRigidbody PredictionRigidbody;

    private void Awake()
    {
        PredictionRigidbody =
            ResettableObjectCaches<PredictionRigidbody>.Retrieve();
        PredictionRigidbody.Initialize(GetComponent<Rigidbody>());
    }

    // Only OnPostTick needed — no input to process.
    public override void OnStartNetwork()
    {
        base.TimeManager.OnPostTick += TimeManager_OnPostTick;
    }

    public override void OnStopNetwork()
    {
        base.TimeManager.OnPostTick -= TimeManager_OnPostTick;
    }

    private void TimeManager_OnPostTick()
    {
        RunInputs(default);
        CreateReconcile();
    }

    [Replicate]
    private void RunInputs(ReplicateData md,
        ReplicateState state = ReplicateState.Invalid,
        Channel channel = Channel.Unreliable)
    {
        // No logic — let physics do its thing.
    }

    public override void CreateReconcile()
    {
        ReconcileData rd = new ReconcileData(PredictionRigidbody);
        ReconcileState(rd);
    }

    [Reconcile]
    private void ReconcileState(ReconcileData data,
        Channel channel = Channel.Unreliable)
    {
        PredictionRigidbody.Reconcile(data.PredictionRigidbody);
    }
}
💡 When to Use This Pattern

Add the RigidbodySync script to any networked physics object that reacts to collisions but isn't directly player-controlled — physics props, rolling barrels, kick-able objects. If the object doesn't have a NetworkObject at all, use OfflineRigidbody instead (covered in Lesson 05).

4. Understanding ReplicateState

The second parameter of every [Replicate] method is a ReplicateState — a set of flags that tells you where this data came from and how it's being used right now. Mastering this enum is the key to professional-grade prediction.

Official Docs: Understanding ReplicateState · Using States in Code

The Three Core Flags

Flag Who Sees It What It Means
Ticked Server + Clients This data has run (or previously ran) outside a reconcile loop. It's a "real" execution, not a replay.
Replayed Clients only This data is being re-executed during a reconcile. The server never replays — it's always authoritative.
Created Server + Clients This data was actually created by the controller (owner or server). If not created, the data is default — the controller sent nothing for this tick.

Common Flag Combinations

Replayed | Ticked | Created → Data is being replayed, previously ran outside reconcile, created by controller. → Most common state during reconcile for known inputs. Replayed (alone) → Data is being replayed, but this tick hasn't occurred outside reconcile yet. This is a "FUTURE" state — you don't know what the controller will actually do. Ticked (alone) → Running outside reconcile, but NOT created by controller. → Controller sent no input for this tick (e.g. they stopped moving).

Practical Pattern: One-Time Effects

You don't want to play a jump sound every time the input replays during reconcile — only on the first real execution:

[Replicate]
private void RunInputs(ReplicateData data, ReplicateState state = ...)
{
    if (data.Jump)
    {
        DoJump();

        // Only play audio on the FIRST real tick — not replays.
        if (state.ContainsTicked() && !state.ContainsReplayed())
            PlayJumpAudio();
    }
}

Practical Pattern: Prevent Future Movement

For spectated objects, you'll replay into ticks where you don't have real input yet — "future" ticks. Without handling these, a spectated player's character can snap forward aggressively.

CharacterController Approach

[Replicate]
private void RunInputs(ReplicateData data, ReplicateState state = ...)
{
    // If we're in the future (no real data for this tick),
    // don't move — prevents teleporting on spectated objects.
    if (state.IsFuture())
        return;

    if (data.Jump)
        _verticalVelocity = 10f;

    Vector3 movement = new(0f, _verticalVelocity, 0f);
    _verticalVelocity -= (float)base.TimeManager.TickDelta;
    _verticalVelocity = Mathf.Max(_verticalVelocity, -5f);
    _characterController.Move(movement);
}

Rigidbody Approach

With rigidbodies you can't just return — the rigidbody carries existing velocity through the physics step. Zero out velocities explicitly:

[Replicate]
private void RunInputs(ReplicateData data, ReplicateState state = ...)
{
    if (state.IsFuture())
    {
        _predictionRigidbody.Velocity(Vector3.zero);
        _predictionRigidbody.AngularVelocity(Vector3.zero);
        return;
    }

    Vector3 movement = new(data.Horizontal, 0f, 0f);
    _predictionRigidbody.AddVelocity(movement);
    _predictionRigidbody.Simulate();
}

Practical Pattern: Predicting Future Input

Sometimes you want to guess what a spectated player will do in future ticks — for smoother visuals. You can cache the last known input and replay it for a limited number of ticks:

Official Docs: Predicting States in Code

private ReplicateData _lastCreatedInput = default;

[Replicate]
private void RunInputs(ReplicateData data, ReplicateState state = ...)
{
    if (state.IsFuture())
    {
        uint lastTick = _lastCreatedInput.GetTick();
        // Only predict up to 2 ticks beyond last known input
        if ((data.GetTick() - lastTick) <= 2)
        {
            data.Vertical = _lastCreatedInput.Vertical;
            data.Horizontal = _lastCreatedInput.Horizontal;
            // Don't predict jumps — that would look weird
        }
    }
    else if (state == ReplicateState.ReplayedCreated)
    {
        _lastCreatedInput.Dispose();
        _lastCreatedInput = data;
    }

    // Run movement logic...
}
🧠 The Prediction Spectrum

Future handling is a design spectrum with three points:

  1. Block future entirely (IsFuture() → return): Safest. Object freezes during unknown ticks. Best for fast-paced competitive games.
  2. Predict N ticks: Middle ground. Smooth for ~33ms into the future, then freezes. Good for casual multiplayer.
  3. Predict all the way to real-time: Smoothest visually, but any misprediction causes visible corrections. Best for co-op games where precision doesn't matter.

5. Advanced Controls: Sprinting, Stamina, and the Reconcile Rule

Adding new mechanics to prediction is straightforward — it works exactly like single-player coding. The only extra requirement: reconcile anything that can diverge.

Official Docs: Advanced Controls

Adding Sprint to ReplicateData

public struct ReplicateData : IReplicateData
{
    public bool Jump;
    public bool Sprint;    // NEW
    public float Horizontal;
    public float Vertical;

    public ReplicateData(bool jump, bool sprint,
        float horizontal, float vertical) : this()
    {
        Jump = jump;
        Sprint = sprint;
        Horizontal = horizontal;
        Vertical = vertical;
    }
    // ... tick boilerplate
}

Using Sprint in [Replicate] with Stamina

private float _stamina;

[Replicate]
private void RunInputs(ReplicateData data, ReplicateState state = ...)
{
    float delta = (float)base.TimeManager.TickDelta;

    // Regen stamina at 3/sec
    _stamina += 3f * delta;

    float sprintCost = 6f * delta;
    Vector3 forces = new Vector3(data.Horizontal, 0f, data.Vertical) * _moveRate;

    if (data.Sprint && _stamina >= sprintCost)
    {
        _stamina -= sprintCost;
        forces *= 1.3f;  // 30% speed boost
    }

    if (data.Jump && IsGrounded())
    {
        PredictionRigidbody.AddForce(
            Vector3.up * _jumpForce, ForceMode.Impulse);
    }

    PredictionRigidbody.AddForce(forces);
    PredictionRigidbody.Simulate();
}

Updating ReconcileData to Include Stamina

public struct ReconcileData : IReconcileData
{
    public PredictionRigidbody PredictionRigidbody;
    public float Stamina;  // NEW — must reconcile this!

    public ReconcileData(PredictionRigidbody pr, float stamina) : this()
    {
        PredictionRigidbody = pr;
        Stamina = stamina;
    }
    // ... tick boilerplate
}

public override void CreateReconcile()
{
    ReconcileData rd = new ReconcileData(PredictionRigidbody, _stamina);
    ReconcileState(rd);
}

[Reconcile]
private void ReconcileState(ReconcileData data, Channel channel = ...)
{
    PredictionRigidbody.Reconcile(data.PredictionRigidbody);
    _stamina = data.Stamina;  // ← Restore stamina during rollback
}
🚨 The Desync Trap

Imagine you sprint successfully on the server, reducing stamina from 10 to 1. Then a reconcile fires. If you don't reconcile stamina, the client still has 1f stamina during replay — but replay needs to re-run the sprint, which requires ≥ sprintCost stamina. The sprint fails on replay → position diverges → visible jitter. Always reconcile external state.

Key Takeaways

  • ReplicateData = inputs, ReconcileData = state. Keep inputs small; include everything that can drift in the reconcile.
  • [Replicate] runs in OnTick, [Reconcile] runs in OnPostTick (for physics). Non-physics can do both in OnTick.
  • Non-controlled objects use the same pattern but pass default — they still need prediction scripts to avoid desync during replay.
  • ReplicateState flags tell you whether data is real, replayed, or future. Use IsFuture() to prevent or limit movement on spectated objects.
  • One-time effects (audio, VFX) should check ContainsTicked() && !ContainsReplayed() to avoid repeating during reconcile replay.
  • Advanced controls — sprinting, stamina, ground checks — work just like single-player code, but you must add any modified state to your ReconcileData or you'll get jitter.
