1. PredictionRigidbody Deep Dive

Official Docs: PredictionRigidbody

PredictionRigidbody wraps Unity's Rigidbody and queues all force changes so they can be replayed deterministically during reconcile. You've already used AddForce() and Simulate() in Lesson 06 — here we'll tour the full API and cover the rules you need to know for production use.

Initialization & Lifecycle

public PredictionRigidbody PredictionRigidbody;

private void Awake()
{
    // Retrieve from FishNet's internal pool to avoid allocations.
    PredictionRigidbody =
        ObjectCaches<PredictionRigidbody>.Retrieve();
    PredictionRigidbody.Initialize(GetComponent<Rigidbody>());
}

private void OnDestroy()
{
    // Return to pool.
    ObjectCaches<PredictionRigidbody>
        .StoreAndDefault(ref PredictionRigidbody);
}

Key API Methods

Method Description When to Use
AddForce(Vector3, ForceMode) Queues a force to apply during Simulate. Works identically to Rigidbody.AddForce(). Movement, jumps, knockback — anywhere you'd normally call rb.AddForce().
Velocity(Vector3) Sets the velocity directly (overrides any existing velocity). Teleport-like instant speed changes, dash abilities.
AddVelocity(Vector3) Adds to the current velocity without overriding it. Gentle nudges, wind effects, additive boosts.
AngularVelocity(Vector3) Sets the angular velocity directly. Spinning objects, rotation corrections.
Simulate() Processes all queued forces and applies them to the underlying Rigidbody. Must be called at the end of your [Replicate] method — and only there.
Reconcile(PredictionRigidbody) Restores the rigidbody to the reconcile snapshot. Called inside your [Reconcile] method.

Setting Velocity Directly

Some controllers set velocity explicitly rather than applying forces. PredictionRigidbody supports this — you still call Simulate() at the end:

[Replicate]
private void RunInputs(ReplicateData data, ReplicateState state = ...)
{
    Vector3 velocity = new Vector3(data.Horizontal, 0f, 0f) * _moveRate;
    PredictionRigidbody.Velocity(velocity);

    // Simulate() is required regardless of how velocities are set.
    PredictionRigidbody.Simulate();
}

2. Applying Forces From Outside Scripts

This is the pattern you'll use most in real games — bumpers, hazards, wind zones, elevators, and anything in the world that pushes a player around.

🚨 The Simulate() Rule

Never call PredictionRigidbody.Simulate() from outside the owning object's [Replicate] method. External scripts should only queue forces — the player's own replicate method processes them during Simulate(). This ensures forces are replayed correctly during reconcile.

Example: World Bumper Using NetworkTrigger

// On the bumper object in the world — NOT on the player.
public class Bumper : NetworkBehaviour
{
    [SerializeField] private float _launchForce = 15f;
    private NetworkTrigger _networkTrigger;

    private void Awake()
    {
        _networkTrigger = GetComponent<NetworkTrigger>();
        _networkTrigger.OnEnter += OnPlayerEnter;
    }

    private void OnDestroy()
    {
        if (_networkTrigger != null)
            _networkTrigger.OnEnter -= OnPlayerEnter;
    }

    private void OnPlayerEnter(Collider other)
    {
        // Get the player's prediction component.
        if (other.TryGetComponent<RigidbodyPlayer>(out var player))
        {
            // Queue force — do NOT call Simulate() here.
            player.PredictionRigidbody.AddForce(
                Vector3.up * _launchForce, ForceMode.Impulse);
        }
    }
}
💡 Why This Works

When a reconcile fires, FishNet replays the player's replicate method for every tick from the reconcile point forward. At each replayed tick, the physics step runs, the bumper trigger fires again during replay, AddForce() queues the impulse again, and the player's Simulate() processes it — identical to the original execution. That's deterministic replay.

3. NetworkColliders — Prediction-Safe Collision Events

Official Docs: Using NetworkColliders

Unity's built-in OnCollisionEnter/OnTriggerEnter callbacks have a fundamental problem with prediction: during reconcile replay, Unity doesn't re-fire Enter/Exit events for collisions that happened in the past. FishNet solves this with four drop-in replacement components:

Component Unity Equivalent Returns
NetworkCollision OnCollisionEnter/Stay/Exit Collider
NetworkCollision2D OnCollisionEnter2D/Stay2D/Exit2D Collider2D
NetworkTrigger OnTriggerEnter/Stay/Exit Collider
NetworkTrigger2D OnTriggerEnter2D/Stay2D/Exit2D Collider2D
🧠 FishNet Exclusive

FishNet is the only networking framework that accurately simulates Enter/Exit events during prediction replay — not even Fusion does this. This is critical for any game with physics-based hazards, pickups, or trigger zones.

The Three Callbacks

All four components expose the same three events:

  • OnEnter — Invoked when this collider first intersects another collider's bounds.
  • OnStay — Invoked while this collider remains in contact with another. Note: returns Collider, not Collision, to avoid hot-path allocations. If you need the Collision object, use Unity's native OnCollisionStay instead — it works correctly with prediction.
  • OnExit — Invoked when this collider leaves another collider's bounds.

Full Example: Player Collision Handler

// Placed on the player object alongside NetworkCollision component.
public class PlayerCollisionHandler : NetworkBehaviour
{
    [SerializeField] private AudioSource _hitSound;
    private NetworkCollision _networkCollision;

    private void Awake()
    {
        _networkCollision = GetComponent<NetworkCollision>();
        _networkCollision.OnEnter += OnCollisionEnter;
        _networkCollision.OnStay += OnCollisionStay;
        _networkCollision.OnExit += OnCollisionExit;
    }

    private void OnDestroy()
    {
        if (_networkCollision != null)
        {
            _networkCollision.OnEnter -= OnCollisionEnter;
            _networkCollision.OnStay -= OnCollisionStay;
            _networkCollision.OnExit -= OnCollisionExit;
        }
    }

    private void OnCollisionEnter(Collider other)
    {
        // Guard one-time effects against reconcile replays.
        if (!base.PredictionManager.IsReconciling)
            _hitSound.Play();

        // Forces ALWAYS apply — even during reconcile.
        // This ensures deterministic replay.
        PlayerMover pm = GetComponent<PlayerMover>();
        Vector3 dir = (transform.position -
            other.gameObject.transform.position).normalized;
        pm.PredictionRigidbody.AddForce(
            dir * 50f, ForceMode.Impulse);
    }

    private void OnCollisionStay(Collider other)
    {
        // Continuous contact logic here.
    }

    private void OnCollisionExit(Collider other)
    {
        // Cleanup logic, deactivate VFX, etc.
    }
}
⚠️ The Two-Rule Pattern

This is the universal pattern for collision callbacks in predicted code:

  1. Physics forces → always apply, including during reconcile replays. If you skip them during replay, determinism breaks.
  2. One-time effects (audio, particles, UI) → guard with IsReconciling. Otherwise the sound plays dozens of times per second during replays.

4. Custom Comparers

Official Docs: Custom Comparers

FishNet auto-generates equality comparers for your IReplicateData and IReconcileData structs to perform internal optimizations (e.g., skip sending identical inputs). Most types work automatically, but certain types cannot have comparers generated — specifically generics and arrays.

When You'll See the Error

If your struct contains an array or generic type, you'll get a console error like:

// This struct will produce a console error:
public struct MoveData : IReplicateData
{
    public Vector2 MoveDirection;
    public byte[] CustomData;  // ← byte[] needs a custom comparer
    // ...
}

Writing a Custom Comparer

The fix is simple: create a static method with the [CustomComparer] attribute. It takes two parameters of the type you're comparing and returns bool.

// Can be placed in any class — just needs to be static.
[CustomComparer]
public static bool CompareByteArray(byte[] a, byte[] b)
{
    bool aNull = (a is null);
    bool bNull = (b is null);

    // Both null → equal
    if (aNull && bNull) return true;
    // One null, other not → not equal
    if (aNull != bNull) return false;
    // Different lengths → not equal
    if (a.Length != b.Length) return false;

    // Compare each byte
    for (int i = 0; i < a.Length; i++)
    {
        if (a[i] != b[i]) return false;
    }

    return true;
}
💡 Performance Consideration

FishNet requires you to write your own comparer for these types because auto-generated byte-by-byte comparison might be expensive. You know your data — maybe you can compare a hash or length first. Keep comparers fast; they run every tick on the hot path.

When You Don't Need a Custom Comparer

Primitive types (float, int, bool), Unity value types (Vector3, Quaternion), and PredictionRigidbody all have auto-generated comparers. You only need custom comparers when the console tells you to.

5. Putting It All Together: Hazard Zone Example

Let's combine everything from Lessons 06 and 07 into a realistic scenario — a lava zone that damages and repels players with prediction-safe collisions:

// On the hazard zone object — has a NetworkTrigger + IsTrigger collider.
public class LavaZone : NetworkBehaviour
{
    [SerializeField] private float _repelForce = 20f;
    private NetworkTrigger _networkTrigger;

    private void Awake()
    {
        _networkTrigger = GetComponent<NetworkTrigger>();
        _networkTrigger.OnEnter += OnPlayerEnter;
    }

    private void OnDestroy()
    {
        if (_networkTrigger != null)
            _networkTrigger.OnEnter -= OnPlayerEnter;
    }

    private void OnPlayerEnter(Collider other)
    {
        if (!other.TryGetComponent<PlayerMover>(out var player))
            return;

        // 1. Apply knockback force — ALWAYS (replays too).
        Vector3 awayDir = (other.transform.position -
            transform.position).normalized;
        awayDir.y = 0.5f;  // Slight upward arc
        player.PredictionRigidbody.AddForce(
            awayDir.normalized * _repelForce, ForceMode.Impulse);

        // 2. Damage — only on non-replay ticks (server handles health).
        if (base.IsServerInitialized && !base.PredictionManager.IsReconciling)
        {
            player.GetComponent<Health>()?.TakeDamage(10f);
        }

        // 3. VFX — only on first real execution.
        if (!base.PredictionManager.IsReconciling)
        {
            PlayBurnEffect(other.transform.position);
        }
    }
}

Key Takeaways

  • PredictionRigidbody wraps all force application for deterministic replay. Use it exclusively — never modify Rigidbody.velocity directly.
  • Simulate() must be called only inside [Replicate] — external scripts queue forces but never call Simulate.
  • NetworkCollision/NetworkTrigger replace Unity's collision callbacks with prediction-safe versions that correctly re-fire during reconcile replay.
  • In collision handlers: forces always apply, one-time effects guard with IsReconciling.
  • Custom comparers are only needed for arrays and generics — the console will tell you. Keep them fast.
