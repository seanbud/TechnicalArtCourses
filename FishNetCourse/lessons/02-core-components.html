Lesson 02

The FishNet Foundations

Mastering NetworkObject, NetworkManager, and RPC routing logic.

Introduction

Unlike Quantum where you query the Frame state for entities, FishNet is deeply integrated into Unity's component system. Data and logic are attached directly to GameObjects using specialised behaviours.

🧠 Key Concept: NetworkBehaviour

Any script that needs to synchronize data over the network or send RPCs (Remote Procedure Calls) must inherit from NetworkBehaviour instead of MonoBehaviour.

Core Architecture

1. NetworkManager

This is the heart of FishNet. It manages the server instance, client connections, transport layer (e.g., Tugboat or Unity Transport via Relay), and scene loading. In your SlopBox project, this is injected and heavily wrapped by ConnectionManager to track player states.

2. NetworkObject

Every GameObject that exists on the network requires a NetworkObject component. This assigns the object a unique network ID. Prefabs with a `NetworkObject` must be registered with the `NetworkManager`'s spawnable prefabs list.

Lifecycle Hooks

In standard Unity, you use Awake() and Start(). In FishNet, you must wait until the network is ready before interacting with networked variables.

Code Example: Basic Lifecycle

public class HealthSystem : NetworkBehaviour {
    // Fired when the network starts for this object (Client OR Server)
    public override void OnStartNetwork() {
        base.OnStartNetwork();
        if (IsServerInitialized) {
            Debug.Log("I am the Server!");
        }
        if (IsOwner) {
            Debug.Log("I have authority over this object!");
        }
    }
}
⚠️ Common Pitfall: The Start() Trap

Do not initialize network states or send RPCs inside Unity's Start() method! The NetworkObject may not have been assigned its network ID yet. Always use OnStartNetwork, OnStartServer, or OnStartClient.

RPCs: Remote Procedure Calls

RPCs are how you explicitly send commands across the network. There are two primary types you need to know:

1. [ServerRpc]

Sent by a client, executed only on the server. Use this for actions like validating a shot, requesting to open a door, or applying damage.

[ServerRpc(RequireOwnership = false)]
public void RequestOpenDoor(GameObject door) {
    // Only runs on the Server. Validate logic here.
}

2. [ObserversRpc]

Sent by the server, executed on all clients observing the object. Use this for ephemeral visual effects, like spawning particles or playing a sound that doesn't need strict state synchronization.

[ObserversRpc]
public void PlayHitEffect(Vector3 position) {
    // Runs on all clients
    Instantiate(sparkPrefab, position, Quaternion.identity);
}
💡 Pro Tip: SyncVars vs RPCs

Always prefer SyncVar for persistent state (like Health, Armor, Team) and only use RPCs for ephemeral events (Explosions, UI notifications, Voice lines). If a player joins late, they will receive the current value of a SyncVar, but they will not re-receive old RPCs!

Key Takeaways

  • Use NetworkBehaviour for custom logic.
  • Replace Start() with OnStartNetwork() when dealing with multiplayer initialization.
  • Clients ask the Server to do things via [ServerRpc].
  • Servers tell Clients about physical events via [ObserversRpc].
  • Use SyncVar for state, RPCs for events.
