Lesson 02

The FishNet Foundations

Mastering NetworkObject, NetworkBehaviour, RPCs, SyncVars, and the lifecycle hooks that replace your Quantum Systems.

⚠️ The Biggest Trap

The single most common mistake by newcomers to FishNet: initializing networked state inside Unity's Start() or Awake(). At that point, the NetworkObject may not have been assigned a network ID yet. All networked initialization must happen inside FishNet's lifecycle hooks: OnStartNetwork(), OnStartServer(), or OnStartClient().

1. The Component Hierarchy

In Quantum, the architecture is: Quantum Simulation (ECS) → Unity View (MonoBehaviours). In FishNet, everything lives in Unity's component system, with a clear hierarchy of networked components:

┌──────────────────────────────────────────────────────────────────────────┐ │ FISHNET COMPONENT HIERARCHY │ ├──────────────────────────────────────────────────────────────────────────┤ │ │ │ ┌─────────────────────────────────────────────────────────┐ │ │ │ NetworkManager │ │ │ │ (Singleton — one per scene) │ │ │ │ │ │ │ │ ├── ServerManager → runs authoritative logic │ │ │ │ ├── ClientManager → handles local client connection │ │ │ │ ├── TransportManager → Tugboat, FishyUnityTransport │ │ │ │ ├── SceneManager → networked scene loading │ │ │ │ ├── TimeManager → tick clock, TickDelta │ │ │ │ └── ObserverManager → controls who sees what │ │ │ └─────────────────────────────────────────────────────────┘ │ │ │ │ │ ┌──────────────┼──────────────┐ │ │ ▼ ▼ ▼ │ │ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ │ │NetworkObject│ │NetworkObject│ │NetworkObject│ (one per networked │ │ │ (Player) │ │ (Enemy) │ │ (Pickup) │ GameObject) │ │ ├─────────────┤ ├─────────────┤ ├─────────────┤ │ │ │NetworkBhvr │ │NetworkBhvr │ │NetworkBhvr │ (your custom │ │ │NetworkBhvr │ │NetworkBhvr │ └─────────────┘ scripts) │ │ │NetworkBhvr │ └─────────────┘ │ │ └─────────────┘ │ │ │ │ RULE: Every GameObject with ANY networked behaviour must │ │ have a NetworkObject component. Period. │ │ │ └──────────────────────────────────────────────────────────────────────────┘
🧠 Quantum Mapping

Think of NetworkObject as the equivalent of a Quantum EntityRef — it gives the object a unique identity across the network. Think of NetworkBehaviour as the equivalent of a Quantum System — it contains the logic that runs on both server and client. The key difference: in Quantum, Systems are global singletons that iterate over components. In FishNet, NetworkBehaviours are attached directly to GameObjects.

2. Lifecycle Hooks: When to Initialize What

FishNet provides a series of callbacks that fire at different stages of a NetworkObject's life. Understanding their execution order is critical:

Hook Fires On Use For Quantum Equivalent
Awake() Unity standard Non-networked setup ONLY (caching components) —
OnStartNetwork() Server AND Client Subscribing to TimeManager events OnInit in a Quantum System
OnStartServer() Server only Spawning AI, initializing game state —
OnStartClient() Client only (including Host's client) Setting up UI, camera, input bindings —
OnOwnershipServer() Server, when ownership changes Transferring authority between players —
OnOwnershipClient() Client, when ownership changes Enabling/disabling input for new owner —
OnStopNetwork() Server AND Client Cleanup, unsubscribing events —

Execution Order on Spawn

// When a NetworkObject is spawned, hooks fire in this order:
// 1. Awake()              ← Unity standard. No network ID yet!
// 2. OnStartNetwork()     ← Network ID assigned. Safe for TimeManager.
// 3. OnStartServer()      ← Server-only. Safe for server initialization.
// 4. OnStartClient()      ← Client-only. Safe for UI, camera, input.
//
// When checking ownership inside these hooks:
// • IsOwner         → true if the local player owns this object
// • IsServerInitialized → true if running on the server
// • IsClientInitialized → true if running on a client
// • IsHostInitialized   → true if running as Host (server + client)

3. Full Example: A Networked Health System

Let's build a complete, production-quality health system. This demonstrates SyncVars, lifecycle hooks, ownership checks, and RPCs working together:

C#: NetworkHealth.cs — Full Server-Authoritative Health

using FishNet.Object;
using FishNet.Object.Synchronizing;
using UnityEngine;
using System;

/// <summary>
/// Server-authoritative health system using SyncVars.
/// Only the server can modify health. Clients observe changes via OnChange.
/// </summary>
public class NetworkHealth : NetworkBehaviour
{
    [Header("Configuration")]
    [SerializeField] private float _maxHealth = 100f;

    // ── SyncVars: Server → All Clients automatically ──
    // SyncVar replaces Quantum's "component fields on an entity."
    // The server sets the value; all clients receive the update.
    private readonly SyncVar<float> _currentHealth = new();
    private readonly SyncVar<bool> _isDead = new();

    // ── Events for local systems (UI, VFX, etc.) ──
    public event Action<float, float> OnHealthChanged;  // (oldHP, newHP)
    public event Action OnDied;

    // ── Properties ──
    public float CurrentHealth => _currentHealth.Value;
    public float MaxHealth => _maxHealth;
    public bool IsDead => _isDead.Value;

    // ═══════════════════════════════════════════════════════
    // LIFECYCLE
    // ═══════════════════════════════════════════════════════

    public override void OnStartNetwork()
    {
        base.OnStartNetwork();

        // Subscribe to SyncVar change events.
        // This fires on BOTH server and clients when the value changes.
        _currentHealth.OnChange += HandleHealthChanged;
        _isDead.OnChange += HandleDeathChanged;
    }

    public override void OnStartServer()
    {
        base.OnStartServer();

        // Server initializes health. Clients receive this via SyncVar.
        _currentHealth.Value = _maxHealth;
        _isDead.Value = false;
    }

    public override void OnStopNetwork()
    {
        base.OnStopNetwork();
        _currentHealth.OnChange -= HandleHealthChanged;
        _isDead.OnChange -= HandleDeathChanged;
    }

    // ═══════════════════════════════════════════════════════
    // SERVER-SIDE LOGIC (only executes on the server)
    // ═══════════════════════════════════════════════════════

    /// <summary>
    /// Apply damage. Called by hazard systems, combat systems, etc.
    /// Only runs on the server — clients cannot call this directly.
    /// </summary>
    [Server]
    public void TakeDamage(float amount, NetworkBehaviour source = null)
    {
        if (_isDead.Value) return;

        float oldHealth = _currentHealth.Value;
        _currentHealth.Value = Mathf.Max(0f, oldHealth - amount);

        Debug.Log($"[Health] {gameObject.name} took {amount} damage " +
                  $"({oldHealth} → {_currentHealth.Value})");

        if (_currentHealth.Value <= 0f)
        {
            _isDead.Value = true;
            // Tell all clients to play death VFX
            PlayDeathEffectRpc();
        }
        else
        {
            // Tell all clients to play hit VFX
            PlayHitEffectRpc(_currentHealth.Value / _maxHealth);
        }
    }

    [Server]
    public void Heal(float amount)
    {
        if (_isDead.Value) return;
        _currentHealth.Value = Mathf.Min(_maxHealth, _currentHealth.Value + amount);
    }

    // ═══════════════════════════════════════════════════════
    // RPCs (Server → Clients)
    // ═══════════════════════════════════════════════════════

    /// <summary>
    /// Ephemeral VFX event — late-joining players don't need to see this.
    /// </summary>
    [ObserversRpc]
    private void PlayHitEffectRpc(float healthPercent)
    {
        // Spawn particles, flash screen red, camera shake, etc.
        Debug.Log($"[VFX] Hit! Health at {healthPercent:P0}");
    }

    [ObserversRpc]
    private void PlayDeathEffectRpc()
    {
        Debug.Log("[VFX] Death effect triggered!");
        OnDied?.Invoke();
    }

    // ═══════════════════════════════════════════════════════
    // SYNCVAR CALLBACKS (fires on ALL clients when value changes)
    // ═══════════════════════════════════════════════════════

    private void HandleHealthChanged(float oldValue, float newValue, bool asServer)
    {
        // Update UI, health bar, etc.
        OnHealthChanged?.Invoke(oldValue, newValue);
    }

    private void HandleDeathChanged(bool oldValue, bool newValue, bool asServer)
    {
        if (newValue && !oldValue)
        {
            OnDied?.Invoke();
        }
    }
}
⚠️ [Server] vs [ServerRpc]

[Server] means "this method only runs on the server, period." If a client calls it, nothing happens. Use it for internal server-side logic. [ServerRpc] means "a client can call this, and it will execute on the server." Use it when a client needs to request an action. In the health system above, TakeDamage is [Server] because only other server-side systems (hazards, combat) should call it. A client should never directly set their own health.

4. RPCs: The Communication Channels

RPCs (Remote Procedure Calls) are how you explicitly send commands across the network. Think of them as typed messages between server and client:

┌──────────────────────────────────────────────────────────────────────────┐ │ RPC FLOW DIAGRAM │ ├──────────────────────────────────────────────────────────────────────────┤ │ │ │ CLIENT (Owner) SERVER │ │ ───────────── ────── │ │ Player presses "Use" │ │ │ │ │ ├──→ [ServerRpc] RequestOpenDoor() ──→ Server validates: │ │ │ • Is player close enough? │ │ │ • Is door unlocked? │ │ │ • Is player alive? │ │ │ │ │ │ IF valid: │ │ │ door.Open() │ │ │ │ │ │ │ ←── [ObserversRpc] OnDoorOpened() ──────┘ │ │ │ │ │ ▼ │ │ ALL CLIENTS play door animation │ │ and sound effect │ │ │ │ ──────────────────────────────────────────────────────────────────── │ │ │ │ [ServerRpc] = Client → Server (request an action) │ │ [ObserversRpc] = Server → ALL Clients (broadcast an event) │ │ [TargetRpc] = Server → ONE Client (private message) │ │ │ └──────────────────────────────────────────────────────────────────────────┘

C#: Door Interaction Example

C#: NetworkDoor.cs — ServerRpc + ObserversRpc Pattern

using FishNet.Object;
using FishNet.Object.Synchronizing;
using FishNet.Connection;
using UnityEngine;

public class NetworkDoor : NetworkBehaviour
{
    [SerializeField] private float _interactionRange = 2.5f;
    [SerializeField] private Animator _animator;

    // Persistent state — late-joiners see if the door is open or closed
    private readonly SyncVar<bool> _isOpen = new();

    public override void OnStartClient()
    {
        base.OnStartClient();
        // When a new client joins, sync the door to its current state
        _isOpen.OnChange += OnDoorStateChanged;
        // Apply initial state
        if (_isOpen.Value)
            _animator.Play("DoorOpen", 0, 1f); // snap to open
    }

    /// <summary>
    /// Client asks the server to open this door.
    /// RequireOwnership = false: ANY client can call this, not just the owner.
    /// </summary>
    [ServerRpc(RequireOwnership = false)]
    public void RequestOpenDoorServerRpc(NetworkConnection caller = null)
    {
        if (_isOpen.Value) return; // Already open

        // Validate: is the caller close enough?
        if (caller?.FirstObject != null)
        {
            float dist = Vector3.Distance(
                caller.FirstObject.transform.position,
                transform.position
            );
            if (dist > _interactionRange)
            {
                Debug.LogWarning($"[Door] Player too far: {dist:F1}m");
                return;
            }
        }

        // Server approves — update SyncVar
        _isOpen.Value = true;
        // Fire ephemeral VFX/SFX to all clients
        PlayDoorOpenEffectRpc();
    }

    [ObserversRpc]
    private void PlayDoorOpenEffectRpc()
    {
        _animator.SetTrigger("Open");
        // AudioSource.PlayClipAtPoint(openSFX, transform.position);
    }

    private void OnDoorStateChanged(bool prev, bool next, bool asServer)
    {
        // React to SyncVar change (e.g., update UI prompts)
    }
}

5. SyncVar vs RPC: The Decision Matrix

This is a critical decision you'll make constantly. The wrong choice leads to subtle, hard-to-debug issues (especially with late-joining players):

Scenario Use SyncVar? Use RPC? Why
Player health ✅ SyncVar ❌ Persistent state. Late-joiners must see current health.
Door open/closed ✅ SyncVar ❌ Persistent state. Late-joiners must see door position.
Explosion VFX ❌ ✅ ObserversRpc Ephemeral event. Late-joiners don't need to see old explosions.
Chat message ❌ ✅ ObserversRpc Ephemeral event. Log if needed, but don't sync perpetually.
Player team/name ✅ SyncVar ❌ Persistent. Scoreboard must show on join.
Damage number popup ❌ ✅ ObserversRpc Ephemeral visual. Gone after 1 second.
"Player X opened a door" log ❌ ✅ ObserversRpc Ephemeral notification.
Trap armed/disarmed state ✅ SyncVar ❌ Persistent. Late-joiner must know if trap is active.
🧠 The Rule of Thumb

Ask yourself: "If a player joins the game 5 minutes late, do they need to know about this?" If yes → SyncVar. If no → RPC. SyncVars automatically send their current value to any newly connected client. RPCs only fire once and are not replayed for late-joiners (unless you use BufferLast = true).

6. The Zenject Bridge: Dependency Injection + FishNet

Your Slop-Box project uses Zenject for dependency injection. This creates a timing challenge: when do Zenject bindings resolve relative to FishNet's lifecycle?

C#: The Timing Dance

// In Slop-Box's NetworkMovement.cs, you'll see:

public class NetworkMovement : NetworkBehaviour
{
    private IInputProvider _input;

    [Inject]
    public void Construct(IInputProvider input)
    {
        // Zenject calls this BEFORE OnStartNetwork.
        // This is safe because Construct() only stores references.
        _input = input;
    }

    public override void OnStartNetwork()
    {
        base.OnStartNetwork();
        // FishNet calls this AFTER Zenject injection.
        // Now safe to use both _input AND network features.
        TimeManager.OnTick += TimeManager_OnTick;
    }
}

// ORDER OF EXECUTION:
// 1. Awake()                 ← Unity
// 2. [Inject] Construct()    ← Zenject
// 3. Start()                 ← Unity (DON'T use for network init!)
// 4. OnStartNetwork()        ← FishNet (SAFE for network init)
// 5. OnStartServer()         ← FishNet (server only)
// 6. OnStartClient()         ← FishNet (client only)
⚠️ The Gotcha

Zenject's [Inject] fires during object construction (before Start()), so it's safe to cache references there. But do NOT use injected network services (like NetworkManager) in Construct() to send RPCs or modify SyncVars — the NetworkObject hasn't been initialized yet. Always wait for OnStartNetwork().

7. Spawning Networked Objects

In Quantum, entities exist in the simulation frame and the view layer instantiates visual representations. In FishNet, you spawn GameObjects on the server and they automatically replicate to all clients:

C#: Server-Side Player Spawning

using FishNet;
using FishNet.Object;
using FishNet.Connection;
using FishNet.Managing;
using UnityEngine;

public class PlayerSpawner : MonoBehaviour
{
    [SerializeField] private GameObject _playerPrefab;
    [SerializeField] private Transform[] _spawnPoints;

    private NetworkManager _networkManager;

    private void Start()
    {
        _networkManager = InstanceFinder.NetworkManager;
        // Subscribe to player connections (server-side only)
        _networkManager.SceneManager.OnClientLoadedStartScenes += OnClientReady;
    }

    private void OnClientReady(NetworkConnection conn, bool asServer)
    {
        if (!asServer) return;

        // Pick a spawn point
        Transform spawn = _spawnPoints[conn.ClientId % _spawnPoints.Length];

        // Instantiate on the server
        GameObject player = Instantiate(
            _playerPrefab,
            spawn.position,
            spawn.rotation
        );

        // Spawn on the network — this replicates to ALL clients
        // and assigns ownership to the connecting player.
        _networkManager.ServerManager.Spawn(player, conn);

        Debug.Log($"[Spawner] Spawned player for Client {conn.ClientId}");
    }
}
💡 Ownership Matters

The second argument to Spawn(player, conn) assigns ownership to that connection. Only the owner can send [ServerRpc] calls (unless RequireOwnership = false), and IsOwner will return true only on that client. This is how you control which player controls which character — the same concept as Quantum's PlayerRef.

8. NetworkBehaviour Properties Quick Reference

These properties are available in any NetworkBehaviour and replace a lot of the context you'd get from Quantum's Frame:

Property Type What It Means
IsServerInitialized bool True if this object exists on the server (authoritative)
IsClientInitialized bool True if this object exists on a client
IsOwner bool True if the local player owns this specific object
IsHostInitialized bool True if running as Host (server + client simultaneously)
Owner NetworkConnection The connection that owns this object (can be null for server-owned)
OwnerId int Integer ID of the owning connection (-1 for server-owned)
ObjectId int Unique network ID for this object (like Quantum's EntityRef)
TimeManager TimeManager Access to the tick clock, TickDelta, OnTick events

Key Takeaways

  • Every networked GameObject needs a NetworkObject. It's the network identity. Custom scripts inherit from NetworkBehaviour, not MonoBehaviour.
  • Never initialize network state in Start() or Awake(). Use OnStartNetwork() for shared init, OnStartServer() for server-only, OnStartClient() for client-only.
  • [ServerRpc] = Client asking the server to do something. Always validate on the server side — never trust client input.
  • [ObserversRpc] = Server telling all clients about an event. Use for ephemeral VFX, sounds, notifications.
  • SyncVars are for persistent state (health, door state, team). RPCs are for ephemeral events (explosions, sounds). Ask: "Does a late-joiner need this?"
  • Zenject's [Inject] fires before OnStartNetwork(). Cache references in Construct(), but don't use network features until OnStartNetwork().
  • Spawn objects on the server with Spawn(obj, conn). The second argument assigns ownership, controlling which client can send ServerRpcs for that object.
