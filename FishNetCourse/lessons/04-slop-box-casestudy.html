Lesson 04

Slop-Box Architecture Case Study

A deep analysis of a real FishNet codebase — what's production-ready, what's missing, and how to scale it for a co-op Prince of Persia.

🧠 Why This Lesson Matters

Theory is one thing — reading someone else's real code is another. This lesson walks through the actual Slop-Box codebase script by script, annotates the architectural decisions, identifies gaps, and gives you a clear list of what needs to change to ship a polished co-op experience.

1. The Architecture Overview

Slop-Box uses a clean layered architecture. Here's the full system map, from bootstrap to physics:

💡 Architecture Grade: A-

This is a genuinely good architecture. The separation of NetworkMovement (FishNet-aware) from CharacterMovement (pure simulation) is the single most important pattern in FishNet development. It means you can test CharacterMovement offline without a network, iterate on physics feel without touching netcode, and swap networking frameworks without rewriting movement math. The only missing piece is completeness of the reconcile state (see Section 4).

2. NetworkBootstrap: The Entry Point

The bootstrap layer wraps FishNet's NetworkManager into a clean API:

C#: NetworkBootstrap.cs (Annotated)

public class NetworkBootstrap : MonoBehaviour
{
    private NetworkManager _networkManager;

    // Zenject injects the NetworkManager found in the scene.
    // This fires BEFORE OnStartNetwork — safe for reference caching.
    [Inject]
    public void Construct(NetworkManager networkManager)
    {
        _networkManager = networkManager;
    }

    /// <summary>
    /// Two calls = Host mode (Server + Client simultaneously).
    /// This is the standard pattern for co-op.
    /// </summary>
    public void StartHost()
    {
        _networkManager.ServerManager.StartConnection();  // ← Server logic
        _networkManager.ClientManager.StartConnection();  // ← Local client
    }

    // ARCHITECTURE NOTE:
    // This class is a "thin wrapper." It doesn't add logic — it just
    // provides a clean API for other systems to call. This keeps
    // the FishNet API from leaking into game code.
    //
    // In production, you'd add:
    // - Lobby/matchmaking integration
    // - Unity Relay allocation before starting
    // - Error handling for failed connections
    // - Reconnection logic
}

3. ConnectionManager: Tracking Players

This script hooks into FishNet's OnRemoteConnectionState event to track when players join and leave:

C#: ConnectionManager.cs (Annotated)

public class ConnectionManager : MonoBehaviour
{
    private NetworkManager _networkManager;

    // Events that other systems can subscribe to
    public event Action<NetworkConnection> OnPlayerConnected;
    public event Action<NetworkConnection> OnPlayerDisconnected;

    // Server-side only: all currently connected clients
    public IReadOnlyDictionary<int, NetworkConnection> ConnectedClients =>
        _networkManager?.ServerManager?.Clients ??
        new Dictionary<int, NetworkConnection>();

    private void OnEnable()
    {
        // Subscribe to FishNet's connection state changes
        _networkManager.ServerManager.OnRemoteConnectionState +=
            HandleRemoteConnectionState;
    }

    private void HandleRemoteConnectionState(
        NetworkConnection conn,
        RemoteConnectionStateArgs args)
    {
        switch (args.ConnectionState)
        {
            case RemoteConnectionState.Started:
                OnPlayerConnected?.Invoke(conn);
                break;
            case RemoteConnectionState.Stopped:
                OnPlayerDisconnected?.Invoke(conn);
                break;
        }
    }

    // ARCHITECTURE NOTE:
    // This is a MEDIATOR pattern. Instead of every system
    // (UI, spawning, game loop) subscribing directly to
    // FishNet's raw events, they subscribe to ConnectionManager.
    // This gives you one place to add logging, filtering,
    // and connection validation.
    //
    // WHAT'S MISSING:
    // - No reconnection handling (if a player disconnects
    //   and reconnects, they get a new ClientId)
    // - No player data (name, team) associated with connections
    // - No host migration (if the host leaves, game ends)
}

4. NetworkMovement: The CSP Wrapper

This is the most critical script in the project. It bridges between FishNet's tick system and CharacterMovement's pure simulation:

C#: NetworkMovement.cs (Fully Annotated)

public class NetworkMovement : NetworkBehaviour
{
    [SerializeField] private CharacterMovement _simulation;

    // Camera pitch is synced as a SyncVar instead of through MoveData.
    // This is a BANDWIDTH OPTIMIZATION — see Section 5.
    private readonly SyncVar<float> _replicatedPitch = new();
    private readonly SyncVar<float> _replicatedCameraYaw = new();

    private IInputProvider _input;

    [Inject]
    public void Construct(IInputProvider input) => _input = input;

    // ─── LIFECYCLE ───

    public override void OnStartNetwork()
    {
        base.OnStartNetwork();
        // Hook into FishNet's fixed tick loop — NOT Unity's Update().
        // TimeManager.OnTick fires at the NetworkManager's tick rate
        // (typically 60Hz), which is independent of frame rate.
        TimeManager.OnTick += TimeManager_OnTick;
        TimeManager.OnPostTick += TimeManager_OnPostTick;
    }

    // ─── TICK LOOP ───

    private void TimeManager_OnTick()
    {
        // Step 1: Collect input (only relevant for the owner)
        MoveData data = BuildMoveData();

        // Step 2: Execute simulation.
        // On OWNER: runs immediately as prediction.
        // On SERVER: runs authoritatively with the client's input.
        Replicate(data);
    }

    private void TimeManager_OnPostTick()
    {
        // Step 3: After the server executes [Replicate],
        // update the camera SyncVars so non-owners can see
        // where this player is looking.
        if (IsServerInitialized)
        {
            var pivot = _simulation?.CameraPivot;
            if (pivot != null)
            {
                _replicatedPitch.Value = pivot.eulerAngles.x;
                _replicatedCameraYaw.Value = pivot.eulerAngles.y;
            }
        }
    }

    // ─── INPUT COLLECTION ───

    private MoveData BuildMoveData()
    {
        // Non-owners return empty data (they don't generate input)
        if (!IsOwner || _input == null) return default;

        // IMPORTANT: Cursor lock check prevents movement while
        // interacting with UI. Input still collected for camera.
        if (Cursor.lockState != CursorLockMode.Locked)
            return new MoveData { RotationY = ..., Pitch = ... };

        return new MoveData
        {
            InputDirection = _input.MoveInput,
            RotationY      = _simulation.CameraYaw,
            Pitch          = _simulation.CameraPitch,
            JumpPressed    = _input.IsJumpHeld, // ← BUG: see Section 6
            JumpHeld       = _input.IsJumpHeld,
            SprintHeld     = _input.IsSprintHeld
        };
    }

    // ─── CSP CORE ───

    [Replicate]
    private void Replicate(MoveData data,
        ReplicateState state = ReplicateState.Invalid,
        Channel channel = Channel.Unreliable)
    {
        // CORRECT: uses TickDelta, not Time.deltaTime
        float delta = (float)TimeManager.TickDelta;
        _simulation.Simulate(data, delta);
    }

    public override void CreateReconcile()
    {
        MoveState state = new MoveState
        {
            Position   = transform.position,
            RotationY  = transform.eulerAngles.y,
            Pitch      = ...,
            Velocity   = _simulation.Velocity,
            IsGrounded = _simulation.IsGrounded,
            JumpTimer    = 0f,   // ← ISSUE: see Section 6
            StaggerTimer = 0f    // ← ISSUE: see Section 6
        };
        Reconcile(state);
    }

    [Reconcile]
    private void Reconcile(MoveState state,
        Channel channel = Channel.Unreliable)
    {
        _simulation.SetState(state);
    }
}

5. The SyncVar Pitch Optimization

One of the smartest decisions in the codebase is how camera rotation is handled. Instead of stuffing the full camera state into MoveData (which is sent 60 times per second), the camera pitch and yaw are synced via lightweight SyncVars:

Approach Bandwidth per Tick Affects Physics? Verdict
Camera rotation in MoveData ~16 bytes (two floats sent every tick) Only if aiming affects movement Wasteful for non-physics data
Camera rotation via SyncVar ~8 bytes (only when changed, delta-compressed) No — purely cosmetic for spectators ✅ Correct choice
🧠 The Bandwidth Rule

MoveData should only contain values that affect the physics simulation. If the camera pitch doesn't change how the CharacterController moves (it doesn't — the body faces the camera yaw, not the pitch), then it's cosmetic data that doesn't need to be replicated through the prediction system. Syncing it as a SyncVar is cheaper and simpler. This is the equivalent of putting "visual-only" data in Quantum's View layer instead of the Simulation.

However, there's a subtlety: RotationY (the camera yaw) IS in MoveData because it drives the character's body rotation and movement direction. If you removed it, the server wouldn't know which direction "forward" means for the WASD input, and the character would drift.

6. Reconcile Completeness Audit

This is the most important health check for any CSP implementation. Every variable that is modified inside [Replicate] (via CharacterMovement.Simulate()) and affects future tick calculations MUST be included in MoveState. Let's audit:

Variable in CharacterMovement Modified in Simulate()? Affects Future Ticks? In MoveState? Status
transform.position ✅ Yes (CharacterController.Move) ✅ Yes ✅ Yes (Position) ✅ OK
transform.rotation ✅ Yes (Quaternion.Slerp) ✅ Yes (body facing affects movement) ✅ Yes (RotationY) ✅ OK
Velocity ✅ Yes (combined horizontal + vertical) ✅ Yes ✅ Yes ✅ OK
IsGrounded ✅ Yes (CharacterController.isGrounded) ✅ Yes (affects jump availability) ✅ Yes ✅ OK
_jumpTimer ✅ Yes (incremented during jump sustain) ✅ Yes (determines jump sustain duration) ⚠️ Hardcoded to 0f ❌ BUG
_staggerTimer ✅ Yes (decremented during stagger) ✅ Yes (determines input lockout) ⚠️ Hardcoded to 0f ❌ BUG
_coyoteTimer ✅ Yes (decremented when airborne) ✅ Yes (allows late jumps after leaving edge) ❌ Not included ❌ MISSING
_hasJumped ✅ Yes (set true on jump, reset on ground) ✅ Yes (prevents double-jump) ❌ Not included ❌ MISSING
_horizontalVelocity ✅ Yes (MoveTowards each tick) ✅ Yes (used as base for next tick's acceleration) ⚠️ Derived from Velocity.Flat() in SetState ⚠️ FRAGILE
_verticalVelocity ✅ Yes (gravity, jump force) ✅ Yes ⚠️ Derived from Velocity.y in SetState ⚠️ FRAGILE
⚠️ The JumpTimer / StaggerTimer Bug

In CreateReconcile(), JumpTimer and StaggerTimer are hardcoded to 0f instead of reading from the simulation. This means every reconcile resets these timers to zero. The symptom: if a client is mid-jump when a reconcile arrives, the replay loop restarts with JumpTimer=0, causing the jump sustain to re-apply and the character to jump higher than intended. Similarly, knockback stagger will be shorter on the client than on the server.

C#: The Fix — Complete CreateReconcile

public override void CreateReconcile()
{
    MoveState state = new MoveState
    {
        Position     = transform.position,
        RotationY    = transform.eulerAngles.y,
        Pitch        = _simulation.CameraPitch,
        Velocity     = _simulation.Velocity,
        IsGrounded   = _simulation.IsGrounded,
        JumpTimer    = _simulation.JumpTimer,      // ← FIX: read actual value
        StaggerTimer = _simulation.StaggerTimer,   // ← FIX: read actual value
        CoyoteTimer  = _simulation.CoyoteTimer,    // ← ADD: missing field

7. What's Missing: Networking Hazards & Co-op Systems

Currently, the Slop-Box project networks player movement but doesn't yet network environmental hazards or co-op interactions. Here's what needs to be built:

System Current State Required For Co-op Recommended Approach
Hazard Damage Local only — each client resolves damage independently Server-authoritative damage Hazards call [Server] TakeDamage() on the health system. Clients see damage VFX via [ObserversRpc].
Knockback Sync Local ApplyKnockback on CharacterMovement Server-authoritative knockback Server calls ApplyKnockback → next [Reconcile] includes the new StaggerTimer and velocity, so CSP handles the visual sync automatically.
Ragdoll Death Local ragdoll activation All players see ragdoll at the same time [ObserversRpc] TriggerRagdoll(). Ragdolls are cosmetic — they don't need to be deterministic.
Puzzle Triggers Not implemented Both players stand on pressure plates, etc. Server checks conditions (both players in trigger zone). State changes via SyncVar.
Pickup Items Not implemented One player picks up item, both see it disappear [ServerRpc] request → server validates → despawn NetworkObject → [ObserversRpc] for VFX.
💡 The Pattern

Notice the repeating pattern: Client requests → Server validates → Server applies → Server notifies. This is the fundamental loop of server-authoritative game design. In Quantum, this loop is implicit (all input goes through deterministic sim). In FishNet, you build it explicitly with [ServerRpc] → server logic → SyncVar/[ObserversRpc].

8. The JumpPressed Bug

There's a subtle input bug in BuildMoveData() that you should fix immediately:

C#: The Bug

// CURRENT CODE:
return new MoveData
{
    JumpPressed = _input.IsJumpHeld,  // ← BUG: uses "Held" for "Pressed"
    JumpHeld    = _input.IsJumpHeld,  // ← Both are the same value!
};

// THE PROBLEM:
// JumpPressed should be TRUE only on the FIRST tick the button is down.
// JumpHeld should be TRUE for every tick the button is held.
//
// Currently, both read IsJumpHeld, so JumpPressed fires EVERY tick
// while the button is held. In CharacterMovement.Simulate(), the jump
// check uses JumpPressed:
//
//   if (data.JumpPressed && canJump) → fire jump
//
// Because _hasJumped prevents double-jumping, this bug is MASKED —
// the player can't actually double-jump. But if you ever add wall-jumps
// or other jump-consuming mechanics, this bug will re-surface.

// THE FIX:
return new MoveData
{
    JumpPressed = _input.IsJumpPressed,  // ← Single-frame press detection
    JumpHeld    = _input.IsJumpHeld,     // ← Continuous hold
};

9. Scaling Recommendations

To ship Slop-Box as a polished co-op experience, here's the priority list:

Priority Task Effort Impact
P0 Fix CreateReconcile — add JumpTimer, StaggerTimer, CoyoteTimer, HasJumped 30 min Eliminates jump jitter and knockback desyncs
P0 Fix JumpPressed vs JumpHeld input bug 5 min Prevents future jump-mechanic bugs
P1 Network hazard damage (server-authoritative health) 2-4 hours Core co-op gameplay — both players see consistent damage
P1 Network ragdoll deaths via ObserversRpc 1-2 hours Visual consistency for co-op partner
P2 Add Physics.simulationMode = Script configuration 30 min Prevents physics desync on different framerates
P2 Add reconnection support to ConnectionManager 2-3 hours Graceful recovery from network drops
P3 Integrate Unity Relay or Steam Networking transport 4-8 hours Required for players who can't port-forward

Key Takeaways

  • The architecture is solid. Separating NetworkMovement (network-aware) from CharacterMovement (pure simulation) is the correct production pattern. Maintain this separation as you add new systems.
  • The reconcile state is incomplete. JumpTimer and StaggerTimer are hardcoded to 0, and CoyoteTimer/HasJumped are missing entirely. Fix this first — it's the source of any jitter or inconsistent jump behaviour between host and client.
  • The SyncVar pitch optimization is clever. Camera rotation that doesn't affect physics should always be synced separately from the CSP data to save bandwidth.
  • JumpPressed has a masked bug. Both JumpPressed and JumpHeld read the same input. Fix it now before adding wall-jumps or multi-jump mechanics.
  • Hazards, damage, and interactions all follow the same pattern: Client requests → Server validates → Server applies → Server notifies. Build this loop for every new networked system.
  • The Zenject + FishNet integration works cleanly. Continue using [Inject] for reference caching and OnStartNetwork() for network initialization.
