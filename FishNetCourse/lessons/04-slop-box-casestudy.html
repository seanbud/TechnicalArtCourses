Lesson 04

Slop-Box Architecture Case Study

Analyzing NetworkMovement.cs and decoupling visual simulation from the networking layer.

Introduction

Let's look closely at your existing project, Slop-Box. The core of your networking logic exists inside SlopBox.Core.Movement.NetworkMovement. You have chosen an excellent architectural pattern: wrapping a separate simulation script.

🧠 Key Architecture: Decoupling Simulation from Network

The NetworkMovement.cs script handles purely the network data (Client-Side Prediction, Inputs, RPCs, SyncVars). It passes data into CharacterMovement.cs, which handles the actual rigidbodies and math. This is highly analogous to Quantum's System vs View logic.

Reviewing the Implementation

Inside NetworkMovement.cs, you properly use the FishNet `TimeManager` to hook into the Tick system:

public override void OnStartNetwork() {
    base.OnStartNetwork();
    TimeManager.OnTick += TimeManager_OnTick;
    TimeManager.OnPostTick += TimeManager_OnPostTick;
}

This guarantees that input collection and physics execution happens in sync with the server's expected clock step, avoiding the float precision errors typical of Update() loops.

The Replicate Data Collection

In your TimeManager_OnTick(), you collect local hardware input constraints (checking `CursorLockMode`) and assemble your MoveData struct. The simulation is correctly driven purely by this structure.

Pitch and Yaw Replicating

Your implementation of updating Camera Pitch independently of physics via `SyncVar` is a clever solution to a common problem:

private readonly SyncVar<float> _replicatedPitch = new();

Because camera rotation (where the player is looking) often diverges from the physics orientation (where the body is facing), decoupling the Yaw for the simulation and the Pitch via a simple SyncVar saves massive bandwidth over sending full Quaternions in the Replicate data.

⚠️ Potential Pitfall: Reconcile Depth

Inside CreateReconcile(), you ensure properties like JumpTimer and StaggerTimer are captured. If these variables are somehow modified inside Update() inside CharacterMovement.cs rather than explicitly via Replicate ticks, CSP will eventually fail and clients will vibrate as they continually mispredict logic based on variable framerates. Always ensure timers run off `TimeManager.TickDelta`, just like Quantum's deterministic frames.

Key Takeaways

  • Your wrap-around architecture (NetworkLogic → View/Simulation) is the correct path for clean, testable code.
  • Hooking into TimeManager.OnTick ensures CSP inputs align with sever clock rates.
  • Ensure every single variable that can affect a player's physical position is encapsulated inside the Reconcile struct to prevent client-side desyncs.
