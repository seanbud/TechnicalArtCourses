Lesson 01

FishNet vs Quantum & Server Topologies

Bridging the gap from deterministic lockstep to server-authoritative state replication — and understanding the network topology that will power your co-op Prince of Persia.

⚠️ The Core Paradigm Shift

In Quantum, every client runs the exact same deterministic simulation. If any client deviates by a single floating-point bit, the game desyncs and breaks. In FishNet, clients are allowed to be wrong. They predict, they guess, they drift — and the server periodically corrects them. This is fundamentally different. Everything you know about Quantum's precision and determinism must be replaced with a tolerance-based, correction-driven mental model.

1. The Two Philosophies of Multiplayer

Before diving into FishNet's specifics, let's clearly map the two paradigms side-by-side. You already know one of these perfectly — the point is to show exactly where the differences lie:

Dimension Photon Quantum (What You Know) FishNet (What You're Learning)
Core Model Deterministic Lockstep / Rollback Server-Authoritative State Replication
Math Types Fixed-point (FP, FPVector3) — no floats allowed Standard Unity floats, Vector3, Quaternion
Simulation Every client runs the full simulation identically Server runs the authoritative sim; clients predict locally
Latency Handling Input delay + rollback of the global state Client-Side Prediction + per-object reconciliation
View Separation Strict: Quantum Systems (logic) vs Unity View (visuals) Flexible: NetworkBehaviour handles both, but best practice separates them
Physics Custom deterministic physics engine Unity's built-in PhysX (non-deterministic)
Input Handling Polled per frame into a Quantum Input struct Polled per tick into an IReplicateData struct
State Authority Shared — all clients own the truth Centralized — one server owns the truth
Adding a New Client Full state snapshot transfer + rollback catch-up Server spawns objects and SyncVars auto-sync current values
Prototyping Speed Slower — everything must be deterministic Faster — use standard Unity APIs
🧠 The Mental Model Shift

In Quantum, you think: "Every client must execute identical math." In FishNet, you think: "The server is the boss. Everyone else is guessing, and the boss corrects them." This means you can use Unity's standard Transform, Rigidbody, and CharacterController APIs directly — because the server will fix any float-precision drift via reconciliation.

2. Server Topologies: Who Runs the Authority?

In Quantum, there's typically a Photon Cloud relay that synchronizes inputs, but every client runs the same simulation — there is no "server" in the traditional sense. In FishNet, someone must run the authoritative server logic. You have three main options:

Option A: Dedicated Server

A headless Unity instance runs in the cloud. No player has a zero-ping advantage. It's the gold standard for competitive games.

Option B: Host Mode (Player-as-Server)

One player's Unity instance runs both the Client and the Server simultaneously. Other players connect to this Host. This is the most common pattern for co-op games because it requires zero infrastructure costs.

Option C: Relay-Backed Host Mode (What Slop-Box Uses)

Same as Host Mode, but connections are routed through a relay service (like Unity Relay, Steam Networking, or Photon Realtime). The relay handles NAT punchthrough so players don't need to port-forward their routers. The Host still runs the server logic — the relay is just a dumb packet bouncer.

⚠️ The Relay Misconception

A common mistake: people think using a Relay service means "nobody is the server." That's wrong. The Relay only handles packet routing and NAT punchthrough. The Host player's Unity instance still runs the ServerManager, executes all authoritative [Replicate] logic, and broadcasts [Reconcile] states. The Relay is a postal service — it delivers letters but doesn't read or write them.

3. Latency Impact by Topology

Understanding ping distribution is critical for a platformer where frame-perfect jumps and responsive movement matter:

Topology Host Player Latency Remote Player Latency Best For
Dedicated Server 20-60ms (datacenter ping) 20-60ms (same) Competitive PvP, large lobbies
Host Mode 0ms (local) 30-150ms (peer-to-peer) Small co-op (2-4 players)
Relay-Backed Host 0ms (local) 40-180ms (peer + relay hop) Co-op when NAT traversal needed
💡 Slop-Box Relevance

For your Prince of Persia co-op game with 2 players, Host Mode (optionally relay-backed) is the correct choice. The Host player will experience perfect, zero-latency controls. The remote player will experience 30-80ms of latency — which is where Client-Side Prediction (Lesson 3) becomes essential. In a co-op game, the Host advantage is less problematic than in PvP because you're on the same team.

4. What Happens When You Press Jump: Quantum vs FishNet

Let's trace a single button press through both systems to see where the experience diverges:

In Quantum (What You Know)

Frame 100: Player presses Jump
  → Input is polled into Quantum's Input struct
  → Quantum's deterministic simulation advances Frame 100 locally
  → Input is sent to Photon Cloud
  → Other clients receive the input, also advance Frame 100
  → If any input arrived late, ALL clients rollback to the
     earliest unconfirmed frame and re-simulate forward
  → Result: every client has identical state, always

In FishNet (What You're Learning)

Tick 100: Player presses Jump
  → Input is polled into a MoveData struct
  → CLIENT immediately executes [Replicate] locally (PREDICTION)
     → Character jumps instantly on screen — zero perceived latency
  → Input is sent to SERVER via unreliable channel
  → SERVER receives input, executes [Replicate] authoritatively
  → SERVER fires [Reconcile] back to client with the TRUE state
  → CLIENT compares its predicted position to the server's state
     → If close enough: no visible correction
     → If diverged: snap to server state, replay all unacked inputs
  → Result: client FEELS instant, server ENFORCES truth
🧠 The Key Difference

In Quantum, correctness is guaranteed by design — identical math produces identical results. In FishNet, correctness is enforced by correction — the server periodically tells clients "you're wrong, here's where you actually are." The client's job is to be wrong as rarely as possible, so corrections are invisible.

5. Why Networked Physics "Feels Slow" Without CSP

If you use FishNet's built-in NetworkTransform component to sync a Rigidbody, without Client-Side Prediction, here's the experience for a remote player:

┌──────────────────────────────────────────────────────────────────────┐ │ NAIVE NETWORKED PHYSICS (NO CSP) │ ├──────────────────────────────────────────────────────────────────────┤ │ │ │ Timeline (Remote Player pressing "Jump"): │ │ │ │ t=0ms Player presses Jump │ │ t=0ms Input sent to Host as [ServerRpc] │ │ ⏳ (nothing happens locally — waiting for server) │ │ t=60ms Host receives input │ │ t=60ms Host applies jump force to Rigidbody │ │ t=60ms Host syncs new position via NetworkTransform │ │ ⏳ (network transit back to client) │ │ t=120ms Client receives updated position │ │ t=120ms Character visually jumps on screen │ │ │ │ TOTAL INPUT DELAY: ~120ms (feels like input lag) │ │ │ │ For a platformer requiring frame-perfect timing, │ │ this is UNACCEPTABLE. │ │ │ └──────────────────────────────────────────────────────────────────────┘

This is why naive networked physics — using NetworkTransform to sync Rigidbody positions — feels terrible for players. There's a full round-trip delay (RTT) between pressing a button and seeing a result. For a Prince of Persia game with wall-running, double-jumps, and grab-ledge mechanics, 120ms of input delay would make the game unplayable.

💡 The Solution: CSP

Client-Side Prediction (CSP) eliminates this delay by letting the client immediately execute the jump locally, then verifying with the server afterwards. From the player's perspective, the jump happens at t=0ms. We'll build this system in detail in Lesson 3.

6. FishNet's Transport Layer

In Quantum, the transport is handled by Photon's cloud infrastructure. In FishNet, you choose your transport:

Transport Protocol Use Case
Tugboat (Default) UDP LAN, direct connections, development
FishyUnityTransport Unity Transport (UDP/DTLS) Unity Relay integration, production co-op
FishySteamworks Steam Networking Steam-based games with built-in relay
Bayou WebSocket (TCP) WebGL builds, browser games

The transport sits beneath the NetworkManager and is swappable — your game logic doesn't care which transport you use. For Slop-Box with a co-op Prince of Persia game, you'd likely start with Tugboat for local development and switch to FishyUnityTransport + Unity Relay or FishySteamworks for production.

7. Setting Up a Minimal FishNet Project

Here's the minimal scene setup to get a FishNet host running — the equivalent of setting up a Quantum Runner in Quantum:

├── NetworkManager          ← FishNet's NetworkManager component
│   ├── Component: NetworkManager
│   ├── Component: Tugboat (Transport)

│   └── Component: NetworkAutoStart
└── SpawnPoint
    └── (empty transform marking where players appear)

C#: Minimal Auto-Start Host Script

C#: NetworkAutoStart.cs — Boots FishNet in Host Mode

using FishNet;
using FishNet.Managing;
using FishNet.Transporting;
using UnityEngine;

/// <summary>
/// Automatically starts FishNet as a Host (Server + Client) on scene load.
/// In production, you'd replace this with a lobby/matchmaking flow.
/// </summary>
public class NetworkAutoStart : MonoBehaviour
{
    [Header("Configuration")]
    [SerializeField] private NetworkManager _networkManager;
    [SerializeField] private bool _startAsHost = true;

    [Header("Connection")]
    [SerializeField] private string _address = "localhost";
    [SerializeField] private ushort _port = 7770;

    private void Start()
    {
        if (_networkManager == null)
            _networkManager = InstanceFinder.NetworkManager;

        if (_networkManager == null)
        {
            Debug.LogError("[NetworkAutoStart] No NetworkManager found!");
            return;
        }

        // Configure the transport address/port
        Transport transport = _networkManager.TransportManager.Transport;
        if (transport is FishNet.Transporting.Tugboat.Tugboat tugboat)
        {
            tugboat.SetClientAddress(_address);
            tugboat.SetPort(_port);
        }

        if (_startAsHost)
        {
            // Start as Host = Server + Client simultaneously
            _networkManager.ServerManager.StartConnection();
            _networkManager.ClientManager.StartConnection();
            Debug.Log($"[NetworkAutoStart] Started HOST on {_address}:{_port}");
        }
        else
        {
            // Start as Client only (connecting to a remote host)
            _networkManager.ClientManager.StartConnection();
            Debug.Log($"[NetworkAutoStart] Connecting to {_address}:{_port}");
        }
    }
}
🧠 Host = Server + Client

Notice how starting as a Host requires two calls: ServerManager.StartConnection() and ClientManager.StartConnection(). The Host player is simultaneously a server (running authoritative game logic) and a client (rendering the game and processing local input). In Quantum terms, this is similar to how every Quantum client runs the full simulation — but in FishNet, only the Host runs the server simulation.

8. The Architecture Decision for Slop-Box

Given what we now know, here's how the topology decision maps to your co-op Prince of Persia game:

Requirement Decision Rationale
Player count 2 players (co-op) Host mode is ideal for small player counts
Competitive fairness Not critical (co-op, same team) Host's 0ms advantage is acceptable
Infrastructure cost Zero (no dedicated server) Host mode is free to run
NAT traversal Use Unity Relay or Steam Players shouldn't need to port-forward
Physics responsiveness CSP required (Lesson 3) Platformer needs frame-perfect input
Transport Tugboat (dev) → FishyUnityTransport (prod) Swap at ship time without code changes
💡 Why Not Quantum for This Project?

If you already know Quantum, you might wonder: "Why not just use Quantum for this game?" The answer is prototyping speed. With FishNet, you can use Unity's standard CharacterController, Rigidbody, and Physics APIs directly. You don't need to rewrite your movement system in fixed-point math. For a co-op PvE game where perfect determinism isn't critical, FishNet lets you iterate 3-5x faster during prototyping. If the game later needs competitive PvP or rollback netcode, Quantum would be the better choice.

Key Takeaways

  • Quantum = Deterministic Lockstep. Every client runs identical math. Correctness is guaranteed by design. FishNet = Server-Authoritative State. One boss enforces the truth; clients predict and get corrected.
  • FishNet uses standard Unity APIs. Floats, Vector3, Rigidbody, CharacterController — all safe to use because the server corrects any drift.
  • Host Mode is ideal for 2-player co-op. Free, simple, and the Host advantage doesn't matter when you're on the same team.
  • A Relay is not a server. It's a packet bouncer for NAT traversal. The Host still runs all authoritative logic.
  • Naive networked physics introduces a full RTT of input delay. For a platformer, this is unacceptable — Client-Side Prediction (Lesson 3) is mandatory.
  • Transports are swappable. Use Tugboat for dev, switch to FishyUnityTransport or FishySteamworks for production without changing game code.
