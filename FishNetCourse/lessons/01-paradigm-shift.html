Lesson 01

FishNet vs Quantum & Server Topologies

Bridging the gap from deterministic lockstep to server-authoritative state replication.

Introduction

As a senior technical artist comfortable with Photon Quantum, you are accustomed to a very specific set of rules: perfect determinism, separation of input/simulation/view, and lockstep rollbacks. FishNet represents a fundamentally different paradigm that is more closely aligned with traditional Unity development but comes with its own networking challenges.

🧠 Key Concept: Server-Authoritative State

Instead of relying on every client playing the exact same simulation perfectly in sync (Quantum), FishNet relies on a single authority (the Server) simulating the true state, and broadcasting that state to "dumb" clients who try to interpolate or predict the result.

The Paradigm Shift

In Quantum, logic must use fixed mathematical types (FP, FPVector3) and strictly avoid Unity's transform API or float-based physics to ensure determinism. In FishNet, you can use Unity's built-in physics and math because the server enforces the truth. If a client simulates something slightly different due to a float precision error, the server's update will eventually correct it (reconciliation).

This makes prototyping significantly faster, but it means dealing with network conditions differently. In Quantum, latency is handled by delaying input and rolling back the deterministic state. In FishNet, latency is handled by Client-Side Prediction (CSP) and Interpolation.

Server Topologies

When running a FishNet instance for a cooperative game like your Prince of Persia sandbox, you have two main architecture choices:

1. Dedicated Server (Authoritative)

The game server runs as a headless Unity instance in the cloud. No player has a zero-ping advantage. It's secure and highly reliable, but it costs money to host.

2. Host Mode (Peer-to-Peer / Relay-Backed)

One player acts as both a Client and the Server (the "Host"). The other players connect to the Host.

⚠️ Common Pitfall: Host Advantage

In Host mode, the hosting player has 0ms of latency and doesn't experience rubberbanding, while remote clients do. Co-op games often use Relay services (like Unity Relay or Steam lobbies) to bypass port forwarding, but the architecture remains Host-Client. A Relay is essentially a dumb pipe bouncing packets, but the Host is still running the server logic.

Why Networked Physics Can Feel "Slow"

When you use basic networked physics (like a Rigidbody synced over a NetworkTransform), remote clients are watching a delayed feed of the host's simulation. If a client presses "Jump", that input travels to the host, the host jumps, and the result travels back to the client. This results in sluggish, delayed controls.

💡 Pro Tip: The Solution

To get instantaneous, responsive physics (like in Quantum), we must employ Client-Side Prediction (CSP), which allows the client to immediately move its local Rigidbody/CharacterController, and only accept corrections from the server if it made a mistake. We'll cover this deeply in Lesson 3.

Key Takeaways

  • Quantum = Deterministic Lockstep. FishNet = Server-Authoritative State Replication.
  • FishNet allows standard Unity floats and Physics, trusting the server to correct discrepancies.
  • For a co-op game, Host mode (often backed by a Relay to avoid port forwarding) is standard, but you must account for host ping advantage.
  • Raw networked physics feels delayed without Client-Side Prediction (CSP).
