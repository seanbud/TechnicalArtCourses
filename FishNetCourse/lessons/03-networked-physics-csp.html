Lesson 03

Smooth Multiplayer Physics & Client-Side Prediction

Building frame-perfect, responsive movement for a co-op Prince of Persia — the single most important system in your game's netcode.

⚠️ This Is The Most Important Lesson

If you get CSP wrong, your game feels like controlling a character through mud. If you get it right, players won't even notice they're playing over a network. This lesson is the core of everything that makes FishNet worth using for a physics-heavy platformer.

1. The CSP Mental Model

Client-Side Prediction is conceptually similar to Quantum's rollback — but applied per-object instead of globally. Here's the full flow, tick by tick:

┌──────────────────────────────────────────────────────────────────────────┐ │ CLIENT-SIDE PREDICTION TIMELINE (CSP) │ ├──────────────────────────────────────────────────────────────────────────┤ │ │ │ CLIENT (Owner) SERVER (Authority) │ │ ────────────── ───────────────── │ │ │ │ Tick 100: Read input (Jump pressed) │ │ → Build MoveData { jump=true } │ │ → Execute [Replicate] LOCALLY ← Instant! Player jumps. │ │ → Send MoveData to Server ─────────────→ │ │ Tick 100: Receive MoveData │ │ Tick 101: Read input (Move forward) → Execute [Replicate] │ │ → Execute [Replicate] LOCALLY → Build MoveState │ │ → Send MoveData to Server ──────────────→ → Send [Reconcile] ──┐ │ │ │ │ │ Tick 102: Read input (still moving) Tick 101: Receive │ │ │ → Execute [Replicate] LOCALLY MoveData... │ │ │ │ │ │ ┌─ Tick 102: Receive Reconcile for Tick 100 ←────────────────────┘ │ │ │ Compare predicted position vs server: │ │ │ │ │ │ IF positions match (within tolerance): │ │ │ → Do nothing. Prediction was correct. ✅ │ │ │ │ │ │ IF positions diverge: │ │ │ → SNAP to server's Tick 100 state │ │ │ → REPLAY Tick 101 inputs on top of corrected state │ │ │ → REPLAY Tick 102 inputs on top of that │ │ │ → Player's screen shows corrected position │ │ │ → This all happens in ONE frame (invisible to player) ✅ │ │ └─────────────────────────────────────────────────────────────────── │ │ │ └──────────────────────────────────────────────────────────────────────────┘
🧠 The Musician Analogy

Think of CSP like a musician playing a piece from memory over a slightly delayed earpiece connected to a conductor. You play confidently ahead of the conductor. Most of the time, you're in perfect sync. If you miss a note and the conductor corrects you through the earpiece, you instantly snap back to that measure, speed-play all the notes you played since then, and get back in sync. To the audience, they never notice the correction. That's Replicate and Reconcile.

2. The Data Structures: MoveData & MoveState

CSP requires two structs: one for inputs going UP to the server, and one for state coming DOWN to the client. Getting these right is critical for bandwidth and correctness:

What Goes in Each Struct

Struct Direction Contains Sent How Often
MoveData (IReplicateData) Client → Server Player inputs ONLY (WASD, jump, sprint, camera yaw) Every tick (~60/sec)
MoveState (IReconcileData) Server → Client Authoritative physics state (position, velocity, grounded, timers) Every tick (~60/sec)
⚠️ Keep MoveData Small

MoveData is sent every single tick from client to server. If your tick rate is 60, that's 60 packets per second. Every extra byte costs bandwidth. Include ONLY the raw inputs needed to reproduce the movement — never include derived values like position or velocity in MoveData. Those belong in MoveState.

C#: MoveData — Input Struct (Client → Server)

using FishNet.Object.Prediction;
using UnityEngine;

/// <summary>
/// Inputs sent from the owning client to the server every tick.
/// Keep this as SMALL as possible — it's sent 60 times per second.
/// </summary>
public struct MoveData : IReplicateData
{
    // ── Raw Inputs ──
    public Vector2 InputDirection;  // WASD mapped to X/Z
    public float RotationY;         // Camera yaw (drives character facing)
    public float Pitch;             // Camera pitch (for aiming/looking)
    public bool JumpPressed;        // Was jump pressed THIS tick?
    public bool JumpHeld;           // Is jump HELD this tick? (variable jump height)
    public bool SprintHeld;         // Is sprint held?

    // ── FishNet Tick Tracking (REQUIRED) ──
    private uint _tick;
    public void Dispose() { }
    public uint GetTick() => _tick;
    public void SetTick(uint value) => _tick = value;
}

// ─── WHAT NOT TO PUT IN MOVEDATA ───
// ❌ Position       → This is OUTPUT, not input
// ❌ Velocity       → This is OUTPUT, not input
// ❌ IsGrounded     → This is OUTPUT, not input
// ❌ AnimationState → Derive from MoveState on the client
// ❌ Camera Pitch   → UNLESS it affects physics (e.g., aim direction)
//                     In Slop-Box, Pitch is included because it
//                     drives the camera pivot which affects simulation.

C#: MoveState — Authoritative State Struct (Server → Client)

using FishNet.Object.Prediction;
using UnityEngine;

/// <summary>
/// The authoritative physics state sent from server to client.
/// Must contain EVERY variable that affects physical position.
/// If a variable is missing, reconciliation will fail silently.
/// </summary>
public struct MoveState : IReconcileData
{
    // ── Core Physics State ──
    public Vector3 Position;         // World position
    public float RotationY;          // Body facing direction
    public Vector3 Velocity;         // Current movement velocity
    public bool IsGrounded;          // Is the character touching ground?

    // ── Simulation Timers ──
    // CRITICAL: Any timer that affects movement MUST be reconciled.
    // If JumpTimer ticks down in [Replicate] but isn't in MoveState,
    // the client and server will desync on jump duration.
    public float JumpTimer;          // Remaining jump force time
    public float CoyoteTimer;       // Time since leaving ground (coyote time)
    public float StaggerTimer;       // Knockback recovery timer

    // ── Camera State (for non-owner interpolation) ──
    public float Pitch;              // Look-up/down angle

    // ── FishNet Tick Tracking (REQUIRED) ──
    private uint _tick;
    public void Dispose() { }
    public uint GetTick() => _tick;
    public void SetTick(uint value) => _tick = value;
}

// ─── THE RECONCILE COMPLETENESS RULE ───
// If a variable is modified inside [Replicate] and it affects
// the character's position on subsequent ticks, it MUST be
// in MoveState. Otherwise, after reconciliation, the client
// will replay ticks with WRONG timer values and desync again.
//
// Symptoms of a missing variable:
// • Character "vibrates" or jitters visually
// • Jump height is inconsistent between host and client
// • Knockback distance differs between players

3. The Complete CSP NetworkBehaviour

Here is a production-grade CSP implementation. This is the pattern you'll use for any predicted object in your game — player characters, physics props, vehicles:

C#: NetworkMovement.cs — Full CSP Implementation

using FishNet.Object;
using FishNet.Object.Prediction;
using FishNet.Transporting;
using UnityEngine;

/// <summary>
/// FishNet Client-Side Prediction wrapper.
/// Collects input → ships to server → predicts locally → reconciles.
///
/// ARCHITECTURE: This script handles ONLY networking.
/// Actual physics/movement math lives in CharacterMovement.cs.
/// This separation keeps your simulation testable offline.
/// </summary>
public class NetworkMovement : NetworkBehaviour
{
    [Header("Simulation")]
    [SerializeField] private CharacterMovement _sim;

    private IInputProvider _input;

    // ═══════════════════════════════════════════════════════
    // LIFECYCLE
    // ═══════════════════════════════════════════════════════

    private void Awake()
    {
        if (_sim == null)
            _sim = GetComponent<CharacterMovement>();
    }

    public override void OnStartNetwork()
    {
        base.OnStartNetwork();
        // OnTick = fixed network timestep (like Quantum's frame advance)
        TimeManager.OnTick += OnTick;
        TimeManager.OnPostTick += OnPostTick;
    }

    public override void OnStopNetwork()
    {
        base.OnStopNetwork();
        if (TimeManager != null)
        {
            TimeManager.OnTick -= OnTick;
            TimeManager.OnPostTick -= OnPostTick;
        }
    }

    // ═══════════════════════════════════════════════════════
    // TICK LOOP (runs at fixed network rate, e.g. 60Hz)
    // ═══════════════════════════════════════════════════════

    private void OnTick()
    {
        // Step 1: Build input data (only meaningful for the owner)
        MoveData input = BuildMoveData();

        // Step 2: Execute [Replicate]
        // - On OWNER CLIENT: executes immediately as prediction
        // - On SERVER: executes authoritatively with the client's input
        // - On NON-OWNER CLIENTS: executes with default/empty data
        Replicate(input);
    }

    private void OnPostTick()
    {
        // Step 3: After the server's [Replicate], build authoritative state
        // FishNet calls CreateReconcile() automatically on the server
    }

    // ═══════════════════════════════════════════════════════
    // INPUT COLLECTION (runs on owner client only)
    // ═══════════════════════════════════════════════════════

    private MoveData BuildMoveData()
    {
        // Return empty data if we're not the owner
        // (server and non-owner clients don't generate input)
        if (!IsOwner || _input == null)
            return default;

        // Don't move if the cursor is unlocked (menu, DevBar, etc.)
        if (Cursor.lockState != CursorLockMode.Locked)
            return new MoveData
            {
                RotationY = _sim.CameraYaw,
                Pitch = _sim.CameraPitch
            };

        return new MoveData
        {
            InputDirection = _input.MoveInput,
            RotationY = _sim.CameraYaw,
            Pitch = _sim.CameraPitch,
            JumpPressed = _input.IsJumpPressed,
            JumpHeld = _input.IsJumpHeld,
            SprintHeld = _input.IsSprintHeld
        };
    }

    // ═══════════════════════════════════════════════════════
    // [REPLICATE] — The Core Simulation Step
    // ═══════════════════════════════════════════════════════

    /// <summary>
    /// Executes movement physics for one tick.
    ///
    /// This runs on:
    ///   • Owner Client: IMMEDIATELY (prediction)
    ///   • Server: with the SAME input (authoritative)
    ///   • Owner Client AGAIN: during reconciliation replays
    ///
    /// CRITICAL RULES:
    ///   1. Use TimeManager.TickDelta, NEVER Time.deltaTime
    ///   2. Only use data from the MoveData struct + simulation state
    ///   3. Never read external state (Time.time, Input.GetKey, etc.)
    ///   4. Must be DETERMINISTIC given the same inputs + state
    /// </summary>
    [Replicate]
    private void Replicate(
        MoveData data,
        ReplicateState state = ReplicateState.Invalid,
        Channel channel = Channel.Unreliable)
    {
        // Delegate to the pure physics simulation
        float delta = (float)TimeManager.TickDelta;
        _sim.Simulate(data, delta);
    }

    // ═══════════════════════════════════════════════════════
    // [RECONCILE] — Server Correction
    // ═══════════════════════════════════════════════════════

    /// <summary>
    /// Called automatically by FishNet. The server builds
    /// the authoritative state, sends it to the owning client.
    /// The client snaps to this state and replays unacked inputs.
    /// </summary>
    public override void CreateReconcile()
    {
        MoveState state = new MoveState
        {
            Position = transform.position,
            RotationY = transform.eulerAngles.y,
            Velocity = _sim.Velocity,
            IsGrounded = _sim.IsGrounded,
            JumpTimer = _sim.JumpTimer,
            CoyoteTimer = _sim.CoyoteTimer,
            StaggerTimer = _sim.StaggerTimer,
            Pitch = _sim.CameraPitch
        };
        Reconcile(state);
    }

    /// <summary>
    /// Applies the server's authoritative state to the local simulation.
    /// After this, FishNet replays all unacknowledged [Replicate] calls.
    /// </summary>
    [Reconcile]
    private void Reconcile(
        MoveState state,
        Channel channel = Channel.Unreliable)
    {
        // Snap the simulation to the server's truth
        transform.position = state.Position;
        transform.eulerAngles = new Vector3(0f, state.RotationY, 0f);
        _sim.SetState(state);
    }
}

4. The Replay Loop Explained

When a reconcile correction arrives, FishNet performs the following steps in a single frame. This is the replay loop — the heart of CSP:

┌──────────────────────────────────────────────────────────────────────────┐ │ THE RECONCILIATION REPLAY LOOP │ ├──────────────────────────────────────────────────────────────────────────┤ │ │ │ 1. Server sends MoveState for Tick 100 │ │ (Position: [10, 0, 5], Velocity: [2, 0, 0], Grounded: true) │ │ │ │ 2. Client is currently at Tick 103 (predicted ahead by 3 ticks) │ │ Client's predicted position: [16.1, 0, 5] │ │ Server's authoritative position: [16.0, 0, 5] │ │ │ │ 3. Client compares: difference = 0.1 units │ │ (FishNet handles this comparison internally) │ │ │ │ 4. IF difference exceeds tolerance: │ │ ┌─────────────────────────────────────────────────────────┐ │ │ │ a. SNAP transform to server state at Tick 100 │ │ │ │ position = [10, 0, 5] │ │ │ │ velocity = [2, 0, 0] │ │ │ │ │ │ │ │ b. REPLAY Tick 101 input on corrected state │ │ │ │ → [Replicate](savedInput101) │ │ │ │ │ │ │ │ c. REPLAY Tick 102 input │ │ │ │ → [Replicate](savedInput102) │ │ │ │ │ │ │ │ d. REPLAY Tick 103 input │ │ │ │ → [Replicate](savedInput103) │ │ │ │ │ │ │ │ e. Client is now at Tick 103 with CORRECTED state │ │ │ │ New predicted position: [16.0, 0, 5] ✅ │ │ │ └─────────────────────────────────────────────────────────┘ │ │ │ │ 5. ALL of this happens in ONE Unity frame — invisible to the player │ │ │ └──────────────────────────────────────────────────────────────────────────┘
🧠 Why This Works

The key insight is that [Replicate] is a pure function — given the same input and the same starting state, it produces the same result. So by snapping to the server's state and replaying all buffered inputs, the client arrives at the correct predicted position. As long as your Replicate logic is deterministic (no Time.deltaTime, no random calls), the correction is invisible.

5. TickDelta vs DeltaTime: The Silent Killer

This is the single most common bug in FishNet CSP implementations, and it causes the most infuriating symptom: characters that jitter and vibrate.

Time.deltaTime TimeManager.TickDelta
Value Variable (depends on frame rate) Fixed (e.g., 1/60 = 0.01667s)
On Client at 144fps ~0.00694s 0.01667s
On Server at 30fps ~0.03333s 0.01667s
During Replay ~0.00001s (runs in a burst) 0.01667s (correct)
CSP Compatible? ❌ NO — breaks everything ✅ YES — required
⚠️ The Symptom

If you use Time.deltaTime inside [Replicate], here's what happens: the client predicts movement using its frame rate (e.g., 144fps = tiny delta). The server runs at a different frame rate (e.g., 60fps = larger delta). Even if the same input is applied, the results diverge. Then during reconciliation replay, Time.deltaTime is near-zero because 3 ticks are replayed instantly in one frame. The character position oscillates between prediction and correction every frame, causing visible jitter. Always use (float)TimeManager.TickDelta inside [Replicate].

6. Interpolation for Non-Owner Observers

CSP only handles the owning player's experience. Other players watching you move see a different challenge: they receive your position updates at the server's tick rate, which can look choppy without interpolation.

Technique How It Works Latency Best For
Interpolation Smoothly blend between the last two known positions Adds ~1 tick of visual delay Co-op allies, NPCs, most objects
Extrapolation Predict where the object is going based on velocity No added delay, but can overshoot Fast projectiles
Snap Teleport to each new position None Nothing — looks terrible

FishNet's built-in NetworkTransform component handles interpolation for non-predicted objects automatically. For CSP-predicted objects, non-owners see the result of the server's [Replicate] execution, which naturally provides smooth updates at the tick rate.

7. Physics Settings for CSP

Unity's physics engine needs specific configuration to work correctly with FishNet's tick-based simulation:

C#: Physics Configuration for FishNet CSP

// In your bootstrap or NetworkManager setup:

void ConfigurePhysicsForCSP()
{
    // 1. Disable automatic Physics simulation.
    //    FishNet advances physics manually per-tick.
    Physics.simulationMode = SimulationMode.Script;

    // 2. Set fixed timestep to match FishNet's tick rate.
    //    If NetworkManager.TickRate = 60, then:
    Time.fixedDeltaTime = 1f / 60f;  // 0.01667s

    // 3. FishNet's PredictionManager handles calling
    //    Physics.Simulate() at the right time during
    //    [Replicate] execution and reconciliation replays.
    //
    //    You do NOT call Physics.Simulate() yourself.
    //    FishNet does it for you.
}

// WHY Physics.simulationMode = Script?
//
// If Unity runs physics automatically in FixedUpdate(),
// it will run at its own rate, independent of network ticks.
// This causes the client's physics to drift from the server's.
// By switching to Script mode, FishNet controls EXACTLY when
// physics steps occur, ensuring client and server stay in sync.

8. Prince of Persia: Encoding Complex Actions in MoveData

For a platformer with advanced movement, your MoveData needs to encode everything the simulation needs to reproduce acrobatic actions:

C#: Extended MoveData for a Platformer

public struct PlatformerMoveData : IReplicateData
{
    // ── Basic Movement ──
    public Vector2 InputDirection;
    public float CameraYaw;
    public bool SprintHeld;

    // ── Jump Mechanics ──
    public bool JumpPressed;      // Tap = normal jump
    public bool JumpHeld;         // Hold = variable jump height
    // Note: double-jump is NOT an input — it's a state check
    // inside [Replicate]. If IsGrounded==false && JumpPressed
    // && doubleJumpAvailable, consume the double jump.

    // ── Wall Interactions ──
    public bool GrabPressed;      // Grab ledge / wall grip
    // Note: whether the player IS on a wall is checked by raycasts
    // inside [Replicate], not sent as input. Only the INTENT
    // to grab is input. The physics determines if they CAN grab.

    // ── Combat ──
    public bool AttackPressed;    // Swing sword
    public bool BlockHeld;        // Shield up

    // ── FishNet Required ──
    private uint _tick;
    public void Dispose() { }
    public uint GetTick() => _tick;
    public void SetTick(uint value) => _tick = value;
}

// ─── THE INPUT vs STATE RULE ───
//
// INPUT (MoveData): What the player WANTS to do.
//   → "I pressed jump"   "I'm holding grab"   "I pressed attack"
//
// STATE (MoveState): What the physics ACTUALLY resulted in.
//   → "I'm at position X"  "I'm on a wall"  "My velocity is Y"
//
// The [Replicate] function connects them:
//   Given INPUT + current STATE → produce next STATE
💡 The Golden Rule of MoveData

Only encode player intent. Never encode simulation results. "I pressed jump" is input. "I'm in the air" is state. "I want to grab a ledge" is input. "I'm currently grabbing a ledge" is state. If you mix these up, your reconcile loop will replay with stale state data and produce constant jitter.

Key Takeaways

  • CSP = predict locally, correct from server. The client executes movement immediately. The server sends corrections. FishNet replays unacked inputs automatically during reconciliation.
  • [Replicate] must be a pure function. Same input + same state = same result. No Time.deltaTime, no Random, no external reads.
  • Always use TimeManager.TickDelta. Using Time.deltaTime inside [Replicate] causes jitter because client, server, and replay all run at different frame rates.
  • MoveData = player intent (inputs). MoveState = physics results (positions, velocities, timers). Never mix them.
  • Every variable modified in [Replicate] that affects position MUST be in MoveState. Missing a timer or flag causes invisible desyncs that manifest as jitter.
  • Set Physics.simulationMode = Script. FishNet controls when physics steps occur to keep client and server in sync.
  • Non-owners see interpolated positions. CSP only helps the owner. Everyone else sees smooth interpolation of server-authoritative updates.
