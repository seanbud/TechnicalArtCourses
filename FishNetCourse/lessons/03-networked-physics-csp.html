Lesson 03

Smooth Multiplayer Physics & CSP

Client-Side Prediction (CSP) for responsive movement in authoritative servers.

Introduction

This is where FishNet shines over generic built-in Unity networking. If you want a Prince of Persia style game where movement must feel instantaneous and perfectly smooth, standard server-authoritative logic (sending input, waiting for server to move you) feels terrible due to latency.

🧠 Key Concept: Client-Side Prediction (CSP)

In CSP, the client executes its input immediately upon pressing a button. At the same time, it sends that input to the server. The server executes it authoritatively. If the client's result diverges from the server's authoritative result, the client "snaps" back to the server's reality and replays any inputs that happened in between. It is functionally similar to a Quantum rollback, but applied only to specific NetworkBehaviours instead of the entire global game state.

The Replicate & Reconcile Pattern

FishNet implements CSP through two critically important method attributes: [Replicate] and [Reconcile].

1. [Replicate]

This method houses your actual physics or movement logic (e.g., characterController.Move). It is called multiple times:

  • On the Client: First to predict the movement immediately, and later during replays during reconciliation.
  • On the Server: To execute the authoritative movement using the client's shipped inputs.

2. [Reconcile]

When the Server finishes a `[Replicate]`, it fires a `[Reconcile]` back to the owners. This method receives a struct containing the true state of the object (position, rotation, velocity). The Client receives this, applies the true state, and then FishNet automatically calls `[Replicate]` again for any unacknowledged inputs (the replay).

⚠️ Common Pitfall: Non-deterministic logic in Replicate

If your [Replicate] logic relies on `Time.deltaTime` instead of the data provided by FishNet's `TimeManager.TickDelta`, your client and server will desync immediately. CSP relies on executing the exact same mathematical instructions using fixed tick time steps.

Setting up CSP

To use this, you must define data structures for Inputs (which go up to the server) and State (which comes down to the client).

Code Example: CSP Boilerplate

public struct MoveData : IReplicateData {
    public Vector3 InputDirection;
    public bool Jump;
    private uint _tick;
    public uint GetTick() => _tick;
    public void SetTick(uint value) => _tick = value;
    public void Dispose() {}
}

public struct MoveState : IReconcileData {
    public Vector3 Position;
    public Vector3 Velocity;
    private uint _tick;
    public uint GetTick() => _tick;
    public void SetTick(uint value) => _tick = value;
    public void Dispose() {}
}
💡 Pro Tip: Struct Layouts

Always keep your Replicate and Reconcile structs as small as possible. This data is being sent up to 60 times a second.

Key Takeaways

  • CSP gives clients the illusion of zero latency by predicting authoritative results locally.
  • [Replicate] logic runs on both Client and Server. It must only use inputs provided to it, never outside variables.
  • [Reconcile] snaps the Client to the exact Server state.
  • FishNet handles the rollback and replay (re-executing Replicate) automatically.
