Lesson 05

Prediction Setup & Configuration

Everything you need to configure in the Unity Inspector before writing a single line of prediction code — PredictionManager, TimeManager, NetworkObject, and the interpolation pipeline.

🧠 Why This Lesson Matters

FishNet's prediction system is powerful, but it's distributed across three separate components — each with inspector settings that interact in non-obvious ways. Get the configuration wrong and your code will compile but produce jitter, rubber-banding, or silent desync. This lesson gives you a complete, annotated walkthrough of every prediction-related setting, so when you write prediction code in the next lesson, you'll have a solid foundation.

What You'll Learn

  • What Client-Side Prediction (CSP) actually is — and how it differs from other server-authoritative patterns
  • How to configure the PredictionManager for interpolation and replicate queue management
  • Why the TimeManager must own physics and how to enable it
  • Every prediction-related setting on the NetworkObject inspector — state forwarding, graphical smoothing, and adaptive interpolation
  • How to handle Offline Rigidbodies in prediction scenes
  • The critical difference between PredictionManager interpolation and NetworkObject interpolation

1. What Is Client-Side Prediction?

Client-Side Prediction (CSP) is a technique that lets the owning client execute actions immediately — without waiting for the server — while the server still retains full authority over the game state. The client "predicts" the outcome of its own input, and the server corrects it later if the prediction was wrong.

This is different from a naive server-authoritative approach where the client sends input, waits for the server to process it, and then receives the result. That pattern introduces latency equal to the full round-trip time (RTT), which makes movement feel sluggish — especially above 80ms.

❌ Naive Server-Authoritative

Client sends input → waits for server → receives result → moves.

Latency: Full RTT before any visual movement. Feels unresponsive at >50ms.

✅ Client-Side Prediction (CSP)

Client sends input + immediately predicts result → server validates → client reconciles if wrong.

Latency: Zero perceived input lag. Server corrections are invisible when prediction is accurate.

💡 CSP vs "Server Authoritative Movement"

You'll hear both terms used interchangeably, but they're not the same. Server-authoritative movement is the broad category — it just means the server has final say. CSP is a specific implementation of server-authoritative movement where the client predicts ahead. FishNet's prediction system gives you CSP out of the box with [Replicate] and [Reconcile] attributes.

The CSP Tick Loop

Here's what happens on each network tick when CSP is active:

CLIENT (Owner) SERVER ━━━━━━━━━━━━━━━━━━ ━━━━━━━━━━━━━━━━━━━━ 1. Collect input (MoveData) 2. Run [Replicate] locally (prediction) 3. Send input to server ───────────────→ 4. Receive input 5. Run [Replicate] authoritatively 6. CreateReconcile() → MoveState ←─── 7. Send reconcile to client 8. Receive reconcile 9. Compare predicted state vs server state 10. If different → rollback to server state and replay all inputs since

Steps 9-10 are the reconciliation loop — FishNet handles this automatically. If the client's prediction was correct (which it usually is for simple movement), steps 9-10 are effectively a no-op and the player never notices.

2. Configuring the PredictionManager

The PredictionManager is a component on the NetworkManager GameObject. It controls global prediction settings that apply to all predicted objects in the scene. You don't strictly need to add it — FishNet will use defaults — but adding it lets you fine-tune two critical systems.

Official Docs: Configuring PredictionManager

2.1 Interpolation (Input Buffer)

This is not graphical smoothing — it's how many ticks of input the system holds in a queue before running them. Think of it as a shock absorber for network jitter.

Setting What It Controls Recommended Values
Client Interpolation How many ticks of server/other-client input are buffered before running. 1 for fast-paced games. 2-3 for casual games where stability matters more than latency.
Server Interpolation How many ticks the server buffers received client inputs before executing them. Usually lower than client — 0-1 is typical. Higher values add server-side input lag.
⚠️ Interpolation = Added Latency

Every 1 point of interpolation adds latency equal to interpolation × TickDelta. At 60 ticks/sec (TickDelta ≈ 16.7ms), an interpolation of 2 adds ~33ms of delay. This is the trade-off: more interpolation means fewer situations where input isn't available (preventing "future prediction"), but it adds delay to when those inputs actually run.

2.2 Excessive Replicate Dropping

The server's input queue should normally contain ServerInterpolation ±1 entries. But if a client has network issues and sends inputs in bursts, the queue could spike to 5+ entries at once.

When the queue exceeds the maximum, FishNet drops the oldest values. This serves two purposes:

  • Security: Prevents allocation attacks (malicious clients flooding the queue)
  • Anti-cheat: Prevents extra inputs from being processed (which could be used to speed-hack)
🚨 Don't Disable Replicate Dropping

You can disable excessive replicate dropping in the inspector, but doing so opens your game to cheating — multiple replicates will run per tick on the server to consume the extras. FishNet still has a hard-coded maximum as a safety net, but this is a last resort, not a feature you should rely on.

3. Configuring the TimeManager

The TimeManager component on the NetworkManager controls timing for the entire network session. For prediction, there is one critical setting you must change.

Official Docs: Configuring TimeManager

🚨 Mandatory: Set Physics Mode to "Time Manager"

By default, Unity manages its own physics timing via FixedUpdate. When using FishNet prediction, you must switch physics control to the TimeManager. This ensures that physics steps happen at the network tick rate rather than Unity's default fixed timestep, which is essential for deterministic replay during reconciliation.

Setup Steps

  1. Select your NetworkManager GameObject in the scene
  2. Ensure the TimeManager component is present (add it if missing)
  3. Change Physics Mode from Unity to Time Manager

That's it. This single setting change is easy to miss but will cause subtle, hard-to-debug physics desync if forgotten. The symptom: movement works in editor (where the host has 0ms latency) but drifts on remote clients.

4. Configuring the NetworkObject

The NetworkObject component is where prediction settings are configured per-object. This is the most complex inspector in the prediction pipeline, with settings that control how the object is predicted, how it smooths, and how non-owners see it.

Official Docs: Configuring NetworkObject

4.1 Enable Prediction

The first step is checking Enable Prediction on the NetworkObject inspector. Without this, prediction methods ([Replicate], [Reconcile]) won't function on scripts attached to this object.

Next, set the Prediction Type:

Prediction Type Use When
Rigidbody The object uses a Rigidbody for physics (forces, velocity, collisions)
Rigidbody2D The object uses a Rigidbody2D (2D physics)
Other Non-physics movement — CharacterController, transform-based, or custom simulation

4.2 State Forwarding

This is one of the most important architectural decisions for your predicted objects.

✅ State Forwarding: Enabled

All clients run the same inputs as the server. Non-owners keep a state buffer and reconcile just like the owner. This means every player sees identical simulation logic running on their machine.

Pros: Easier coding, more reliable simulation, identical behaviour on all clients.

Cons: Higher CPU cost (every client maintains a buffer and reconciles).

⚠️ State Forwarding: Disabled

Only the owner and server run inputs. Non-owners see nothing by default — you must manually forward information via NetworkAnimator, RPCs, or NetworkTransform.

Pros: Lower CPU. Fine for simple scenarios.

Cons: You manually relay all visual information to spectators. Easy to miss things.

💡 When In Doubt, Enable State Forwarding

State forwarding is the most commonly preferred approach because it lets all clients and the server run identical logic. You lose some CPU to the per-client buffers, but you gain a much simpler programming model where you don't have to think about "what does the spectator see?" — they see the same thing the owner sees, just with interpolation delay.

4.3 Graphical Object

This is the child transform that holds your visual representation — the mesh, sprite, particle systems, and anything rendered. FishNet uses this reference to apply tick-based smoothing separate from the physics root.

🚨 Don't Put Gameplay Colliders on the Graphical Object

During a network tick, FishNet runs your [Replicate] method using TickDelta. If your graphical object is being smoothed (rolled back and interpolated), any colliders on it will be in the wrong position during the physics step. Keep gameplay colliders (CapsuleCollider, triggers that affect state) on the same GameObject as the NetworkObject — or at least outside the graphical hierarchy.

Attached vs Detached Graphical

Mode Behaviour Best For
Attached (default) Graphical object rolls back to its pre-tick transform, then smooths to the post-tick result over the tick duration. Most setups. Works well when you have no third-party assets that fight for transform control.
Detached Graphical object exists in world space and smoothly moves toward the goal position. No rollback. When animation systems, cameras, or third-party assets don't handle the rollback-then-smooth pattern gracefully. Often the better choice in practice.

4.4 Smoothing Options

The smoothing settings only appear when you've assigned a graphical object. They control what gets smoothed and how aggressively.

Setting Description
Smoothed Properties (Owner) Which transform properties (position, rotation, scale) are smoothed for the owning client. Untick rotation if you're self-managing it (e.g., mouse-look).
Smoothed Properties (Spectator) Same as above but for non-owners watching this object. Often you want all three smoothed here.
Adaptive Interpolation (Spectator) Controls the level of visual interpolation on spectated objects. Lower = less delay but desyncs are more visible. Higher = smoother but more visual lag.
🧠 Smoothing ≠ Prediction

Changing smoothing settings has no effect on prediction accuracy. Smoothing is purely visual — it controls how the graphical child moves between ticks. The prediction, reconcile, and replay loop that keeps the physics root in sync with the server is completely separate. If you see jitter, tuning smoothing might hide it visually, but the real fix is almost always in your reconcile state completeness (see Lesson 06).

5. Offline Rigidbodies

If your scene has non-networked rigidbodies (props, debris, physics objects that aren't synced), they need special handling when prediction is active.

Official Docs: Offline Rigidbodies

When prediction runs its replay loop during reconciliation, it re-simulates physics. Any rigidbody in the scene will be affected by this re-simulation — even ones that aren't networked. Without the OfflineRigidbody component, these objects will jitter, teleport, or behave erratically because they're being rolled back and replayed as part of the global physics step.

⚠️ Rule: Every Non-Networked Rigidbody Needs OfflineRigidbody

Add the OfflineRigidbody component to any rigidbody in your scene that doesn't have a NetworkObject. This tells FishNet's prediction system to exclude them from the replay loop. Common targets: decorative physics props, breakable objects, ragdolls, loose crates. See the component documentation for details.

6. Understanding the Two Interpolation Systems

This is one of the most commonly confused aspects of FishNet's prediction. There are two completely different interpolation systems, and they serve different purposes.

Official Docs: Interpolations

PredictionManager Interpolation NetworkObject / TickSmoother Interpolation
What it does Holds states in a queue before running them. Buffers input to handle network jitter. Smooths the graphical object between tick positions. Purely visual.
Where it's set PredictionManager component on NetworkManager NetworkObject inspector (Smoothing section) or NetworkTickSmoother component
Affects When states are executed (adds delay) How smooth movement looks between ticks (no effect on simulation)
Trade-off More buffer = more resilience to packet loss, but more input lag More smoothing = less visual jitter on spectated objects, but slightly behind "true" position
🧠 Stacking Example

If your PredictionManager has ClientInterpolation = 1 and a spectated object's TickSmoother has SpectatorInterpolation = 2, the spectator sees the object's graphical representation 3 ticks behind real-time: 1 tick of queue delay + 2 ticks of visual smoothing. At 60 ticks/sec, that's ~50ms — perfectly acceptable for most games.

7. Scene Setup Checklist

Before writing any prediction code, make sure every item on this list is checked:

# Item Location Status
1 TimeManager Physics Mode set to Time Manager NetworkManager → TimeManager component Required ✅
2 PredictionManager added with appropriate interpolation values NetworkManager → PredictionManager component Recommended ⚠️
3 Enable Prediction checked on all predicted NetworkObjects Player prefab → NetworkObject inspector Required ✅
4 Prediction Type set correctly (Rigidbody / Other) Player prefab → NetworkObject inspector Required ✅
5 State Forwarding enabled (if spectators need to run logic) Player prefab → NetworkObject inspector Recommended ⚠️
6 Graphical Object assigned (if using tick smoothing) Player prefab → NetworkObject inspector Recommended ⚠️
7 OfflineRigidbody on all non-networked rigidbodies Any physics prop without a NetworkObject Required ✅
8 Gameplay colliders NOT on the graphical object child Player prefab hierarchy Required ✅

Key Takeaways

  • CSP lets the client move immediately while the server retains authority. If the client's prediction matches the server, the player never perceives any correction.
  • PredictionManager interpolation buffers input — it's not graphical smoothing. Higher values = more resilience to jitter, but more input lag.
  • TimeManager must own physics. Switch Physics Mode to Time Manager or prediction replay will use the wrong delta and produce silent desync.
  • NetworkObject is where per-object prediction lives. Enable prediction, choose the right type, consider state forwarding, and set up your graphical object hierarchy carefully.
  • OfflineRigidbody is mandatory for any non-networked rigidbody in a prediction scene. Without it, props will jitter during reconciliation replay.
  • Two interpolation systems exist — PredictionManager (queue buffer) and NetworkObject/TickSmoother (visual smoothing). They stack. Understand both to tune your game's feel.
