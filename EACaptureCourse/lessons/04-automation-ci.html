Automation & GitLab CI

Building the "Invisible Robot": .gitlab-ci.yml, Docker, and Runners.

The Invisible Robot

Imagine you hired a new employee. Their desk is in the server room. They have no monitor, no mouse, no keyboard. But they work 24/7 and never make a mistake. This employee is your CI System.

🧠 Concept: Headless Mode

How do you run Maya without a screen? You use Batch Mode.
mayapy.exe -batch -command "python(\"import my_script\")"
If your script tries to open a popup window ("Are you sure?"), the robot waits forever. The build hangs. Rule #1: Separate Logic from UI.

2. Implementation: The .gitlab-ci.yml

This is what you need to be able to write from memory (or close to it).

The Pipeline Configuration

stages:
  - build
  - test
  - deploy

# Define a "Base Job" to share config
.maya_job:
  image: polypipeline/maya-2024:latest # Custom Docker Image
  tags:
    - gpu-runner # Runs on a machine with a GPU
  before_script:
    - echo "Setting up License..."
    - export MAYA_LICENSE=...

validate_pyside:
  stage: test
  extends: .maya_job
  script:
    # Run the tests in Headless Maya
    - mayapy.exe -batch -command "python(\"import tests; tests.run()\")"
  artifacts:
    when: always
    paths:
      - test_results.xml

build_docs:
  stage: deploy
  image: python:3.9
  script:
    - pip install sphinx
    - sphinx-build -b html docs/ public/
  only:
    - main # Only run on Main branch

3. The "Clean Room" (Docker)

The most dangerous sentence in engineering is: "It works on my machine." Docker solves this by creating a "Container" (a lightweight Virtual Machine) that matches the production environment exactly.

The Dockerfile

This recipe builds the environment your pipeline runs in.

# Start from a base OS
FROM centos:7

# Install Dependencies
RUN yum install -y git python3

# Install Maya (Simplified)
COPY Install/Maya2024 /tmp/Maya2024
RUN /tmp/Maya2024/setup --silent

# Set Environment Variables
ENV MAYA_LOCATION=/usr/autodesk/maya2024
ENV PATH=$MAYA_LOCATION/bin:$PATH

# The command to run when container starts
CMD ["mayapy"]
⚠️ Environment Variables

Never hardcode paths like `Z:\Projects`. Use Environment Variables (`$PROJECT_ROOT`). Docker injects these at runtime, allowing the same code to run on your laptop AND the cloud.

4. Technical Drill: Interview Prep

🗣️ Interview Prompt

"How do you handle License Servers in a Docker Container?"

The Answer: Network Bridging.

Docker containers by default are isolated. They cannot see the corporate network where the License Server lives. You must run the container with --network host or pass the License Server IP as an ENV Variable so the container can "phone home" to check out a license.

Key Takeaways

  • "Works on my machine" is dead: CI/CD ensures code works in a clean, neutral environment.
  • Headless Mode: Tools must run without a GUI (Batch Mode) to work on the farm/CI.
  • Docker: A lightweight VM that defines the exact environment (OS, Maya version, Libraries).
  • GitLab CI YAML: The recipe file that tells the runner what to do (Build -> Test -> Deploy).
  • GPU Runners: You need specialized hardware runners to run Maya/Unity tests effectively.
