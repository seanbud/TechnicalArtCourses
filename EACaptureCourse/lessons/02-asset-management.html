Lesson 02

Pipeline & Perforce Administration

Scripting the Server: P4Python, Triggers, and Typemaps.

1. The Merge Conflict Problem

Programmers use Git because code is text. Artists use Perforce because assets are binary. If two artists edit the same 2GB `.ZTL` file, Git cannot merge them. One person loses their work. This is unacceptable in production.

🧠 Concept: Pessimistic Locking

Git uses Optimistic Locking (anyone can edit anything). Perforce uses Pessimistic Locking (you must "Check Out" a file before editing). The server enforces a lock (`+l`) that prevents anyone else from modifying the file until you submit.

2. Implementation: Automating P4 with Python

As a Technical Artist, you don't use the GUI. You write tools that manage the pipeline. You must know P4Python.

Python: The "Unlocker" Tool

Scenario: "Artist A went on vacation but left a critical file Checked Out. Write a script to find and unlock it."

from P4 import P4, P4Exception

def force_unlock_file(depot_path):
    p4 = P4()
    p4.user = "pipeline_admin"
    p4.password = "secret123"
    
    try:
        p4.connect()
        # 1. Find who has the file
        # fstat returns a list of dictionaries
        info = p4.run_fstat(depot_path)[0]
        
        if 'action' in info:
            user = info['otherOpen'][0] # User who has it open
            client = info['otherClient'][0] # Their workspace
            
            print(f"File locked by {user} on {client}. Reverting...")
            
            # 2. Impersonate the user to Revert (Admin privilege required)
            p4.client = client
            p4.user = user
            p4.run_revert(depot_path)
            print("File Unlocked!")
            
    except P4Exception as e:
        print(e)
    finally:
        p4.disconnect()

Python: The "Bad Name" Rejector

This script runs on the Perforce Server itself. If it returns exit(1), the user's submit fails.

import sys
# Arguments passed by P4: [TriggerName, ChangelistID, User]
changelist = sys.argv[2] 

def validate_filename(filename):
    # Rule: Must be lowercase and no spaces
    if " " in filename:
        return False
    if filename != filename.lower():
        return False
    return True

# Pseudo-code for checking a changelist
files = p4.run_describe(changelist)
for file in files:
    if not validate_filename(file['depotFile']):
        print(f"ERROR: File '{file['depotFile']}' has spaces or caps.")
        sys.exit(1) # BLOCK THE SUBMIT

sys.exit(0) # ALLOW THE SUBMIT

The Typemap

The `p4 typemap` table tells the server how to handle file extensions. This is your first line of defense.

TypeMap:
    binary+l //....fbx   <-- Auto-Lock all FBX files
    binary+l //....uasset
    text     //....py
    text     //....json

4. Streams: The Branching Strategy

We organize data into Streams to allow parallel development. A "Stream" is a branch with strict flow rules.

  • Mainline: The "Truth". Always stable. The game builds from here.
  • Development: The "Lab". Unstable. Where new features are tested.
  • Release: The "Snapshot". Frozen code for shipping (e.g., E3 Demo).
⚠️ The Flow Rules

Merge Down, Copy Up. You must merge changes from Main down to Dev daily (resolve conflicts). You only copy changes from Dev up to Main when they are stable. This protects the Mainline from breaking.

🎬 EA Studio Relevance: Battlefield Destruction

Imagine Battlefield level destruction. A single building might have 500 destruction swatches (rubble, rebar, glass). That asset is 50GB. If you try to merge 50GB of binary data in Git, your repo explodes. EA uses Perforce because we need to lock that file so two artists don't destroy the same building at the same time.

Key Takeaways

  • Binaries Don't Merge: This is why we use Perforce (locking) instead of Git (branching) for assets.
  • Pessimistic Locking: The +l modifier ensures only one person edits a file at a time.
  • Streams: Define the flow of data (Dev → Main → Release) and isolate stability.
  • P4Python: The automation layer. Use it to write Triggers and administration scripts.
  • Typemap: The `p4 typemap` file controls file locking behavior globally.
